モナドひとつかみ

ファンクタを最初に学んだとき、map できる値の概念として有用だと知りました。さらに踏み込んでアプリカティブファンクタを導入し、特定のデータ型の値を「文脈付きの値」と捉え、文脈の意味を保ちつつ通常の関数を使えるようにしました。

この章では、モナドを学びます。アプリカティブファンクタがファンクタを強化したものであるように、モナドはアプリカティブファンクタを強化したものです。

ファンクタから始めたとき、さまざまなデータ型に関数を map できると見ました。Functor 型クラスが導入され、「a -> b 型の関数とデータ型 f a があるとき、関数を map して f b を得るには？」という問いに答えました。Maybe a、[a]、IO a などへの map を見ました。a -> b 関数を r -> a 関数に map して r -> b 関数を得る方法も。fmap の型を見るだけで答えが分かります： 

fmap :: (Functor f) => (a -> b) -> f a -> f b

あとはデータ型に応じた Functor インスタンスを書けば動きます。

次に、ファンクタの拡張を考えました。「a -> b 関数がファンクタ値の中に既に包まれていたら？」と。Just (*3) を Just 5 にどう適用するか？　Nothing に適用したいときは？　[(*2),(+4)] を [1,2,3] にどう適用するか？　そのために Applicative 型クラスが導入され、次の型に答えました： 

(<*>) :: (Applicative f) => f (a -> b) -> f a -> f b

通常の値をデータ型に包む方法も見ました。1 を Just 1 や [1]、1 を結果として返すだけの IOアクションに包む。これをするのが pure。

アプリカティブ値は「文脈付きの値」、技術的に言えば 派手な (fancy) 値と捉えられます。文字 'a' はただの文字、Just 'a' は文脈付き。Char ではなく Maybe Char で「文字かもしれないし不在かもしれない」を表します。

Applicative 型クラスのおかげで、文脈付きの値に通常関数を使え、文脈が保たれる： 

ghci> (*) <$> Just 2 <*> Just 8
Just 16
ghci> (++) <$> Just "klingon" <*> Nothing
Nothing
ghci> (-) <$> [3,4] <*> [1,2,3]
[2,1,0,3,2,1]

アプリカティブとして見ると、Maybe a は失敗するかもしれない計算、[a] は複数の結果を持つ非決定的計算、IO a は副作用を持つ計算、と捉えられます。

モナドはアプリカティブファンクタの自然な拡張で、こう考えます：「文脈付きの値 m a があり、通常の a を取って文脈付きの値を返す関数があったら、どう適用するか？」　つまり a -> m b 型の関数を m a 型の値にどう適用するか。要するに次の関数： 

(>>=) :: (Monad m) => m a -> (a -> m b) -> m b

派手な値と「通常値を取って派手な値を返す関数」があるとき、派手な値を関数にどう渡すか？　これがモナドを扱うときの主な問い。m は Monad の頭文字（f と書かない）。モナドは >>= をサポートするアプリカティブファンクタにすぎません。>>= は バインド (bind) と読みます。

通常の値 a と通常関数 a -> b なら、関数を値に適用するだけ。文脈付きの値だと、ちょっと考える必要がありますが、簡単です。

Maybeから始めるモナド

モナドの大まかなイメージが掴めたところで、もう少し具体的に見ます。

驚きでもないですが、Maybe はモナドです。モナドの知識と組み合わせてみます。

アプリカティブ をしっかり理解しておいてください。各 Applicative インスタンスがどう動き、何の計算を表すかが分かっていると、モナドはその知識を強化するだけ。

Maybe a 型の値は「失敗するかもしれない文脈」付きの a 型の値。Just "dharma" は文字列 "dharma" がある、Nothing は不在（または計算が失敗した）。

Maybe をファンクタとして見ると、fmap は Just なら中身に関数を適用し、Nothing ならそのまま： 

ghci> fmap (++"!") (Just "wisdom")
Just "wisdom!"
ghci> fmap (++"!") Nothing
Nothing

アプリカティブとしても同様。ただしアプリカティブでは関数も包まれている。Maybe をアプリカティブとして使うと、<*> は両方が Just のときだけ Just、いずれかが Nothing なら Nothing。関数か対象が無ければ何も生み出せないので失敗を伝播： 

ghci> Just (+3) <*> Just 3
Just 6
ghci> Nothing <*> Just "greed"
Nothing
ghci> Just ord <*> Nothing
Nothing

アプリカティブスタイルで通常関数を Maybe 値に作用させるのも同様。すべて Just でなければ Nothing： 

ghci> max <$> Just 3 <*> Just 6
Just 6
ghci> max <$> Just 3 <*> Nothing
Nothing

では Maybe での >>= を考えます。モナド値と「通常値を取ってモナド値を返す関数」を取り、関数を適用する。文脈を考慮する必要があります。

Maybe では >>= は Maybe a 値と a -> Maybe b 関数を取って、関数を Maybe a に適用。アプリカティブの直感を活かします。\x -> Just (x+1) という関数：数を取って 1 足して Just に包む： 

ghci> (\x -> Just (x+1)) 1
Just 2
ghci> (\x -> Just (x+1)) 100
Just 101

1 を渡せば Just 2、100 なら Just 101。さて、Maybe をこの関数にどう渡すか？　アプリカティブファンクタとしての Maybe の動きを思い出せば簡単。Just なら中身を取り出して関数に適用、Nothing なら結果は Nothing。

仮に applyMaybe と呼びましょう： 

applyMaybe :: Maybe a -> (a -> Maybe b) -> Maybe b
applyMaybe Nothing f  = Nothing
applyMaybe (Just x) f = f x

中置関数として使い、左に Maybe、右に関数： 

ghci> Just 3 `applyMaybe` \x -> Just (x+1)
Just 4
ghci> Just "smile" `applyMaybe` \x -> Just (x ++ " :)")
Just "smile :)"
ghci> Nothing `applyMaybe` \x -> Just (x+1)
Nothing
ghci> Nothing `applyMaybe` \x -> Just (x ++ " :)")
Nothing

Just 値と関数なら、Just 内の値に関数が適用される。Nothing なら結果も Nothing。関数が Nothing を返したら？ 

ghci> Just 3 `applyMaybe` \x -> if x > 2 then Just x else Nothing
Just 3
ghci> Just 1 `applyMaybe` \x -> if x > 2 then Just x else Nothing
Nothing

予想通り。左がモナド値の Nothing なら全体が Nothing。関数が Nothing を返しても結果は Nothing。アプリカティブとしての Maybe でどこかに Nothing があれば全体 Nothing になるのと似ています。

Maybe で「派手な値を取って、通常値を取り派手な値を返す関数に渡す」方法が分かりました。Maybe 値が「失敗するかもしれない計算」を表すという文脈を意識して。

これが何の役に？　アプリカティブファンクタの方が強そうに見えますが、モナドはアプリカティブの強化版なので同じことができ、さらにアプリカティブにできないクールなこともできます。

すぐに Maybe に戻りますが、まずモナドの型クラスを見ます。

Monad型クラス

ファンクタには Functor、アプリカティブには Applicative、モナドには Monad！　定義： 

class Monad m where
    return :: a -> m a

    (>>=) :: m a -> (a -> m b) -> m b

    (>>) :: m a -> m b -> m b
    x >> y = x >>= \_ -> y

    fail :: String -> m a
    fail msg = error msg

1 行目は class Monad m where。モナドはアプリカティブの強化版なので class (Applicative m) => Monad m where と書くべきでは？　はい、本来そうあるべきですが、Haskell が作られた当時アプリカティブが思いつかれていなかったので。それでも全てのモナドはアプリカティブファンクタです（クラス宣言には書いてなくても）。

1 つ目の関数は return、pure と同じ別名。型 (Monad m) => a -> m a。値を取って最小デフォルト文脈に包む。Maybe なら Just に包む。I/O では値を結果として持つだけで何もしない IOアクションを作るのに使いました。

リマインダ：return は他の言語の return とは全く違います。関数実行を終えるのではなく、ただ値を文脈に包むだけ。

次は >>=（バインド）。関数適用に似ていますが、通常値ではなくモナド値を、通常値を取ってモナド値を返す関数に渡します。

>> はデフォルト実装があり、自分で実装することはほぼないので深入りしません。

fail は明示的には使いません。Haskell が後で見るモナドの特殊構文で失敗を扱うのに使います。今は気にしなくて OK。

では Maybe の Monad インスタンス： 

instance Monad Maybe where
    return x = Just x
    Nothing >>= f = Nothing
    Just x >>= f  = f x
    fail _ = Nothing

return は pure と同じで Just に包むだけ。

>>= は applyMaybe と同じ。Nothing なら Nothing、Just なら中身に f を適用。

モナドとしての Maybe で遊んでみます： 

ghci> return "WHAT" :: Maybe String
Just "WHAT"
ghci> Just 9 >>= \x -> return (x*10)
Just 90
ghci> Nothing >>= \x -> return (x*10)
Nothing

1 行目は pure と同じ。残り 2 行は >>= の使用例。

Just 9 を \x -> return (x*10) に渡したとき、x は 9 になります。パターンマッチなしで Maybe から値を取り出せたかのよう。Nothing のときは Nothing になるので、文脈は失われていません。

綱渡り

Maybe a を a -> Maybe b 型の関数に文脈を考慮しつつ渡す方法が分かったので、>>= を繰り返し使って複数の Maybe a 値の計算を扱います。

ピエールは魚の養殖場の仕事に休みを取り、綱渡りを試しています。腕は悪くないですが 1 つ問題が：鳥がバランス棒に着地し続けるのです！　鳥たちは仲間とおしゃべりして飛んでいきます。左右の鳥の数が常に等しいなら気にならないのですが、片側に偏るとピエールはバランスを崩してこっぱずかしい目に遭います（セーフティネットがあります）。

左右の鳥の数の差が 3 以下ならバランスが保てるとします。右に 1 羽、左に 4 羽なら OK。でも左に 5 羽になるとピエールは落下。

鳥が来る・去るをシミュレートして、ピエールが何回かの飛来・飛び去り後にまだ立っているか見ます。たとえば左に 1 羽、右に 4 羽、左の 1 羽が飛び去る、と。

バランス棒は整数のペアで表します。第 1 要素が左の鳥数、第 2 要素が右の鳥数： 

type Birds = Int
type Pole = (Birds,Birds)

Int の型シノニム Birds と、(Birds,Birds) のシノニム Pole（ポーランド人の意味ではない）。

鳥の数を片側に着地させる関数： 

landLeft :: Birds -> Pole -> Pole
landLeft n (left,right) = (left + n,right)

landRight :: Birds -> Pole -> Pole
landRight n (left,right) = (left,right + n)

シンプル。試します： 

ghci> landLeft 2 (0,0)
(2,0)
ghci> landRight 1 (1,2)
(1,3)
ghci> landRight (-1) (1,2)
(1,1)

鳥が飛び去るのは負の数。Pole を返すので連鎖できます： 

ghci> landLeft 2 (landRight 1 (landLeft 1 (0,0)))
(3,1)

landLeft 1 を (0,0) に → (1,0)。右に 1 → (1,1)。左に 2 → (3,1)。関数を先に書く形式ですが、棒を先に書いて着地関数を後にしたい。こんな関数を作れば： 

x -: f = f x

引数を先、関数を後で書けます： 

ghci> 100 -: (*3)
300
ghci> True -: not
False
ghci> (0,0) -: landLeft 2
(2,0)

これで読みやすく： 

ghci> (0,0) -: landLeft 1 -: landRight 1 -: landLeft 2
(3,1)

クール！　(0,0) から始まり、左に 1、右に 1、左に 2、と分かりやすい。

順調ですが、片側に 10 羽来たら？ 

ghci> landLeft 10 (0,3)
(10,3)

左 10 で右 3？　ピエールは確実に落下！　でもこういう連鎖だったら？ 

ghci> (0,0) -: landLeft 1 -: landRight 4 -: landLeft (-1) -: landRight (-2)
(0,2)

一見大丈夫そうですが、追っていくと途中で右に 4 羽、左に 0 羽の瞬間がある！　修正には landLeft・landRight に失敗を導入します。バランスが取れているなら新しい棒、偏ったら失敗。失敗の文脈を表すには Maybe ！ 

landLeft :: Birds -> Pole -> Maybe Pole
landLeft n (left,right)
    | abs ((left + n) - right) < 4 = Just (left + n, right)
    | otherwise                    = Nothing

landRight :: Birds -> Pole -> Maybe Pole
landRight n (left,right)
    | abs (left - (right + n)) < 4 = Just (left, right + n)
    | otherwise                    = Nothing

Pole の代わりに Maybe Pole を返します。差が 4 未満なら Just、そうでなければ Nothing。

試します： 

ghci> landLeft 2 (0,0)
Just (2,0)
ghci> landLeft 10 (0,3)
Nothing

バランスが保たれれば Just、崩れれば Nothing。ただ連鎖できなくなりました。landLeft 1 (landRight 1 (0,0)) はもう動きません。landRight 1 (0,0) は Maybe Pole を返しますが、landLeft 1 は Pole を取るので。

Maybe Pole を取って Pole を取って Maybe Pole を返す関数に渡す方法が要る。それが >>=！ 

ghci> landRight 1 (0,0) >>= landLeft 2
Just (2,1)

landLeft 2 の型は Pole -> Maybe Pole。Maybe Pole の結果を直接渡せないので >>= で。Nothing を渡せば結果も Nothing： 

ghci> Nothing >>= landLeft 2
Nothing

これで失敗するかもしれない着地を連鎖できます： 

ghci> return (0,0) >>= landRight 2 >>= landLeft 2 >>= landRight 2
Just (2,4)

最初に return で (0,0) を Just に。landRight 2 から (0,0) に直接適用しても同じですが、すべて >>= で揃えると一貫します。Just (0,0) → Just (0,2) → Just (2,2) → ...

失敗を導入する前の例： 

ghci> (0,0) -: landLeft 1 -: landRight 4 -: landLeft (-1) -: landRight (-2)
(0,2)

これは途中でバランス崩壊を反映しなかった。モナド適用（>>=）で： 

ghci> return (0,0) >>= landLeft 1 >>= landRight 4 >>= landLeft (-1) >>= landRight (-2)
Nothing

最終結果は失敗、期待通り！　追ってみます：return で (0,0) を Just (0,0) に。Just (0,0) >>= landLeft 1 → Just (1,0)（バランス OK）。Just (1,0) >>= landRight 4 → Just (1,4)（ギリギリ OK）。Just (1,4) >>= landLeft (-1) → Nothing（バランス崩壊）。Nothing >>= landRight (-2) → Nothing。

これは Maybe をアプリカティブとしては実現できません。アプリカティブはアプリカティブ値同士の相互作用が薄く、関数の引数として並列に扱うだけ。ここでは各ステップが前のステップの結果に依存します。各着地で前回の結果を見てバランスをチェックする必要があります。

バランス棒の鳥の数を無視して必ず転倒させる関数も作れます。banana： 

banana :: Pole -> Maybe Pole
banana _ = Nothing

鳥の着地と連鎖： 

ghci> return (0,0) >>= landLeft 1 >>= banana >>= landRight 1
Nothing

Just (1,0) が banana に渡って Nothing、以降ずっと Nothing。

入力を無視して固定モナド値を返す関数の代わりに >> が使えます： 

(>>) :: (Monad m) => m a -> m b -> m b
m >> n = m >>= \_ -> n

モナドでは文脈と意味も考慮されます。Maybe での >>： 

ghci> Nothing >> Just 3
Nothing
ghci> Just 3 >> Just 4
Just 4
ghci> Just 3 >> Nothing
Nothing

>> を >>= \_ -> に置き換えれば動きが分かります。

連鎖中の banana を >> Nothing に置き換えられます： 

ghci> return (0,0) >>= landLeft 1 >> Nothing >>= landRight 1
Nothing

保証された明確な失敗！

Maybe を「失敗の文脈付き値」として扱わない場合の書き方も見ます： 

routine :: Maybe Pole
routine = case landLeft 1 (0,0) of
    Nothing -> Nothing
    Just pole1 -> case landRight 4 pole1 of
        Nothing -> Nothing
        Just pole2 -> case landLeft 2 pole2 of
            Nothing -> Nothing
            Just pole3 -> landLeft 1 pole3

左に着地して、失敗・成功を見て、成功なら次へ。これを >>= でスマートな連鎖に変換できるのが Maybe モナドの威力。

>>= の Maybe 実装は「Nothing なら即 Nothing、そうでなければ Just 内を取り出して進む」というロジック。

失敗をサポートする Maybe に値を変えて >>= を使うことで、失敗ハンドリングがほぼ無料で得られました。>>= が文脈を保つように設計されているからです。

do記法

Haskell ではモナドが便利すぎて専用構文 do 記法があります。I/O のときに見ましたが、実は I/O 専用ではなく任意のモナドで使えます。原則は同じ：モナド値を順次糊付け。do 記法の動きと有用性を見ます。

お馴染みのモナド適用の例： 

ghci> Just 3 >>= (\x -> Just (show x ++ "!"))
Just "3!"

モナド値を関数に渡すだけ。ラムダ内で x は 3 になり、もうモナド値ではなく通常値。>>= をネスト： 

ghci> Just 3 >>= (\x -> Just "!" >>= (\y -> Just (show x ++ y)))
Just "3!"

>>= のネスト！　外側で Just "!" を内側のラムダに渡し、y は "!"、x は外側から 3。これは次の式に似ています： 

ghci> let x = 3; y = "!" in show x ++ y
"3!"

違いは前者の値がモナド値（失敗の文脈付き）であること。どこかを失敗にすれば： 

ghci> Nothing >>= (\x -> Just "!" >>= (\y -> Just (show x ++ y)))
Nothing
ghci> Just 3 >>= (\x -> Nothing >>= (\y -> Just (show x ++ y)))
Nothing
ghci> Just 3 >>= (\x -> Just "!" >>= (\y -> Nothing))
Nothing

let の値の代わりにモナド値を扱っているようなもの。

スクリプトに書いて各 Maybe 値を 1 行に： 

foo :: Maybe String
foo = Just 3   >>= (\x ->
      Just "!" >>= (\y ->
      Just (show x ++ y)))

ラムダの嵐を避けるために do 記法があります： 

foo :: Maybe String
foo = do
    x <- Just 3
    y <- Just "!"
    Just (show x ++ y)

Maybe 値からの値の取り出しを、各ステップで Just・Nothing をチェックせずにできるようになった！　どれかが Nothing なら全体が Nothing。>>= が文脈を扱ってくれるので、私たちは値を取り出せばよい。do 式は単にモナド値の連鎖の別構文。

do 式の各行はモナド値。結果を見るには <-。Maybe String を <- で束縛すれば変数は String（>>= でラムダに渡したのと同じ）。最後のモナド値（Just (show x ++ y) など）は <- で束縛できません（>>= 連鎖に翻訳すると意味をなさないので）。最後のモナド値が全体の結果になります。

例： 

ghci> Just 9 >>= (\x -> Just (x > 8))
Just True

do 記法で： 

marySue :: Maybe Bool
marySue = do
    x <- Just 9
    Just (x > 8)

全体の結果が最後のモナド値の結果になることが分かります。

綱渡りも do 記法で： 

routine :: Maybe Pole
routine = do
    start <- return (0,0)
    first <- landLeft 2 start
    second <- landRight 2 first
    landLeft 1 second

動きを確認： 

ghci> routine
Just (3,2)

成功！　>>= で書いていたときは return (0,0) >>= landLeft 2 でしたが、do では各行がモナド値である必要があるので前の Pole を明示的に渡しています。start は (0,0)、first は (2,0)、と。

do 式は行ごとに書くので命令型に見えますが、実際は順次的：各行の値が前の結果と文脈に依存します。

Maybe のモナド性を使わない場合：

routine :: Maybe Pole
routine =
    case Just (0,0) of
        Nothing -> Nothing
        Just start -> case landLeft 2 start of
            Nothing -> Nothing
            Just first -> case landRight 2 first of
                Nothing -> Nothing
                Just second -> landLeft 1 second

成功時に Just (0,0) 内のタプルが start に、landLeft 2 start の結果が first に、と。

do 記法でピエールにバナナの皮を投げる： 

routine :: Maybe Pole
routine = do
    start <- return (0,0)
    first <- landLeft 2 start
    Nothing
    second <- landRight 2 first
    landLeft 1 second

do で <- なしの行はモナド値の後に >> を置くのと同じ。結果を無視する。_ <- Nothing と書くより綺麗。

do 記法と >>= の使い分けはあなた次第。各ステップが前の結果に強く依存するならこの例は >>= が向くかも。

do 記法でモナド値を名前に束縛するときパターンマッチが使えます： 

justH :: Maybe Char
justH = do
    (x:xs) <- Just "hello"
    return x

"hello" の最初の文字を取り出して結果に。justH は Just 'h'。

パターンマッチが失敗したら？　関数では次のパターンへ、すべて失敗するとエラーで落ちる。let でも即エラー。do では fail 関数が呼ばれます。Monad 型クラスの一部で、現在のモナドの文脈で失敗を表現できます。デフォルト： 

fail :: (Monad m) => String -> m a
fail msg = error msg

デフォルトでは落ちますが、失敗の文脈を持つモナド（Maybe など）は独自実装します。Maybe： 

fail _ = Nothing

エラーメッセージを無視して Nothing。do 記法でパターンマッチが失敗すると全体が Nothing になり、プログラムがクラッシュしません。 

wopwop :: Maybe Char
wopwop = do
    (x:xs) <- Just ""
    return x

パターンマッチ失敗で Nothing： 

ghci> wopwop
Nothing

モナドの文脈での失敗で済み、プログラム全体が落ちません。

リストモナド

Maybe 値を「失敗の文脈付き」として、>>= で関数に渡して失敗ハンドリングしました。次はリストのモナド性で「非決定性」を扱います。

リストはアプリカティブとして使うと非決定的な値を表す、と既に話しました。5 は決定的で結果 1 つ。[3,8,9] は複数の結果を持ち、「同時に複数の値である値」と捉えられます。アプリカティブで分かりやすく： 

ghci> (*) <$> [1,2,3] <*> [10,100,1000]
[10,100,1000,20,200,2000,30,300,3000]

左の要素と右の要素を掛けるすべての組み合わせを含む結果。非決定性では選択肢を全部試すので、結果も非決定的でさらに多くの結果を持ちます。

非決定性の文脈はモナドにきれいに翻訳されます。リストの Monad インスタンス： 

instance Monad [] where
    return x = [x]
    xs >>= f = concat (map f xs)
    fail _ = []

return は pure と同じで、値をシングルトンリストに包む。

>>= をリストで動かしてみます： 

ghci> [3,4,5] >>= \x -> [x,-x]
[3,-3,4,-4,5,-5]

Maybe の >>= が失敗を扱ったように、リストでは非決定性を扱います。[3,4,5] は非決定的で、関数 \x -> [x,-x] も非決定的。結果も非決定的で、[3,4,5] から取り出した各要素を関数に渡したすべての結果を含みます。

実装をたどると：[3,4,5] にラムダを map： 

[[3,-3],[4,-4],[5,-5]]

リストのリスト。これを平坦化して完成！

非決定性は失敗もサポート。空リスト [] は Nothing 相当（結果の不在）。fail は空リストです： 

ghci> [] >>= \x -> ["bad","mad","rad"]
[]
ghci> [1,2,3] >>= \x -> []
[]

1 つ目：空リストは要素がないので結果も空。2 つ目：各要素を関数に渡しても空リストを返すので結果は空。

Maybe と同様、複数のリストを >>= で連鎖して非決定性を伝播： 

ghci> [1,2] >>= \n -> ['a','b'] >>= \ch -> return (n,ch)
[(1,'a'),(1,'b'),(2,'a'),(2,'b')]

[1,2] が n に、['a','b'] が ch に。return (n,ch) は [(n,ch)] に対応。最小文脈に置く（追加の非決定性なし）。「[1,2] の各要素について、['a','b'] の各要素についてタプルを作る」と読めます。

return は値を最小文脈に包むだけで追加効果（Maybe での失敗、リストでの非決定性）はなく、結果として値を提示します。

非決定的な値が相互作用するとき、計算は「リストの各結果が分岐の枝」というツリーと見ることができます。

先ほどの式を do 記法で： 

listOfTuples :: [(Int,Char)]
listOfTuples = do
    n <- [1,2]
    ch <- ['a','b']
    return (n,ch)

n が [1,2] から、ch が ['a','b'] から、と分かりやすい。Maybe と同じく、モナド値から要素を取り出して通常値として扱い、>>= が文脈（ここでは非決定性）を扱う。

do 記法でのリスト、似たものを思い出させます： 

ghci> [ (n,ch) | n <- [1,2], ch <- ['a','b'] ]
[(1,'a'),(1,'b'),(2,'a'),(2,'b')]

リスト内包表記！　do 記法と同じく n が [1,2] から、ch が ['a','b'] から取り、最後にデフォルト文脈に (n,ch) を置く。リスト内包表記は出力部分がそれをやってくれるので return が不要。

リスト内包表記はリストをモナドとして使う糖衣構文。

リスト内包表記の出力フィルタ：1〜50 から数字に 7 を含むものを： 

ghci> [ x | x <- [1..50], '7' `elem` show x ]
[7,17,27,37,47]

show で文字列化して '7' の有無をチェック。リストモナドでのフィルタは guard と MonadPlus 型クラスで実現します： 

class Monad m => MonadPlus m where
    mzero :: m a
    mplus :: m a -> m a -> m a

mzero は Monoid の mempty、mplus は mappend 相当。リストはモナドかつモノイドなのでインスタンスにできます： 

instance MonadPlus [] where
    mzero = []
    mplus = (++)

リストでは mzero は結果なしの非決定的計算（失敗）。mplus は 2 つの非決定的値を 1 つに。guard： 

guard :: (MonadPlus m) => Bool -> m ()
guard True = return ()
guard False = mzero

真偽値を取り、True なら () を最小成功文脈に、False なら失敗のモナド値。動作： 

ghci> guard (5 > 2) :: Maybe ()
Just ()
ghci> guard (1 > 2) :: Maybe ()
Nothing
ghci> guard (5 > 2) :: [()]
[()]
ghci> guard (1 > 2) :: [()]
[]

リストモナドでフィルタに使う： 

ghci> [1..50] >>= (\x -> guard ('7' `elem` show x) >> return x)
[7,17,27,37,47]

リスト内包表記と同じ結果。guard と >> の働き： 

ghci> guard (5 > 2) >> return "cool" :: [String]
["cool"]
ghci> guard (1 > 2) >> return "cool" :: [String]
[]

guard 成功なら結果は空タプル、>> でそれを無視して別の結果を提示。失敗なら以降の return も失敗（空リストに >>= しても空）。guard は「真偽値が偽なら失敗、真ならダミー () の成功値」を作って計算続行を許す関数。

do 記法で： 

sevensOnly :: [Int]
sevensOnly = do
    x <- [1..50]
    guard ('7' `elem` show x)
    return x

最後の return x を忘れると空タプルのリストになります。 

ghci> [ x | x <- [1..50], '7' `elem` show x ]
[7,17,27,37,47]

リスト内包表記のフィルタは guard と同じ。

騎士の旅

非決定性で解くにふさわしい問題。チェス盤に騎士（ナイト）が 1 つ。3 手で特定の位置に到達できるか調べたい。位置は数のペアで（列, 行）。

位置の型シノニム： 

type KnightPos = (Int,Int)

(6,2) から始めて 3 手で (6,1) に到達できるか？　次の手をどう選ぶか？　全部選びます！　非決定性！　位置から次に動ける位置すべてを返す関数： 

moveKnight :: KnightPos -> [KnightPos]
moveKnight (c,r) = do
    (c',r') <- [(c+2,r-1),(c+2,r+1),(c-2,r-1),(c-2,r+1)
               ,(c+1,r-2),(c+1,r+2),(c-1,r-2),(c-1,r+2)
               ]
    guard (c' `elem` [1..8] && r' `elem` [1..8])
    return (c',r')

ナイトは縦・横に 1 歩、もう一方に 2 歩。各候補について guard で盤上か確認、外なら空リストで失敗、return (c',r') をスキップ。

filter でも同じ： 

moveKnight :: KnightPos -> [KnightPos]
moveKnight (c,r) = filter onBoard
    [(c+2,r-1),(c+2,r+1),(c-2,r-1),(c-2,r+1)
    ,(c+1,r-2),(c+1,r+2),(c-1,r-2),(c-1,r+2)
    ]
    where onBoard (c,r) = c `elem` [1..8] && r `elem` [1..8]

好きな方を選んでください。試します： 

ghci> moveKnight (6,2)
[(8,1),(8,3),(4,1),(4,3),(7,4),(5,4)]
ghci> moveKnight (8,1)
[(6,2),(7,3)]

動きました！　非決定的な「次の位置」ができたので、>>= でさらに moveKnight に渡せます。3 手で到達可能な位置を返す関数： 

in3 :: KnightPos -> [KnightPos]
in3 start = do
    first <- moveKnight start
    second <- moveKnight first
    moveKnight second

(6,2) から呼ぶと結果リストはかなり大きい（同じ位置が複数経路で出るので重複あり）。do なし版： 

in3 start = return start >>= moveKnight >>= moveKnight >>= moveKnight

1 回目の >>= ですべての初手、2 回目で各初手から次の手、3 回目で最後の手。

return でデフォルト文脈に置いて >>= で関数に渡すのは、関数を直接呼ぶのと同じですが、スタイルとしてこう書きました。

2 つの位置を取って 3 手で到達できるかを返す関数： 

canReachIn3 :: KnightPos -> KnightPos -> Bool
canReachIn3 start end = end `elem` in3 start

3 手で到達可能なすべての位置に目的地が含まれるか。(6,2) から (6,1)： 

ghci> (6,2) `canReachIn3` (6,1)
True

はい！　(6,2) から (7,3)： 

ghci> (6,2) `canReachIn3` (7,3)
False

ダメ！　練習として、到達可能なときどの手順を踏むかも教えるよう変えてみてください。後で手数を引数化する方法も見ます。

モナド則

ファンクタ・アプリカティブと同様、モナドにも法則があり、すべてのモナドインスタンスが守るべきです。Monad インスタンスにしただけではモナドではなく、法則を満たして初めて真のモナド。法則のおかげで型と振る舞いについて妥当な仮定を置けます。

Haskell は型さえ合えば任意の型をインスタンスにできるが、法則は自動チェックしてくれません。標準ライブラリの型は法則を満たすと信頼でき、自前モナドを作るときは手で確認します。難しくないので心配無用。

左恒等性

第 1 のモナド則：値を return でデフォルト文脈に置いてから >>= で関数に渡すのは、値に直接関数を適用するのと同じ。形式的には：

  • return x >>= f は f x と同じ

モナド値を「文脈付き値」、return を「最小文脈に値を置く」と捉えれば、文脈が最小なら関数を直接適用するのと変わらない、というのは納得です。

Maybe モナドでは return は Just。Maybe は失敗の文脈なので、値があれば成功計算として扱うのが筋： 

ghci> return 3 >>= (\x -> Just (x+100000))
Just 100003
ghci> (\x -> Just (x+100000)) 3
Just 100003

リストモナドでは return はシングルトンリストに包む。>>= はリスト要素に関数を適用しますが、シングルトンなので結果は関数を直接適用したのと同じ： 

ghci> return "WoM" >>= (\x -> [x,x,x])
["WoM","WoM","WoM"]
ghci> (\x -> [x,x,x]) "WoM"
["WoM","WoM","WoM"]

IO でも return は副作用なしで値を結果として持つ IOアクションを作るので法則を満たします。

右恒等性

第 2 の法則：モナド値を >>= で return に渡すと元のモナド値と同じ。形式的：

  • m >>= return は m と同じ

直感が働きにくいかもしれませんが、return は最小文脈に値を置くので、Maybe なら失敗を導入しないし、リストなら追加の非決定性を導入しない。試します： 

ghci> Just "move on up" >>= (\x -> return x)
Just "move on up"
ghci> [1,2,3,4] >>= (\x -> return x)
[1,2,3,4]
ghci> putStrLn "Wah!" >>= (\x -> return x)
Wah!

リスト例の >>= は xs >>= f = concat (map f xs)。[1,2,3,4] に return を map すると [[1],[2],[3],[4]]、concat で元のリストに戻る。

左恒等性・右恒等性は return の振る舞いを規定する法則。通常値をモナド値に持ち上げる重要な関数なので、余計なことをしないことが大事。

結合則

最後の法則：>>= によるモナド関数適用の連鎖は、入れ子の付け方に依らない。形式的：

  • (m >>= f) >>= g は m >>= (\x -> f x >>= g) と同じ

モナド値 m、モナド関数 f・g。(m >>= f) >>= g は m を f に渡し、結果を g に。m >>= (\x -> f x >>= g) は m を「f x の結果を g に渡す関数」に渡す。等価性が見えにくいので例で：

ピエールの綱渡り： 

ghci> return (0,0) >>= landRight 2 >>= landLeft 2 >>= landRight 2
Just (2,4)

Just (0,0) から始めて次々モナド関数に渡す。明示的に括弧を付けると： 

ghci> ((return (0,0) >>= landRight 2) >>= landLeft 2) >>= landRight 2
Just (2,4)

あるいは： 

return (0,0) >>= (\x ->
landRight 2 x >>= (\y ->
landLeft 2 y >>= (\z ->
landRight 2 z)))

return (0,0) = Just (0,0)、x は (0,0)。landRight が呼ばれ Just (0,2)、y = (0,2)……最終的に Just (2,4)。

入れ子の付け方は問題ではなく、意味が問題。別の見方：関数 f・g の合成： 

(.) :: (b -> c) -> (a -> b) -> (a -> c)
f . g = (\x -> f (g x))

a -> b と b -> c を合成して a -> c。モナド関数なら（戻り値がモナド値なら）、a -> m b の結果を b -> m c にそのまま渡せません（後者は通常の b を取る）。>>= で実現： 

(<=<) :: (Monad m) => (b -> m c) -> (a -> m b) -> (a -> m c)
f <=< g = (\x -> g x >>= f)

モナド関数を合成： 

ghci> let f x = [x,-x]
ghci> let g x = [x*3,x*2]
ghci> let h = f <=< g
ghci> h 3
[9,-9,6,-6]

クール。結合則は <=< で書くと f <=< (g <=< h) = (f <=< g) <=< h。モナドでは入れ子の順序は問題でない、と言い換えています。

左恒等性・右恒等性も <=< で書けば、左恒等性は「任意のモナド関数 f について f <=< return = f」、右恒等性は「return <=< f = f」。

通常関数で (f . g) . h = f . (g . h)、f . id = f、id . f = f なのと似ています。
