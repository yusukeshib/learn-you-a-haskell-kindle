ファンクタ、アプリカティブファンクタ、モノイド

Haskell の純粋性、高階関数、パラメータ化された代数的データ型、型クラスの組み合わせのおかげで、他言語よりはるかに高水準な多相を実現できます。型を巨大な階層に当てはめて考えるのではなく、「何のように振る舞えるか」を考え、適切な型クラスに繋ぎます。Int はいろいろなものとして振る舞えます — 等価性を持つもの、順序を持つもの、列挙可能なもの、など。

型クラスは「開いて」います。つまり自分でデータ型を作って「何のように振る舞えるか」を考え、対応する型クラスに繋げます。Haskell の優れた型システムのおかげで、関数の型宣言からたくさんの情報を読み取れるので、汎用で抽象的な振る舞いを定義する型クラスを作れます。等価性比較や順序比較の型クラスはこれまで見てきました。とても抽象的でエレガントな振る舞いですが、人生の大半それと付き合ってきたので「特別」とは思いませんね。最近、map できるものとしてファンクタに出会いました。これは型クラスが表現できる、有用かつかなり抽象的な性質の好例。この章ではファンクタをさらに掘り下げ、もう少し強力で便利な「アプリカティブファンクタ」も見ます。さらにモノイドも見ます。靴下のようなものです。

ファンクタ再び

ファンクタは 前章の小さなセクション ですでに扱いました。読んでいなければさらっと目を通しておくのがよいでしょう。あるいは読んだことにしてもOK。

とはいえ簡単な復習：ファンクタは map できるもの、たとえばリスト、Maybe、木など。Haskell では Functor 型クラスで表現され、唯一のメソッドが fmap（型 fmap :: (a -> b) -> f a -> f b）。a を取って b を返す関数と、中に a（あるいは複数）を持つ箱をくれたら、中に b（あるいは複数）を持つ箱を返す。要するに箱の中の要素に関数を適用するイメージ。

アドバイス：　ファンクタの直感を掴むために箱のアナロジーがよく使われ、後でアプリカティブファンクタやモナドにも同じアナロジーを使います。最初の理解を助ける良いアナロジーですが、文字どおり受け取りすぎないでください。一部のファンクタでは箱のアナロジーを引き伸ばしすぎないと成り立ちません。より正確には、ファンクタは 計算文脈 といえます。文脈とは「計算が値を持つかもしれないし失敗したかもしれない」（Maybe、Either a）、「複数の値があるかもしれない」（リスト）など。

型コンストラクタを Functor のインスタンスにするには、カインドが * -> *、つまり具体型を 1 つ取る必要があります。たとえば Maybe は型引数 1 つを取って Maybe Int や Maybe String のような具体型を生むのでインスタンスにできます。Either のように 2 つ取る型コンストラクタは、引数 1 つだけ取る形まで部分適用が必要です。instance Functor Either where とは書けず、instance Functor (Either a) where と書きます。Either a 専用の fmap なら型は fmap :: (b -> c) -> Either a b -> Either a c。Either a 部分は固定で、Either a は型引数 1 つを取ります。Either なら 2 つ取るので fmap :: (b -> c) -> Either b -> Either c は意味をなしません。

これまで多くの型（実際は型コンストラクタ）が Functor のインスタンスでした：[]、Maybe、Either a、自前の Tree 型。それらに関数を map して大いに役立てました。このセクションではあと 2 つのインスタンス、IO と (->) r を見ます。

IO String 型の値は、実行されると現実世界に出ていって文字列を取ってくる IOアクションでした。do 記法の <- でその結果を名前に束縛できます。IOアクションは「外の世界から値を取ってくる小さな足の付いた箱」と言いました。中身を覗いた後は IO に再び包み直す必要があります。この箱のアナロジーから、IO がファンクタとして振る舞えるのが分かります。

IO がどう Functor のインスタンスかを見ます。fmap で IOアクションに関数を map すると、結果値に関数が適用される IOアクションが返るようにします。

instance Functor IO where
    fmap f action = do
        result <- action
        return (f result)

IOアクションへの map 結果は IOアクションなので、do 記法で 2 つのアクションを糊付けして新しいアクションを作ります。実装では、まず元の IOアクションを実行して結果を result に束縛し、次に return (f result)。return は何もしないが結果として何かを提示する IOアクションを作る関数。do ブロックが返す IOアクションの結果値は最後のアクションの結果と同じになるので、return で「f result を結果に持つ何もしない IOアクション」を作って末尾に置いています。

動かして直感を掴みましょう。簡単です：

main = do line <- getLine
          let line' = reverse line
          putStrLn $ "You said " ++ line' ++ " backwards!"
          putStrLn $ "Yes, you really said" ++ line' ++ " backwards!"

ユーザーから 1 行受け取って逆順にして返すプログラム。fmap で書き直すと：

main = do line <- fmap reverse getLine
          putStrLn $ "You said " ++ line ++ " backwards!"
          putStrLn $ "Yes, you really said" ++ line ++ " backwards!"

Just "blah" に reverse を fmap して Just "halb" を得るのと同様、getLine に reverse を fmap できます。getLine は IO String 型の IOアクション、reverse を map すると、現実世界に行って 1 行取ってきて、結果に reverse を適用する IOアクションになります。Maybe 箱の中身に関数を適用するように、IO 箱の中身にも関数を適用できます。違いは、IO は中身を取るために現実世界に行く必要がある点。<- で名前に束縛したとき、その名前には既に reverse 適用済みの結果が入っています。

IOアクション fmap (++"!") getLine は getLine と同じ動きですが、結果に常に "!" が末尾に付きます！

fmap を IO に限定した型は fmap :: (a -> b) -> IO a -> IO b。関数と IOアクションを取って、結果に関数が適用される新しい IOアクションを返します。

IOアクションの結果を名前に束縛してから関数を適用して別名を付ける、ということを繰り返しているなら、fmap を使うほうがきれい。ファンクタ内のデータに複数の変換を適用したいときは、トップレベル関数を作るかラムダか、理想的には関数合成を使います：

import Data.Char
import Data.List

main = do line <- fmap (intersperse '-' . reverse . map toUpper) getLine
          putStrLn line

$ runhaskell fmapping_io.hs
hello there
E-R-E-H-T- -O-L-L-E-H

intersperse '-' . reverse . map toUpper は、文字列を取って toUpper を map し、reverse し、intersperse '-' を適用する関数。(\xs -> intersperse '-' (reverse (map toUpper xs))) と書くより綺麗。

もう 1 つの Functor インスタンスはずっと使ってきたのに気づかなかった (->) r。混乱しますね、(->) r って何でしょう？　r -> a という関数型は (->) r a とも書けます（2 + 3 を (+) 2 3 と書くのと同じ）。(->) r a として見ると、(->) は Either のような「2 つの型引数を取る型コンストラクタ」だと分かります。Functor のインスタンスにするには型引数 1 つにする必要があるので (->) は無理ですが、(->) r に部分適用すれば問題ありません。型コンストラクタをセクションで部分適用できる構文があれば（+ を (2+) と部分適用するように）、(->) r を (r ->) と書けたでしょう。関数がどうファンクタになるのか、Control.Monad.Instances にある実装を見てみます。

何でも取って何でも返す関数は a -> b と書くのが普通です。r -> a も同じで、型変数の文字が違うだけ。
instance Functor ((->) r) where
    fmap f g = (\x -> f (g x))

構文が許せばこう書けたでしょう：

instance Functor (r ->) where
    fmap f g = (\x -> f (g x))

でもダメなので前者の書き方になります。

まず fmap の型を考えます：fmap :: (a -> b) -> f a -> f b。ファンクタインスタンスの役を担う f を (->) r に置き換えると fmap :: (a -> b) -> ((->) r a) -> ((->) r b)。(->) r a と (->) r b を中置で書けば fmap :: (a -> b) -> (r -> a) -> (r -> b)。

関数に関数を map すると関数になる、というのは、Maybe に map すると Maybe、リストに map するとリストになるのと同じ。型 fmap :: (a -> b) -> (r -> a) -> (r -> b) から何が分かるでしょう？　a から b への関数と、r から a への関数を取って、r から b への関数を返す。何かに似ていませんか？　そう、関数合成！　r -> a の出力を a -> b の入力にパイプし、r -> b を得る。これがまさに関数合成。上の実装を見れば、ただの関数合成だと分かります。こう書けば一目瞭然：

instance Functor ((->) r) where
    fmap = (.)

関数に fmap するのは合成、というのが明らか。:m + Control.Monad.Instances（インスタンスはここで定義）してから関数に map して遊んでみてください。

ghci> :t fmap (*3) (+100)
fmap (*3) (+100) :: (Num a) => a -> a
ghci> fmap (*3) (+100) 1
303
ghci> (*3) `fmap` (+100) $ 1
303
ghci> (*3) . (+100) $ 1
303
ghci> fmap (show . (*3)) (*100) 1
"300"

fmap を中置で呼べば . との類似がよく分かります。2 行目では (*3) を (+100) に map し、入力を取って (+100) を呼び結果に (*3) を呼ぶ関数を作ります。それを 1 で呼んでいます。

箱のアナロジーはどう成り立つか？　ちょっと引き伸ばせば。fmap (+3) を Just 3 に使うと、Maybe を中身に (+3) を適用できる箱と見做せます。fmap (*3) (+100) は？　(+100) という関数を「いずれの結果」を含む箱と見做せます。IOアクションが現実世界に行って結果を取ってくる箱と同じ感覚。fmap (*3) を (+100) に使うと、(+100) のように動くが結果を返す前に (*3) を適用する関数ができます。これで関数に対する fmap がまさに . だと分かります。

関数に対する fmap が関数合成だ、という事実は今すぐ役に立つわけではありませんが、興味深い。「箱というより計算」（IO や (->) r）もファンクタになれる、と頭が柔らかくなります。計算に対して関数を map すると、同じ計算だが結果に関数が適用された計算が返る、というわけです。

fmap が従うべき法則に進む前に、もう一度 fmap の型を考えます：fmap :: (a -> b) -> f a -> f b。クラス制約 (Functor f) => は省略しています。カリー化された関数 を学んだとき、Haskell の関数はどれも実は引数 1 つだけ取る、と話しました。a -> b -> c は実は a 型の引数 1 つを取って b -> c 関数を返します。引数を少なく呼べば（部分適用すれば）、残りの引数を取る関数が返るのもそれゆえ。だから a -> b -> c は a -> (b -> c) と書けて、カリー化が明確になります。

同じく fmap :: (a -> b) -> (f a -> f b) と書けば、fmap は「関数とファンクタを取ってファンクタを返す関数」ではなく、「関数を取って『ファンクタを取ってファンクタを返す関数』を返す関数」と捉えられます。a -> b 関数を取って f a -> f b 関数を返す。これを関数の 持ち上げ (lifting) と呼びます。GHCi の :t で遊んでみましょう：

ghci> :t fmap (*2)
fmap (*2) :: (Num a, Functor f) => f a -> f a
ghci> :t fmap (replicate 3)
fmap (replicate 3) :: (Functor f) => f a -> f [a]

fmap (*2) は数のファンクタ f を取って数のファンクタを返す関数。リストでも Maybe でも Either String でも何でも。fmap (replicate 3) は任意の型のファンクタを取って、その型のリストのファンクタを返します。

「数のファンクタ」と言うときは「数を含むファンクタ」と思ってください。前者の方が正確で技術的に正しいですが、後者の方が分かりやすい。

たとえば fmap (++"!") を部分適用して GHCi で名前に束縛するとさらに分かります。

fmap は「関数とファンクタを取って関数を map する」関数とも、「関数を取ってファンクタ上で動くように持ち上げる」関数とも捉えられます。Haskell ではどちらも正しく等価です。

型 fmap (replicate 3) :: (Functor f) => f a -> f [a] は「任意のファンクタで動く」と言っています。実際の動作はファンクタによります。リストに使えばリストの fmap 実装（つまり map）が選ばれます。Maybe a なら Just の中身に replicate 3 を適用、Nothing ならそのまま。

ghci> fmap (replicate 3) [1,2,3,4]
[[1,1,1],[2,2,2],[3,3,3],[4,4,4]]
ghci> fmap (replicate 3) (Just 4)
Just [4,4,4]
ghci> fmap (replicate 3) (Right "blah")
Right ["blah","blah","blah"]
ghci> fmap (replicate 3) Nothing
Nothing
ghci> fmap (replicate 3) (Left "foo")
Left "foo"

次は ファンクタ則。何かがファンクタであるためには法則を満たす必要があります。すべてのファンクタには「map できる」性質と振る舞いが期待されます。fmap はファンクタに関数を map するだけ、それ以外のことはしないはず。これがファンクタ則。Functor のすべてのインスタンスが守るべき法則が 2 つあります。Haskell が自動で強制してくれないので、自分でテストする必要があります。

ファンクタ第 1 法則：ファンクタに id を map した結果は元のファンクタと同じであるべき。　もう少し形式的に書くと、fmap id = id。要するに「ファンクタに fmap id しても、ただ id を呼ぶのと同じ」。id は引数をそのまま返す恒等関数（\x -> x とも書ける）。「map できるもの」というファンクタの捉え方からすると、fmap id = id はかなり自明に思えます。

いくつかのファンクタ値で確かめます：

ghci> fmap id (Just 3)
Just 3
ghci> id (Just 3)
Just 3
ghci> fmap id [1..5]
[1,2,3,4,5]
ghci> id [1..5]
[1,2,3,4,5]
ghci> fmap id []
[]
ghci> fmap id Nothing
Nothing

Maybe の fmap 実装を見れば、第 1 法則がなぜ成り立つか分かります。

instance Functor Maybe where
    fmap f (Just x) = Just (f x)
    fmap f Nothing = Nothing

id を f 引数に充てるとイメージ。fmap id を Just x にやると Just (id x)、id は引数をそのまま返すので Just (id x) = Just x。だから Just 値コンストラクタの Maybe に id を map すると同じ値が返ります。

Nothing に id を map しても同じ値、というのは自明。これら 2 つの式から fmap id = id が成り立つと分かります。

ファンクタ第 2 法則：2 つの関数を合成してから map するのは、それぞれを順に map するのと同じ。　形式的には fmap (f . g) = fmap f . fmap g。あるいは任意のファンクタ F について fmap (f . g) F = fmap f (fmap g F)。

ある型がファンクタ則を満たすことを示せれば、map に関して他のファンクタと同じ基本的な振る舞いを期待できます。fmap を使ったとき、内部で map 以外のことが起きていないと安心できる、つまり「map できるもの」、すなわちファンクタとして振る舞う、と確信できます。第 2 法則の確認方法は、第 1 法則と同じく fmap 実装を見て確かめます。

Maybe で第 2 法則を確かめます。fmap (f . g) を Nothing に適用すると Nothing（Nothing に何を map しても Nothing）。fmap f (fmap g Nothing) も同じ理由で Nothing。OK、Nothing なら自明。

Just 何か なら？　fmap (f . g) (Just x) は実装より Just ((f . g) x)、つまり Just (f (g x))。fmap f (fmap g (Just x)) は fmap g (Just x) = Just (g x) なので fmap f (Just (g x)) = Just (f (g x))。一致！

証明で混乱したら気にしないで。関数合成 がどう動くか分かっていれば大丈夫。多くの場合、型がコンテナや関数のように振る舞うので直感的に法則が成り立つと分かります。具体的な値で試してみるだけでも、型が法則を満たすか確認できます。

では、Functor 型クラスのインスタンスではあるが法則を満たさず、本物のファンクタではない型を見てみます：

data CMaybe a = CNothing | CJust Int a deriving (Show)

C は counter。Maybe a に似ていますが Just 部分にフィールドが 2 つ。1 つ目は Int 型のカウンター、2 つ目は a。試してみます：

ghci> CNothing
CNothing
ghci> CJust 0 "haha"
CJust 0 "haha"
ghci> :t CNothing
CNothing :: CMaybe a
ghci> :t CJust 0 "haha"
CJust 0 "haha" :: CMaybe [Char]
ghci> CJust 100 [1,2,3]
CJust 100 [1,2,3]

CNothing はフィールドなし、CJust はフィールド 2 つ（整数と任意の型）。fmap したときに 2 つ目のフィールドに関数が適用され、1 つ目はカウンターが 1 増える、というインスタンスにします：

instance Functor CMaybe where
    fmap f CNothing = CNothing
    fmap f (CJust counter x) = CJust (counter+1) (f x)

Maybe のインスタンスに似ていますが、空でない箱に fmap したとき、関数を中身に適用するだけでなくカウンターを 1 増やします。一見クールで、遊べます：

ghci> fmap (++"ha") (CJust 0 "ho")
CJust 1 "hoha"
ghci> fmap (++"he") (fmap (++"ha") (CJust 0 "ho"))
CJust 2 "hohahe"
ghci> fmap (++"blah") CNothing
CNothing

ファンクタ則を満たす？　反例 1 つで否定できます：

ghci> fmap id (CJust 0 "haha")
CJust 1 "haha"
ghci> id (CJust 0 "haha")
CJust 0 "haha"

あー！　第 1 法則は「id を map するのは id を呼ぶのと同じ」ですが、CMaybe ではそうなりません。Functor のインスタンスではあるけれどファンクタではない。CMaybe をファンクタとして使うと、ファンクタ則を期待した利用者のコードが壊れる可能性があります。ファンクタを使うなら、合成してから map するか個別に map するかで結果が変わってはいけませんが、CMaybe は変わってしまいます（カウンターを覚えているので）。CMaybe をファンクタ則を満たす形にするには fmap 時に Int フィールドを変えないようにする必要があります。

最初は法則が分かりにくく不要に思えるかもしれませんが、両法則を満たすと分かれば、その型の振る舞いについて確かな仮定が置けます。fmap はただ map するだけ、それ以上のことはしない、と。これにより、より抽象的・拡張的なコードになり、任意のファンクタで信頼性高く動く関数が書けます。

標準ライブラリの Functor インスタンスはすべて法則を満たしていますが、確かめてもよいです。今後自分で Functor インスタンスを作るときも法則を確認しましょう。十分にファンクタを扱うと、共通の性質が直感的に分かるようになります。直感がなくても実装を行ごとに確認したり、反例を探したりできます。

ファンクタは「文脈の中で値を出力するもの」とも見られます。Just 3 は「値を持つかもしれないし持たないかもしれない」文脈で値 3 を出力。[1,2,3] は「複数の値か無値」の文脈で 3 つの値を出力。(+3) は引数次第で値を出力。

ファンクタを「値を出力するもの」と捉えれば、map は「ファンクタの出力に変換を取り付けて値を変える」操作。fmap (+3) [1,2,3] は [1,2,3] の出力に (+3) という変換を取り付け、リストが出力する数すべてに (+3) が適用される。関数への map も同様。fmap (+3) (*3) は (*3) のいずれの出力に (+3) という変換を取り付ける。だから関数への fmap は合成に等しい（fmap (+3) (*3) = (+3) . (*3) = \x -> ((x*3)+3)）。これが合成で起こることです。

アプリカティブファンクタ

このセクションでは、ファンクタを強化したアプリカティブファンクタを見ます。Control.Applicative の Applicative 型クラスで表現されます。

Haskell の関数はカリー化されているので、複数引数を取るように見える関数は実は引数 1 つだけ取り、次の引数を取る関数を返します。型 a -> b -> c は普通「2 引数取って c を返す」と言いますが、実は a を取って b -> c 関数を返します。だから f x y も (f x) y も同じ。これにより部分適用が可能で、別の関数に渡せる関数が作れます。

これまでファンクタに 1 引数の関数を map してきました。では、* のような 2 引数関数を map すると？　Just 3 に fmap (*) (Just 3) したらどうなる？　Maybe の Functor 実装より、Just 何か なら中身に関数を適用するので、fmap (*) (Just 3) は Just ((*) 3)、つまり Just (* 3)。関数が Just に包まれた！

ghci> :t fmap (++) (Just "hey")
fmap (++) (Just "hey") :: Maybe ([Char] -> [Char])
ghci> :t fmap compare (Just 'a')
fmap compare (Just 'a') :: Maybe (Char -> Ordering)
ghci> :t fmap compare "A LIST OF CHARS"
fmap compare "A LIST OF CHARS" :: [Char -> Ordering]
ghci> :t fmap (\x y z -> x + y / z) [3,4,5,6]
fmap (\x y z -> x + y / z) [3,4,5,6] :: (Fractional a) => [a -> a -> a]

compare（型 (Ord a) => a -> a -> Ordering）を文字のリストに map すると、Char -> Ordering 型の関数のリストになります。compare がリスト内の文字で部分適用されたから。(Ord a) => a -> Ordering ではなく Char -> Ordering なのは、最初の a が Char だったので 2 つ目も Char に決まったから。

「複数引数」の関数をファンクタに map すると、関数を含むファンクタが得られます。これで何ができるでしょう？　まず、それらの関数を引数に取る関数を map できます。ファンクタの中身は何でも map した関数の引数になるので。

ghci> let a = fmap (*) [1,2,3,4]
ghci> :t a
a :: [Integer -> Integer]
ghci> fmap (\f -> f 9) a
[9,18,27,36]

しかし、Just (3 *) という関数入りファンクタと Just 5 があり、前者の関数を取り出して Just 5 に map したいときは？　通常のファンクタでは無理。fmap は普通の関数しか map できません。\f -> f 9 を map したのも普通の関数を map していただけ。ファンクタ内の関数を別のファンクタに map することはできません。Just にパターンマッチして取り出せばできますが、もっと汎用・抽象な方法が欲しい。

そこで Applicative 型クラスの登場。Control.Applicative モジュールにあり、pure と <*> の 2 メソッドを定義。デフォルト実装はないので、両方を定義する必要があります。クラス定義：

class (Functor f) => Applicative f where
    pure :: a -> f a
    (<*>) :: f (a -> b) -> f a -> f b

3 行のシンプルな定義から多くを読み取れます。1 行目は Applicative クラスの定義開始とクラス制約。型コンストラクタを Applicative に入れるには、まず Functor である必要があります。だから Applicative なら Functor でもあり、fmap が使えます。

1 つ目のメソッドは pure、型 pure :: a -> f a。f がアプリカティブファンクタインスタンスの役。Haskell の型システムが優れているので、関数のできることは「引数を取って値を返す」だけ。型宣言から多くが読み取れます。pure は任意の型の値を取って、その値を内部に持つアプリカティブファンクタを返す。「内部」と言うとき箱のアナロジーをまた使っていますが、常に通用するわけではないとはいえ、a -> f a は十分説明的。値を取ってアプリカティブファンクタに包む、と。

もっと良い捉え方は「pure は値をデフォルトの（純粋な）文脈、つまりその値を結果として残せる最小の文脈に置く」こと。

<*> は本当に面白い。型は f (a -> b) -> f a -> f b。何か似ていませんか？　fmap :: (a -> b) -> f a -> f b！　強化版 fmap。fmap が関数とファンクタを取って関数をファンクタ内に適用するのに対し、<*> は「関数入りファンクタ」と「別のファンクタ」を取り、第 1 のファンクタから関数を取り出して第 2 のファンクタに map する、というイメージ。「取り出す」と言いましたが、実際は「実行して取り出す」あるいは「順次実行する」感じ。後で分かります。

Maybe の Applicative インスタンス：

instance Applicative Maybe where
    pure = Just
    Nothing <*> _ = Nothing
    (Just f) <*> something = fmap f something

クラス定義から、f は具体型を 1 つ取る型コンストラクタなので instance Applicative Maybe where（instance Applicative (Maybe a) where ではなく）と書きます。

pure。値をアプリカティブファンクタに包むはずなので pure = Just（Just は普通の関数）。pure x = Just x と書いても OK。

<*>。Nothing から関数を取り出すことはできません（中身がないので）。Nothing から取り出そうとしたら結果は Nothing。Applicative のクラス定義に Functor のクラス制約があるので、両引数がファンクタだと仮定できます。第 1 引数が Nothing でなく Just f なら、f を第 2 引数に map します。第 2 引数が Nothing ならどんな関数を map しても Nothing。

つまり Maybe の <*> は、左が Just なら関数を取り出して右に map し、どちらかが Nothing なら Nothing。

OK、動かしてみましょう：

ghci> Just (+3) <*> Just 9
Just 12
ghci> pure (+3) <*> Just 10
Just 13
ghci> pure (+3) <*> Just 9
Just 12
ghci> Just (++"hahah") <*> Nothing
Nothing
ghci> Nothing <*> Just "woot"
Nothing

このケースでは pure (+3) と Just (+3) は同じ。アプリカティブ文脈（<*> と一緒）で Maybe を扱うなら pure、それ以外は Just を使います。最初の 4 行は関数を取り出して map する例。最後は Nothing から関数を取ろうとした例で、当然 Nothing。

普通のファンクタでは関数を map した結果から、たとえ部分適用された関数であっても汎用的に取り出す方法がありません。アプリカティブファンクタなら、複数のファンクタに対して 1 つの関数で操作できます：

ghci> pure (+) <*> Just 3 <*> Just 5
Just 8
ghci> pure (+) <*> Just 3 <*> Nothing
Nothing
ghci> pure (+) <*> Nothing <*> Just 5
Nothing

何が起きているのか？　<*> は左結合なので pure (+) <*> Just 3 <*> Just 5 は (pure (+) <*> Just 3) <*> Just 5 と同じ。まず + がファンクタ（ここでは Maybe）に包まれて pure (+) = Just (+)。次に Just (+) <*> Just 3 = Just (3+)（部分適用）。最後に Just (3+) <*> Just 5 = Just 8。

すごい！　pure f <*> x <*> y <*> ... というアプリカティブスタイルで、ファンクタに包まれていない引数を期待する関数を、ファンクタ文脈にある複数の値に対して使えます。<*> の間で部分適用が連続的に行われるので、引数は何個でも OK。

もっと便利なのは、pure f <*> x は fmap f x と等価という事実。これがアプリカティブ則の 1 つ。後で詳しく見ますが、直感的にも納得できます。pure は値をデフォルト文脈に置くので、関数を置いて取り出して別のファンクタに適用するのは、ただ map するのと同じ。pure f <*> x <*> y <*> ... の代わりに fmap f x <*> y <*> ... と書けます。だから Control.Applicative は <$>（中置の fmap）をエクスポートします：

(<$>) :: (Functor f) => (a -> b) -> f a -> f b
f <$> x = fmap f x
よっ！　型変数は引数名や値名と独立。ここの関数宣言の f は Functor クラス制約を持つ型変数。本体の f は x に map する関数。同じ f を使っているからといって同じものを表すわけではありません。

<$> を使うとアプリカティブスタイルが映えます。3 つのアプリカティブファンクタに関数 f を適用するなら f <$> x <*> y <*> z。普通の値なら f x y z。

具体例。Just "johntra" と Just "volta" を Maybe ファンクタ内で 1 つの String に結合：

ghci> (++) <$> Just "johntra" <*> Just "volta"
Just "johntravolta"

動作を見る前にこちらと比較：

ghci> (++) "johntra" "volta"
"johntravolta"

素晴らしい！　通常の関数をアプリカティブファンクタに使うには、<$> と <*> を散りばめるだけ。関数はアプリカティブで動き、アプリカティブを返します。クール！

(++) <$> Just "johntra" <*> Just "volta" をたどると、まず (++)（型 [a] -> [a] -> [a]）が Just "johntra" に map されて Just ("johntra"++)（型 Maybe ([Char] -> [Char])）に。(++) の第 1 引数が消費され a が Char に決まりました。次に Just ("johntra"++) <*> Just "volta" で関数が取り出されて Just "volta" に map され Just "johntravolta"。どちらかが Nothing なら結果も Nothing。

これまで Maybe しか例にしていませんが、他のインスタンスもあります！

リスト（実は型コンストラクタ []）はアプリカティブファンクタ。サプライズ！　[] の Applicative インスタンス：

instance Applicative [] where
    pure x = [x]
    fs <*> xs = [f x | f <- fs, x <- xs]

pure は値をデフォルト文脈に置く、と話しました。リストの最小文脈は空リスト [] ですが、空リストは値の不在を表すので pure した値を保持できません。だから pure は値をシングルトンリストに置きます。Maybe の最小文脈は Nothing ですが、これも値の不在なので pure は Just として実装されているのと同じ理屈。

ghci> pure "Hey" :: [String]
["Hey"]
ghci> pure "Hey" :: Maybe String
Just "Hey"

<*>？　リストに限定すれば型は (<*>) :: [a -> b] -> [a] -> [b]。リスト内包表記 で実装されています。<*> は左の関数を取り出して右に map するわけですが、左のリストには関数が 0 個・1 個・複数個ありえます。右にも複数の値がありえます。だからリスト内包表記で両方から取り出し、左の各関数を右の各値に適用、すべての組み合わせを生成します。

ghci> [(*0),(+100),(^2)] <*> [1,2,3]
[0,0,0,101,102,103,1,4,9]

左 3 関数 × 右 3 値 = 9 要素。2 引数の関数のリストなら 2 つのリスト間で適用可能：

ghci> [(+),(*)] <*> [1,2] <*> [3,4]
[4,5,5,6,3,4,6,8]

<*> は左結合なので [(+),(*)] <*> [1,2] が先で [(1+),(2+),(1*),(2*)]。次に [(1+),(2+),(1*),(2*)] <*> [3,4] で結果。

リストにアプリカティブスタイルを使うのは楽しい！

ghci> (++) <$> ["ha","heh","hmm"] <*> ["?","!","."]
["ha?","ha!","ha.","heh?","heh!","heh.","hmm?","hmm!","hmm."]

2 文字列を取る通常関数を、適切な演算子を挟むだけで 2 つの文字列ファンクタの間で使えました。

リストを「非決定的計算」と見ることもできます。100 や "what" は決定的な計算（結果 1 つ）、[1,2,3] は「結果が決められない計算」で考え得るすべての結果を列挙したもの。(+) <$> [1,2,3] <*> [4,5,6] は 2 つの非決定的計算を + で結合し、結果がさらに不確定な非決定的計算を作る、と捉えられます。

リストへのアプリカティブスタイルはリスト内包表記の良い代替になります。第 2 章で [2,5,10] と [8,10,11] のすべての積を求めました：

ghci> [ x*y | x <- [2,5,10], y <- [8,10,11]]
[16,20,22,40,50,55,80,100,110]

2 つのリストから取り出して関数を適用しているだけ。アプリカティブスタイルでも：

ghci> (*) <$> [2,5,10] <*> [8,10,11]
[16,20,22,40,50,55,80,100,110]

2 つの非決定的計算の間で * を呼んでいるだけ、と見えてこちらの方が分かりやすいかも。50 を超える積だけ欲しければ：

ghci> filter (>50) $ (*) <$> [2,5,10] <*> [8,10,11]
[55,80,100,110]

リストでも pure f <*> xs = fmap f xs が成り立つのが分かりやすいです。pure f = [f]、[f] <*> xs は左の各関数を右の各値に適用、左に関数 1 つしかないので map と同じ。

IO もアプリカティブ。実装：

instance Applicative IO where
    pure = return
    a <*> b = do
        f <- a
        x <- b
        return (f x)

pure は値を「結果として保持できる最小文脈」に置くので、IO では return。return は何もしないが値を結果として持つ IOアクションを作るので、ピッタリ。

<*> を IO に特化すると (<*>) :: IO (a -> b) -> IO a -> IO b。関数を結果として持つ IOアクションと別の IOアクションを取り、両方を順次実行して結果を関数で結合する新しい IOアクションを返す。do 記法で実装しています。do は複数の IOアクションを 1 つに糊付けするのが本懐ですから。

Maybe・リストでは <*> は「左から関数を取り出して右に適用」と捉えられました。IO でも取り出しの感覚は残りますが、加えて 順次実行 (sequencing) の感覚があります。2 つの IOアクションを糊付けして 1 つにし、左から関数を取り出すには実行が必要です。

例：

myAction :: IO String
myAction = do
    a <- getLine
    b <- getLine
    return $ a ++ b

2 行入力を求めて連結する IOアクション。2 つの getLine と return を糊付け。アプリカティブスタイルなら：

myAction :: IO String
myAction = (++) <$> getLine <*> getLine

2 つの IOアクションの結果に関数を適用する IOアクションを作っているだけ、それと同じ。getLine の型は IO String。アプリカティブファンクタ間の <*> はアプリカティブを返すので、すべて筋が通ります。

箱のアナロジーで言えば、getLine は現実世界に行って文字列を取ってくる箱。(++) <$> getLine <*> getLine は、その 2 つの箱を送り出して取ってきた行を連結して結果とする、より大きな箱。

(++) <$> getLine <*> getLine の型は IO String。普通の IOアクションなのでこんなこともできます：

main = do
    a <- (++) <$> getLine <*> getLine
    putStrLn $ "The two lines concatenated turn out to be: " ++ a

IOアクションを名前に束縛して関数を呼び return で結果として提示する場面では、アプリカティブスタイルを使うと簡潔に書けます。

もう 1 つの Applicative インスタンスは (->) r、つまり関数。アプリカティブスタイルで関数を使うのはコードゴルフ以外あまりありませんが、興味深いので見てみます。

(->) r の意味が不明なら前のセクションを参照。
instance Applicative ((->) r) where
    pure x = (\_ -> x)
    f <*> g = \x -> f x (g x)

pure でアプリカティブに包んだ値は、結果として常にその値を返す必要があります。「最小のデフォルト文脈で値を返す」。だから関数のインスタンスでは、pure は引数を無視して常にその値を返す関数を作ります。pure を (->) r に特化した型は pure :: a -> (r -> a)。

ghci> (pure 3) "blah"
3

カリー化により関数適用は左結合なので括弧は省略できます：

ghci> pure 3 "blah"
3

<*> の実装は少しややこしいので、アプリカティブスタイルでの使い方を見るのが一番：

ghci> :t (+) <$> (+3) <*> (*100)
(+) <$> (+3) <*> (*100) :: (Num a) => a -> a
ghci> (+) <$> (+3) <*> (*100) $ 5
508

2 つのアプリカティブファンクタに <*> を呼ぶと、結果はアプリカティブファンクタ。関数 2 つで使えば結果は関数。(+) <$> (+3) <*> (*100) は (+3) と (*100) の結果に + を使って返す関数。実例で (+) <$> (+3) <*> (*100) $ 5 なら、まず 5 が (+3) と (*100) に適用され 8 と 500、+ で結合して 508。

ghci> (\x y z -> [x,y,z]) <$> (+3) <*> (*2) <*> (/2) $ 5
[8.0,10.0,2.5]

同じ。5 を 3 つの関数に適用し、結果を \x y z -> [x,y,z] に渡す関数を作っています。

関数を「いずれの結果を含む箱」と捉えると、k <$> f <*> g は f と g のいずれの結果を k に渡す関数を作る、と。(+) <$> Just 3 <*> Just 5 は「あるかもしれない値」に + を使って「あるかもしれない値」を返す。(+) <$> (+10) <*> (+5) は「将来の戻り値」に + を使う。

関数のアプリカティブを多用することはありませんが、面白いです。(->) r の Applicative インスタンスを完全に理解する必要はありません。アプリカティブスタイルで関数と遊んで直感を養ってください。

未出のもう 1 つの Applicative インスタンスが ZipList（Control.Applicative にある）。

実はリストはもう 1 通りのアプリカティブにもなれます。すでに見たのは「左の関数を右の各値に適用、すべての組み合わせ」。[(+3),(*2)] <*> [1,2] なら (+3) を 1・2 に、(*2) も 1・2 に適用して 4 要素 [4,5,2,4]。

もう 1 通りは「左の最初の関数を右の最初の値に、左の 2 番目の関数を右の 2 番目の値に……」と zip のように動かす方法。[(+3),(*2)] <*> [1,2] なら 2 要素 [4,4]（[1+3, 2*2]）。

同じ型に同じ型クラスのインスタンスを 2 通り持てないので、ZipList a 型が導入されました。値コンストラクタ ZipList、フィールド 1 つ（リスト）。インスタンス：

instance Applicative ZipList where
        pure x = ZipList (repeat x)
        ZipList fs <*> ZipList xs = ZipList (zipWith (\f x -> f x) fs xs)

<*> は先頭から順に関数を値に適用、zipWith (\f x -> f x) fs xs。zipWith の性質上、結果の長さは短い方に合わせられます。

pure も興味深い。値を取って無限に繰り返したリストに包みます。pure "haha" = ZipList (["haha","haha","haha"...。「最小文脈」と矛盾しそうですが、zip リストでは各位置で値を生む必要があるので筋が通ります。pure f <*> xs = fmap f xs の法則も満たします。pure 3 が ZipList [3] を返したら、pure (*2) <*> ZipList [1,5,10] は ZipList [2] になり短い方に合わせられてしまう。有限と無限を zip すれば結果の長さは有限の長さになります。

アプリカティブスタイルの zip リスト。ZipList a には Show インスタンスがないので、getZipList でリストを取り出します。

ghci> getZipList $ (+) <$> ZipList [1,2,3] <*> ZipList [100,100,100]
[101,102,103]
ghci> getZipList $ (+) <$> ZipList [1,2,3] <*> ZipList [100,100..]
[101,102,103]
ghci> getZipList $ max <$> ZipList [1,2,3,4,5,3] <*> ZipList [5,3,1,2]
[5,3,3,4]
ghci> getZipList $ (,,) <$> ZipList "dog" <*> ZipList "cat" <*> ZipList "rat"
[('d','c','r'),('o','a','a'),('g','t','t')]
(,,) は \x y z -> (x,y,z) と同じ。(,) は \x y -> (x,y) と同じ。

標準ライブラリには zipWith3、zipWith4、…… zipWith7 までありますが、zip リストとアプリカティブスタイルなら、リスト数ごとに別の zip 関数を用意する必要なく、任意個数のリストを関数で zip できます。

Control.Applicative には liftA2 という関数があります（型 liftA2 :: (Applicative f) => (a -> b -> c) -> f a -> f b -> f c）：

liftA2 :: (Applicative f) => (a -> b -> c) -> f a -> f b -> f c
liftA2 f a b = f <$> a <*> b

2 つのアプリカティブに関数を適用するだけで、アプリカティブスタイルを隠蔽。これがアプリカティブファンクタが普通のファンクタより強力な理由を示しています。普通のファンクタは 1 つのファンクタに関数を map できるだけ、アプリカティブは複数のファンクタに渡って関数を適用できます。型を (a -> b -> c) -> (f a -> f b -> f c) と見れば、liftA2 は通常の二項関数を取って 2 つのファンクタで動く関数に持ち上げる、と読めます。

面白い概念：2 つのアプリカティブを取って、結果をリストに持つ 1 つのアプリカティブに合体できます。Just 3 と Just 4。後者をシングルトンリストに：

ghci> fmap (\x -> [x]) (Just 4)
Just [4]

Just 3 と Just [4] から Just [3,4] を得るには？　簡単：

ghci> liftA2 (:) (Just 3) (Just [4])
Just [3,4]
ghci> (:) <$> Just 3 <*> Just [4]
Just [3,4]

: は要素とリストを取って先頭に要素を追加した新しいリストを返す関数。Just [3,4] ができたので Just 2 と組み合わせて Just [2,3,4] も同様。任意個数のアプリカティブを「結果のリストを持つ 1 つのアプリカティブ」に合体できそう。アプリカティブのリストを取って、結果のリストを持つアプリカティブを返す関数 sequenceA を実装します：

sequenceA :: (Applicative f) => [f a] -> f [a]
sequenceA [] = pure []
sequenceA (x:xs) = (:) <$> x <*> sequenceA xs

再帰！　まず型から：アプリカティブのリストを「リストを持つアプリカティブ」に変換。エッジ条件は空リスト → デフォルト文脈に空リストを置く。再帰：先頭 x（アプリカティブ）と尾 xs（アプリカティブのリスト）について sequenceA を尾に適用してリストを持つアプリカティブを得て、そのリストの先頭に x の中身を追加。

sequenceA [Just 1, Just 2] は (:) <$> Just 1 <*> sequenceA [Just 2]、つまり (:) <$> Just 1 <*> ((:) <$> Just 2 <*> sequenceA [])。sequenceA [] = Just [] なので (:) <$> Just 1 <*> ((:) <$> Just 2 <*> Just []) = (:) <$> Just 1 <*> Just [2] = Just [1,2]！

fold で実装する別の方法。リストを 1 要素ずつ走査して結果を蓄積する関数はだいたい fold で書けます：

sequenceA :: (Applicative f) => [f a] -> f [a]
sequenceA = foldr (liftA2 (:)) (pure [])

右からリストにアプローチし、初期アキュムレータは pure []。アキュムレータと末尾要素に liftA2 (:)、シングルトンを持つアプリカティブに。続けて末尾の隣の要素と現アキュムレータに liftA2 (:)、と進めて、最後にすべてのアプリカティブの結果のリストを持つアキュムレータが残ります。

使ってみます：

ghci> sequenceA [Just 3, Just 2, Just 1]
Just [3,2,1]
ghci> sequenceA [Just 3, Nothing, Just 1]
Nothing
ghci> sequenceA [(+3),(+2),(+1)] 3
[6,5,4]
ghci> sequenceA [[1,2,3],[4,5,6]]
[[1,4],[1,5],[1,6],[2,4],[2,5],[2,6],[3,4],[3,5],[3,6]]
ghci> sequenceA [[1,2,3],[4,5,6],[3,4,4],[]]
[]

Maybe 値で使うと、すべての結果をリストとして持つ Maybe を作ります。1 つでも Nothing なら結果も Nothing。リスト中に Nothing がない場合だけ値を取りたいときに便利。

関数で使うと、関数のリストを取って「リストを返す関数」を作ります。sequenceA [(+3),(+2),(+1)] 3 は 3 を 3 つの関数に適用してリストにします。

(+) <$> (+3) <*> (*2) が「引数を (+3) と (*2) に渡して結果を + で結合する関数」を作るのと同様、sequenceA [(+3),(*2)] は引数をリストの全関数に渡す関数を作ります。+ の代わりに : と pure [] でリスト化。

sequenceA は、関数のリストに同じ入力を与えて結果のリストを見たいときに便利。たとえばある数がリスト内のすべての述語を満たすか：

ghci> map (\f -> f 7) [(>4),(<10),odd]
[True,True,True]
ghci> and $ map (\f -> f 7) [(>4),(<10),odd]
True

and は真偽値リストを取って全部 True なら True。sequenceA でも：

ghci> sequenceA [(>4),(<10),odd] 7
[True,True,True]
ghci> and $ sequenceA [(>4),(<10),odd] 7
True

sequenceA [(>4),(<10),odd] は数を取って 3 つの述語に渡し真偽値のリストを返す関数を作る。型 (Num a) => [a -> Bool] を (Num a) => a -> [Bool] に変換。

リストは同質なので、リスト内の関数は全部同じ型でないとダメ。[ord, (+3)] のように混ぜられません（ord は文字 → 数、(+3) は数 → 数）。

[] で使うと、sequenceA はリストのリストを取ってリストのリストを返します。すべての要素の組み合わせのリストを作る、と。比較：

ghci> sequenceA [[1,2,3],[4,5,6]]
[[1,4],[1,5],[1,6],[2,4],[2,5],[2,6],[3,4],[3,5],[3,6]]
ghci> [[x,y] | x <- [1,2,3], y <- [4,5,6]]
[[1,4],[1,5],[1,6],[2,4],[2,5],[2,6],[3,4],[3,5],[3,6]]
ghci> sequenceA [[1,2],[3,4]]
[[1,3],[1,4],[2,3],[2,4]]
ghci> [[x,y] | x <- [1,2], y <- [3,4]]
[[1,3],[1,4],[2,3],[2,4]]
ghci> sequenceA [[1,2],[3,4],[5,6]]
[[1,3,5],[1,3,6],[1,4,5],[1,4,6],[2,3,5],[2,3,6],[2,4,5],[2,4,6]]
ghci> [[x,y,z] | x <- [1,2], y <- [3,4], z <- [5,6]]
[[1,3,5],[1,3,6],[1,4,5],[1,4,6],[2,3,5],[2,3,6],[2,4,5],[2,4,6]]

少し難しいですが、しばらく遊べば動きが分かります。sequenceA [[1,2],[3,4]] の評価をたどってみます：

  • sequenceA [[1,2],[3,4]]
  • (:) <$> [1,2] <*> sequenceA [[3,4]]
  • 内側を展開：(:) <$> [1,2] <*> ((:) <$> [3,4] <*> sequenceA [])
  • エッジ条件：(:) <$> [1,2] <*> ((:) <$> [3,4] <*> [[]])
  • (:) <$> [3,4] <*> [[]] を評価：左 3・4、右 [] で [3:[], 4:[]] = [[3],[4]]。今は (:) <$> [1,2] <*> [[3],[4]]
  • 左 1・2、右 [3]・[4] で [1:[3], 1:[4], 2:[3], 2:[4]] = [[1,3],[1,4],[2,3],[2,4]]

(+) <$> [1,2] <*> [4,5,6] は非決定的計算 x + y（x は [1,2]、y は [4,5,6] から）。可能なすべての結果のリストで表現。sequence [[1,2],[3,4],[5,6],[7,8]] は非決定的計算 [x,y,z,w]（x は [1,2] から、…）。可能なすべてのリストのリスト、というわけです。

IOアクションで使うと sequenceA は sequence と同じ！　IOアクションのリストを取って、各アクションを順次実行して結果のリストを持つ IOアクションを返します。[IO a] を IO [a] に変えるには、すべての IOアクションを順番に実行する必要があり、結果を得るには実行が必要なので。

ghci> sequenceA [getLine, getLine, getLine]
heyh
ho
woo
["heyh","ho","woo"]

普通のファンクタと同様、アプリカティブにも法則があります。最重要なのは先述の pure f <*> x = fmap f x。練習として、この章で見たアプリカティブで証明してみてください。他の法則：

  • pure id <*> v = v
  • pure (.) <*> u <*> v <*> w = u <*> (v <*> w)
  • pure f <*> pure x = pure (f x)
  • u <*> pure y = pure ($ y) <*> u

詳しくは省略しますが、興味があればインスタンスで成り立つか確認してみてください。

結論：アプリカティブファンクタは興味深いだけでなく便利。I/O 計算、非決定的計算、失敗するかもしれない計算などをアプリカティブスタイルで合成できます。<$> と <*> だけで、通常関数を任意個数のアプリカティブファンクタに統一的に適用し、各々のセマンティクスを活かせます。

newtypeキーワード

これまで data キーワードで自前の代数的データ型を作り、type キーワードで既存型のシノニムを作ってきました。このセクションでは、既存型から newtype キーワードで新しい型を作る方法と、その動機を見ます。

前のセクションで、リストはもう 1 通りでアプリカティブファンクタになれると述べました。1 つは「<*> が左の関数を右の値すべてに適用、すべての組み合わせを返す」やり方。 

ghci> [(+1),(*100),(*5)] <*> [1,2,3]
[2,3,4,100,200,300,5,10,15]

もう 1 つは「左の最初の関数を右の最初の値に、左の 2 番目を右の 2 番目に……」と zip 風に動かす方法。リストはすでに Applicative のインスタンスなので、もう 1 つの方法をどう実現したか？　ZipList a 型が導入されたんでした。値コンストラクタ ZipList、フィールド 1 つ。zip 風に使うときはリストを ZipList で包み、終わったら getZipList で取り出します： 

ghci> getZipList $ ZipList [(+1),(*100),(*5)] <*> ZipList [1,2,3]
[2,200,15]

これと newtype キーワードの関係は？　ZipList a をどう書くか考えてみます。1 つは： 

data ZipList a = ZipList [a]

値コンストラクタ 1 つ、フィールド 1 つ（リスト）の型。レコード構文でリストを取り出す関数も自動生成： 

data ZipList a = ZipList { getZipList :: [a] }

よさそうで実際動きます。既存型を別の型に包んで別のインスタンスにする、というのを data でやったわけです。

Haskell の newtype キーワードはまさにこういうケース、つまり 1 つの型を包んで別の型として提示するためのものです。実際のライブラリでは ZipList a はこう定義されています： 

newtype ZipList a = ZipList { getZipList :: [a] }

data ではなく newtype。なぜ？　まず newtype は速い。data で型を包むと実行時に包み・解きのオーバーヘッドがあります。newtype なら Haskell は「既存型を新しい型に包んでいるだけ（だから名前も newtype）。内部的には同じ、型だけ違う」と分かります。値の型解決が済めば包み・解きを取り除けます。

じゃあ data の代わりに常に newtype でいいのでは？　newtype は値コンストラクタ 1 つ・フィールド 1 つしか持てません。data なら複数のコンストラクタ・フィールド 0 個以上： 

data Profession = Fighter | Archer | Accountant

data Race = Human | Elf | Orc | Goblin

data PlayerCharacter = PlayerCharacter Race Profession

newtype ではコンストラクタ 1 つ・フィールド 1 つに制限されます。

newtype でも deriving が使えます。Eq、Ord、Enum、Bounded、Show、Read を派生できます。包む元の型がそのクラスのインスタンスである必要があります（newtype は包むだけなので）。次のように書けば、表示と等価性比較ができます： 

newtype CharList = CharList { getCharList :: [Char] } deriving (Eq, Show)

試してみます： 

ghci> CharList "this will be shown!"
CharList {getCharList = "this will be shown!"}
ghci> CharList "benny" == CharList "benny"
True
ghci> CharList "benny" == CharList "oisters"
False

この newtype の値コンストラクタの型： 

CharList :: [Char] -> CharList

[Char] 値（"my sharona" など）を取って CharList 値を返す。レコード構文で生成された getCharList の型は： 

getCharList :: CharList -> [Char]

CharList 値を [Char] に変換。包み・解きと考えても、型変換と考えてもOK。

newtype で型クラスインスタンスを作る

ある型をある型クラスのインスタンスにしたいけれど型引数が合わない、ということがよくあります。Maybe を Functor インスタンスにするのは簡単： 

class Functor f where
    fmap :: (a -> b) -> f a -> f b

これでスタートし：

instance Functor Maybe where

fmap を実装すれば OK。Maybe が f の役を担うので、Maybe 専用の fmap は： 

fmap :: (a -> b) -> Maybe a -> Maybe b

タプルを Functor インスタンスにして「fmap したらタプルの第 1 要素に関数が適用される」ようにしたら？　fmap (+3) (1,1) = (4,1) となるように。実は難しい。Maybe なら instance Functor Maybe where でよかったのは、引数 1 つの型コンストラクタしか Functor にできないから。(a,b) ではうまくいきません。回避策：タプルを newtype で包み、第 2 型引数を「タプルの第 1 要素の型」とします： 

newtype Pair b a = Pair { getPair :: (a,b) }

これで第 1 要素に関数を map する Functor インスタンスを作れます： 

instance Functor (Pair c) where
    fmap f (Pair (x,y)) = Pair (f x, y)

newtype で定義した型でもパターンマッチできます。タプルを取り出して第 1 要素に f を適用、Pair で包み直して Pair b a に戻す。Pair 専用の fmap の型は： 

fmap :: (a -> b) -> Pair c a -> Pair c b

instance Functor (Pair c) where としたので、Pair c が f の役： 

class Functor f where
    fmap :: (a -> b) -> f a -> f b

タプルを Pair b a に変換すれば fmap が第 1 要素に適用されます： 

ghci> getPair $ fmap (*100) (Pair (2,3))
(200,3)
ghci> getPair $ fmap reverse (Pair ("london calling", 3))
("gnillac nodnol",3)

newtype の遅延性

newtype は通常 data より速い、と話しました。newtype は既存型を新しい型に変えるだけなので、内部表現を元の型と同じにでき、型だけ別に保てる。これにより newtype は速いだけでなく遅延的でもあります。

Haskell はデフォルトで遅延評価。関数の結果を表示するときに初めて計算が起きる。さらに必要な計算しか起きない。undefined はエラー計算を表す値。評価しようとすると Haskell は怒り（厳密には例外）： 

ghci> undefined
*** Exception: Prelude.undefined

ただし undefined を含むリストでも先頭だけ要求すれば問題ありません。Haskell は他の要素を評価する必要がないので： 

ghci> head [3,4,5,undefined,2,undefined]
3

次の型を考えます： 

data CoolBool = CoolBool { getCoolBool :: Bool }

data で定義したありふれた代数的データ型。値コンストラクタ 1 つ、フィールド 1 つ（型 Bool）。CoolBool にパターンマッチして、中身の Bool が True でも False でも常に "hello" を返す関数： 

helloMe :: CoolBool -> String
helloMe (CoolBool _) = "hello"

普通の CoolBool ではなく undefined を渡したら？ 

ghci> helloMe undefined
"*** Exception: Prelude.undefined

あら、例外！　なぜ？　data で定義した型は複数の値コンストラクタを持てるので（CoolBool は 1 つだけど）、引数が (CoolBool _) パターンに合うかを確かめるには、Haskell はどの値コンストラクタが使われたかを知るために値を少しでも評価する必要があります。undefined を少しでも評価すると例外。

data の代わりに newtype にしてみます： 

newtype CoolBool = CoolBool { getCoolBool :: Bool }

helloMe はそのまま（パターンマッチ構文は同じ）。undefined を渡すと： 

ghci> helloMe undefined
"hello"

動いた！　なぜ？　newtype なら新しい型の値を内部的に元の値と同じ表現で持てます。包む箱を追加せず、型が違うことだけ覚えておく。値コンストラクタが 1 つ・フィールドが 1 つしか持てないと分かっているので、(CoolBool _) パターンに合うか確かめるために値を評価する必要すらありません！

この振る舞いの違いは些細に見えますが大事です。data と newtype はプログラマから見ると似ていますが、実は別の仕組み。data は型をゼロから作る、newtype は既存型から新しい型を作る。newtype へのパターンマッチは「箱から取り出す」というより「ある型から別の型への直接変換」です。

type と newtype と data

type、data、newtype の違いを整理します。

type は型シノニムを作る。既存型に別名を付けるだけ： 

type IntList = [Int]

これで [Int] を IntList と呼べる。互換的。値コンストラクタはありません： 

ghci> ([1,2,3] :: IntList) ++ ([1,2,3] :: [Int])
[1,2,3,1,2,3]

型シグネチャを説明的にしたいときに使います。電話帳の [(String,String)] に PhoneBook という名前を付けたように。

newtype は既存型を包んで新しい型を作る。主に型クラスのインスタンスにしやすくするため。包んでできた型は元の型と別物： 

newtype CharList = CharList { getCharList :: [Char] }

CharList と [Char] を ++ でつなぐことはできません。2 つの CharList 同士もダメ（++ はリスト用、CharList はリストではない）。CharList 2 つをリストに変換して ++ し、CharList に戻すのは可能。

newtype でレコード構文を使うと、新しい型と元の型の間の変換関数が手に入ります。元の型が属するクラスを自動継承するわけではないので、派生するか手で書く必要があります。

実用的には、newtype は「コンストラクタ 1 つ・フィールド 1 つの data」と捉えてください。そういう data を書きそうになったら newtype を検討。

data は自前のデータ型をゼロから作る。コンストラクタ・フィールド数は自由。リストや Maybe 風の型から木まで何でも作れます。

型シグネチャを綺麗にしたいだけなら型シノニム。既存型を包んで型クラスのインスタンスにしたいなら newtype。何かをゼロから作りたいなら data。

モノイド

Haskell の型クラスは「振る舞いを共有する型のためのインターフェース」を提供します。等価性比較の Eq、順序の Ord などの単純なものから始まり、Functor や Applicative のような面白いものへと進んできました。

型を作るときは、その型がどんな振る舞いをサポートするか考え、それに応じてどの型クラスのインスタンスにするか決めます。等価性比較が意味をなすなら Eq、ファンクタっぽいなら Functor、と。

さて、* は 2 数を取って掛ける関数。何かに 1 を掛けると常にその数。1 * x でも x * 1 でも結果は x。同様に ++ は 2 つを取って 1 つを返す関数（数を掛けるのではなくリストを連結）。* と同じく、結合してももう片方を変えない値があります：空リスト []。 

ghci> 4 * 1
4
ghci> 1 * 9
9
ghci> [1,2,3] ++ []
[1,2,3]
ghci> [] ++ [0.5, 2.5]
[0.5,2.5]

* と 1、++ と [] には共通の性質があります：

  • 関数は 2 引数を取る
  • 引数と戻り値が同じ型
  • その二項関数で他の値と結合しても他の値を変えない値が存在する

さらに目立たない共通性があります：3 つ以上の値を二項関数で結合するとき、結合の順序は結果に影響しません。(3 * 4) * 5 でも 3 * (4 * 5) でも結果は 60。++ も同様： 

ghci> (3 * 2) * (8 * 5)
240
ghci> 3 * (2 * (8 * 5))
240
ghci> "la" ++ ("di" ++ "da")
"ladida"
ghci> ("la" ++ "di") ++ "da"
"ladida"

これを 結合性 (associativity) と呼びます。* も ++ も結合的ですが、たとえば - は違います。(5 - 3) - 4 と 5 - (3 - 4) は違います。

これらの性質を書き出すうちに、私たちは モノイド に出会いました！　モノイドとは「結合的な二項関数」と「その関数に対して恒等元として振る舞う値」の組です。恒等元とは「その関数で他の値と結合すると、結果が常に他の値と等しくなる値」。1 は * の恒等元、[] は ++ の恒等元。Haskell の世界には他にもたくさんのモノイドがあるので、Monoid 型クラスが存在します。モノイドのように振る舞える型のためのもの。定義： 

class Monoid m where
    mempty :: m
    mappend :: m -> m -> m
    mconcat :: [m] -> m
    mconcat = foldr mappend mempty

Monoid 型クラスは import Data.Monoid。じっくり見ます。

まず、具体型だけが Monoid インスタンスになれます（クラス定義の m は型引数を取らない）。Functor や Applicative（型引数 1 つの型コンストラクタが必要）と違う点。

1 つ目は mempty。引数を取らないので関数というより多相な定数。Bounded の minBound のようなもの。mempty はそのモノイドの恒等元を表します。

次は mappend。二項関数で、同じ型の 2 値を取って同じ型の値を返します。mappend という名前は不幸でした。「2 つを連結 (append) する」と思わせるからです。++ はリスト連結ですが、* は連結ではなく掛け算。他のモノイドの多くも連結ではないので、「連結」と思わず「2 つのモノイド値を取って 3 つ目を返す二項関数」と捉えてください。

最後は mconcat。モノイド値のリストを取って、mappend で 1 つに縮約。デフォルト実装は mempty を初期値にして右から fold。多くの場合これで十分なので深入りしません。Monoid インスタンスを作るときは mempty と mappend だけ実装すれば OK。mconcat は、より効率的な実装が可能なインスタンスのために用意されています。

具体的なインスタンスに進む前にモノイド則を見ます。恒等元と結合的な二項関数が必要、と言いました。これらに従わない Monoid インスタンスも作れますが、誰の役にも立ちません。Monoid を使うときはインスタンスがモノイドのように振る舞うことに頼っているので。インスタンスを作るときは次の法則を守りましょう：

  • mempty `mappend` x = x
  • x `mappend` mempty = x
  • (x `mappend` y) `mappend` z = x `mappend` (y `mappend` z)

最初の 2 つは mempty が mappend の恒等元、3 つ目は mappend が結合的、を述べています。Haskell は強制してくれないので、プログラマが守らなければなりません。

リストはモノイド

リストはモノイド！　++ と [] でモノイド。インスタンスはとてもシンプル： 

instance Monoid [a] where
    mempty = []
    mappend = (++)

要素の型を問わず、リストは Monoid。instance Monoid [] ではなく instance Monoid [a] と書くのは、Monoid が具体型を要求するから。

試してみます： 

ghci> [1,2,3] `mappend` [4,5,6]
[1,2,3,4,5,6]
ghci> ("one" `mappend` "two") `mappend` "tree"
"onetwotree"
ghci> "one" `mappend` ("two" `mappend` "tree")
"onetwotree"
ghci> "one" `mappend` "two" `mappend` "tree"
"onetwotree"
ghci> "pang" `mappend` mempty
"pang"
ghci> mconcat [[1,2],[3,6],[9]]
[1,2,3,6,9]
ghci> mempty :: [a]
[]

最終行は明示的型注釈が必要。mempty だけでは GHCi がどのインスタンスか分からないので、リストインスタンスを使うと指定。[a] としたのは空リストが任意の型を含むかのように振る舞えるから。

mconcat はデフォルト実装で Monoid インスタンスにすると無料で手に入ります。リストの場合は concat と同じ。リストのリストを平坦化（隣接リストの間で ++ を行うのと同じなので）。

リストインスタンスはモノイド則を満たします。複数のリストを mappend（++）するとき順序は影響しません（端で繋ぐだけなので）。空リストが恒等元。a `mappend` b = b `mappend` a はモノイドの要件ではない点に注意（リストでは違います）： 

ghci> "one" `mappend` "two"
"onetwo"
ghci> "two" `mappend` "one"
"twoone"

それで OK。3 * 5 = 5 * 3 は乗算の性質ですが、すべてのモノイド（実はほとんどのモノイド）に当てはまるわけではありません。

Product と Sum

数値のモノイドの 1 つは * と 1。もう 1 つは + と 0： 

ghci> 0 + 4
4
ghci> 5 + 0
5
ghci> (1 + 3) + 5
9
ghci> 1 + (3 + 5)
9

0 を足しても変わらず、加法も結合的なので法則を満たします。両方有効なので、newtype で包んで別々のインスタンスにします。

Data.Monoid は Product と Sum をエクスポート。Product： 

newtype Product a =  Product { getProduct :: a }
    deriving (Eq, Ord, Read, Show, Bounded)

シンプルな newtype ラッパー。Monoid インスタンス： 

instance Num a => Monoid (Product a) where
    mempty = Product 1
    Product x `mappend` Product y = Product (x * y)

mempty は 1 を Product で包んだもの。mappend は Product をパターンマッチで取り出して掛けて包み直す。Num a 制約付き。Num インスタンスである a について Product a は Monoid。使うには newtype 包み・解きが必要： 

ghci> getProduct $ Product 3 `mappend` Product 9
27
ghci> getProduct $ Product 3 `mappend` mempty
3
ghci> getProduct $ Product 3 `mappend` Product 4 `mappend` Product 2
24
ghci> getProduct . mconcat . map Product $ [3,4,2]
24

Monoid 型クラスのデモとしては良いですが、実際は単に 3 * 9 や 3 * 1 と書きます。後で、これらの一見些細な Monoid インスタンスが役立つ場面を見ます。

Sum は Product と同様に定義されインスタンスも同様： 

ghci> getSum $ Sum 2 `mappend` Sum 9
11
ghci> getSum $ mempty `mappend` Sum 3
3
ghci> getSum . mconcat . map Sum $ [1,2,3]
6

Any と All

Bool も 2 通りでモノイドになれます。1 つは or（||）と恒等元 False。論理 or は片方が True なら True。False を恒等元にすると False と or したら False、True と or したら True。Any がこの方法のインスタンス： 

newtype Any = Any { getAny :: Bool }
    deriving (Eq, Ord, Read, Show, Bounded)

インスタンス： 

instance Monoid Any where
        mempty = Any False
        Any x `mappend` Any y = Any (x || y)

名前 Any の理由：x `mappend` y は any（どれか）が True なら True。3 つ以上 mappend しても、どれかが True なら結果は True： 

ghci> getAny $ Any True `mappend` Any False
True
ghci> getAny $ mempty `mappend` Any True
True
ghci> getAny . mconcat . map Any $ [False, False, False, True]
True
ghci> getAny $ mempty `mappend` mempty
False

もう 1 つは反対で、&& と恒等元 True。論理 and は両方 True のときだけ True。newtype： 

newtype All = All { getAll :: Bool }
        deriving (Eq, Ord, Read, Show, Bounded)

インスタンス： 

instance Monoid All where
        mempty = All True
        All x `mappend` All y = All (x && y)

All 値を mappend すると、all（全部）が True のときだけ True： 

ghci> getAll $ mempty `mappend` All True
True
ghci> getAll $ mempty `mappend` All False
False
ghci> getAll . mconcat . map All $ [True, True, True]
True
ghci> getAll . mconcat . map All $ [True, True, False]
False

乗算・加算と同様、普通は newtype で包まずに直接二項関数を使います。mconcat は Any・All で便利そうですが、普通は or・and（真偽値リストを取って True なら True）が使いやすいです。

Ordering モノイド

Ordering 型を覚えていますか？　比較結果として LT、EQ、GT を持つ： 

ghci> 1 `compare` 2
LT
ghci> 2 `compare` 2
EQ
ghci> 3 `compare` 2
GT

リスト・数値・真偽値ではモノイドを見つけるのに既存関数を眺めるだけで済みましたが、Ordering はもう少し探す必要があります。Monoid インスタンスは直感的で便利： 

instance Monoid Ordering where
    mempty = EQ
    LT `mappend` _ = LT
    EQ `mappend` y = y
    GT `mappend` _ = GT

2 つの Ordering を mappend すると、左が保たれる（左が EQ のときは右が結果）。恒等元は EQ。一見任意に見えますが、実は単語の辞書順比較に似ています。最初の 2 文字を比べて違えば辞書順が決まる。同じなら次の文字へ……。

たとえば "ox" と "on" を比較すると、最初の文字は同じなので 2 文字目の比較に進む。'x' は 'n' より大きいので結果が決まる。EQ が恒等元なのは、両方の単語の同じ位置に同じ文字を挿入しても辞書順は変わらないから。"oix" は "oin" より大きいまま。

Ordering の Monoid インスタンスでは x `mappend` y ≠ y `mappend` x。左が保たれるので LT `mappend` GT = LT、GT `mappend` LT = GT： 

ghci> LT `mappend` GT
LT
ghci> GT `mappend` LT
GT
ghci> mempty `mappend` LT
LT
ghci> mempty `mappend` GT
GT

このモノイドは何が便利か？　たとえば 2 つの文字列の長さを比較し、同じなら辞書順比較する関数。普通に書くと： 

lengthCompare :: String -> String -> Ordering
lengthCompare x y = let a = length x `compare` length y
                        b = x `compare` y
                    in  if a == EQ then b else a

長さ比較の結果を a、辞書順比較を b として、長さが同じなら辞書順を返します。

Ordering がモノイドなら、ずっとシンプル： 

import Data.Monoid

lengthCompare :: String -> String -> Ordering
lengthCompare x y = (length x `compare` length y) `mappend`
                    (x `compare` y)

試します： 

ghci> lengthCompare "zen" "ants"
LT
ghci> lengthCompare "zen" "ant"
GT

mappend は左が EQ でない限り左が保たれるので、最重要の比較を最初の引数にします。母音の数も第 2 の基準にしたいなら： 

import Data.Monoid

lengthCompare :: String -> String -> Ordering
lengthCompare x y = (length x `compare` length y) `mappend`
                    (vowels x `compare` vowels y) `mappend`
                    (x `compare` y)
    where vowels = length . filter (`elem` "aeiou")

補助関数で文字列の母音数を求めています。 

ghci> lengthCompare "zen" "anna"
LT
ghci> lengthCompare "zen" "ana"
LT
ghci> lengthCompare "zen" "ann"
GT

1 つ目は長さが違うので LT。2 つ目は長さが同じで "zen" の母音が少ないので LT。3 つ目は長さも母音数も同じで辞書順比較になり "zen" が勝ち GT。

Ordering モノイドは多基準比較を簡単に組み合わせ、優先順位付けできるので便利。

Maybe モノイド

Maybe a をモノイドにする方法を見ます。

1 つは「a もモノイドである場合に限り」Maybe a をモノイドとし、Just 内の値を mappend で結合し、Nothing を恒等元とする方法： 

instance Monoid a => Monoid (Maybe a) where
    mempty = Nothing
    Nothing `mappend` m = m
    m `mappend` Nothing = m
    Just m1 `mappend` Just m2 = Just (m1 `mappend` m2)

クラス制約は「a がモノイドのときだけ Maybe a はモノイド」。Nothing と何かを mappend すると何かのまま。Just 同士なら中身を mappend して Just で包む： 

ghci> Nothing `mappend` Just "andy"
Just "andy"
ghci> Just LT `mappend` Nothing
Just LT
ghci> Just (Sum 3) `mappend` Just (Sum 4)
Just (Sum {getSum = 7})

失敗するかもしれない計算結果のモノイドを扱うときに便利。Nothing か Just かを気にせず、普通のモノイドとして扱えます。

中身がモノイドでなかったら？　前のインスタンスで中身がモノイドでないと困るのは「両方が Just」の場合だけ。中身がモノイドか分からないなら結合できません。1 つの解決策は「2 つ目を捨てて 1 つ目を保つ」。そのために First a 型があります： 

newtype First a = First { getFirst :: Maybe a }
    deriving (Eq, Ord, Read, Show)

Maybe a を newtype で包むだけ。Monoid インスタンス： 

instance Monoid (First a) where
    mempty = First Nothing
    First (Just x) `mappend` _ = First (Just x)
    First Nothing `mappend` x = x

mempty は Nothing を First で包んだもの。mappend の左が Just なら右を無視。左が Nothing なら右を結果に： 

ghci> getFirst $ First (Just 'a') `mappend` First (Just 'b')
Just 'a'
ghci> getFirst $ First Nothing `mappend` First (Just 'b')
Just 'b'
ghci> getFirst $ First (Just 'a') `mappend` First Nothing
Just 'a'

Maybe の集まりから、最初の Just を取りたいときに便利。mconcat が活躍： 

ghci> getFirst . mconcat . map First $ [Nothing, Just 9, Just 10]
Just 9

両方が Just なら 2 つ目を保ちたいなら Last a。First の逆で、最後の非 Nothing 値を保つ： 

ghci> getLast . mconcat . map Last $ [Nothing, Just 9, Just 10]
Just 10
ghci> getLast $ Last (Just "one") `mappend` Last (Just "two")
Just "two"

モノイドでデータ構造を畳み込む

モノイドの面白い使い方の 1 つは、さまざまなデータ構造の fold を定義するのに役立てること。リスト以外にも木など多くのデータ構造を fold できます。

fold できるデータ構造はたくさんあるので、Foldable 型クラスが導入されました。Functor が「map できるもの」のように、Foldable は「fold できるもの」！　Data.Foldable にあり、Prelude と名前が衝突するので qualified import がおすすめ： 

import qualified Foldable as F

F に短縮。foldr、foldl、foldr1、foldl1 などが定義されています。すでに知っている関数なのでは？　Foldable の foldr と Prelude の比較： 

ghci> :t foldr
foldr :: (a -> b -> b) -> b -> [a] -> b
ghci> :t F.foldr
F.foldr :: (F.Foldable t) => (a -> b -> b) -> b -> t a -> b

foldr はリスト専用、Data.Foldable の foldr は fold できる任意の型。リストでは同じ： 

ghci> foldr (*) 1 [1,2,3]
6
ghci> F.foldr (*) 1 [1,2,3]
6

他に fold できるのは？　お馴染み Maybe！ 

ghci> F.foldl (+) 2 (Just 9)
11
ghci> F.foldr (||) False (Just True)
True

Maybe の fold は、Just なら 1 要素のリスト、Nothing なら空リストの fold と同じで、あまり面白くありません。もう少し複雑な構造を：

独自の型と型クラスを作る 章の木構造： 

data Tree a = Empty | Node a (Tree a) (Tree a) deriving (Show, Read, Eq)

木は空木か、値と 2 つの部分木を持つノード。Functor インスタンスにして fmap できるようにしました。次に Foldable にして fold できるようにします。型コンストラクタを Foldable インスタンスにする方法の 1 つは、直接 foldr を実装すること。もう 1 つの簡単な方法は、Foldable 型クラスの一部である foldMap を実装すること： 

foldMap :: (Monoid m, Foldable t) => (a -> m) -> t a -> m

第 1 引数：foldable 構造の中身の型 a を取ってモノイド値を返す関数。第 2 引数：a を含む foldable 構造。関数を構造に map してモノイド値を含む構造を作り、それらを mappend で 1 つにまとめる。foldMap を実装するだけで Foldable インスタンスにでき、foldr・foldl も無料で手に入ります！

Tree の Foldable インスタンス： 

instance F.Foldable Tree where
    foldMap f Empty = mempty
    foldMap f (Node x l r) = F.foldMap f l `mappend`
                             f x           `mappend`
                             F.foldMap f r

考え方：木の要素を取ってモノイド値を返す関数があるとき、木全体を 1 つのモノイド値に縮約するには？　fmap ではノードに関数を適用して左右部分木に再帰しました。ここでは関数の map に加えて、結果を mappend で結合します。空木は値がないので mempty。

非空ノード：左右部分木に foldMap を再帰、ノードの値に f を適用。3 つのモノイド値（部分木 2 つとノード値）ができたので mappend で結合。左、ノード値、右の順で。

モノイド値を返す関数自体は foldMap の引数として受け取るので、私たちは「どこに適用するか・どう結合するか」を決めるだけ。

これで foldr・foldl が無料！　次の木： 

testTree = Node 5
            (Node 3
                (Node 1 Empty Empty)
                (Node 6 Empty Empty)
            )
            (Node 9
                (Node 8 Empty Empty)
                (Node 10 Empty Empty)
            )

ルート 5、左部分木のルートが 3（左 1、右 6）、右部分木のルートが 9（左 8、右 10）。Foldable インスタンスがあるのでリストと同じく fold できます： 

ghci> F.foldl (+) 0 testTree
42
ghci> F.foldl (*) 1 testTree
64800

foldMap は Foldable インスタンスを作るだけでなく、構造を 1 つのモノイドに縮約するのに便利。木に 3 と等しい値があるか調べたいなら： 

ghci> getAny $ F.foldMap (\x -> Any $ x == 3) testTree
True

\x -> Any $ x == 3 は数を取ってモノイド値（Bool を Any で包んだもの）を返す関数。foldMap がこの関数を木のすべての要素に適用し、結果を mappend で 1 つに縮約。これは： 

ghci> getAny $ F.foldMap (\x -> Any $ x > 15) testTree
False

木の各ノードはラムダ適用後 Any False。Any の mappend が True になるには少なくとも 1 つの True が必要。だから結果は False（木に 15 を超える値はないので筋が通る）。

\x -> [x] で foldMap すれば、木をリストにできます。各要素がシングルトンリストになり、mappend ですべて連結された 1 つのリスト： 

ghci> F.foldMap (\x -> [x]) testTree
[1,3,6,5,8,9,10]

クールなのは、これらの技は木に限らず Foldable インスタンスならどれでも使えること。
