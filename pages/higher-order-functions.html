高階関数

Haskell の関数は、関数を引数として取ったり、関数を戻り値として返したりできます。そういうことをする関数のことを「高階関数」と呼びます。高階関数は単に Haskell 体験の一部というだけでなく、ほとんど Haskell 体験そのものです。状態を変更するステップを並べてループで回すのではなく、ものごとが 何で あるかを定義することで計算を表現したいなら、高階関数は不可欠です。問題を解いたりプログラムを考えたりする本当に強力な手段なのです。

カリー化された関数

Haskell のすべての関数は、公式には引数を1つしか取りません。それなら、これまで複数の引数を取る関数をいくつも定義して使ってきたのはどういうことでしょう？　これは賢いトリックです！　ここまでで 複数の引数 を受け付けてきた関数は、すべて カリー化された関数 でした。どういう意味でしょう？　例で考えるのが一番です。お馴染みの max 関数を取り上げましょう。引数を2つ取って大きいほうを返すように見えますね。max 4 5 を実行すると、まず「引数を1つ取って、その引数と 4 のうち大きいほうを返す関数」が作られます。次にその関数に 5 が適用されて、ほしい結果が出ます。早口言葉のようですが、本当に素敵な概念です。次の2つの呼び出しは等価です：

ghci> max 4 5
5
ghci> (max 4) 5
5

2つのものの間にスペースを置くのは 関数適用 です。スペースは演算子のようなもので、優先順位は最も高いです。max の型を見てみましょう。max :: (Ord a) => a -> a -> a です。これは max :: (Ord a) => a -> (a -> a) とも書けます。「max は a を取り、（その -> です）a を取って a を返す関数を返す」と読めます。だから関数の戻り値の型と引数の型はすべて単にアロー（->）で区切られているのです。

これが何の役に立つのでしょう？　簡単に言えば、関数を引数の数より少なく呼び出すと、部分適用された 関数、つまり「省略した分の引数を取る関数」が返ってきます。部分適用（引数を少なく呼び出すこと）は、別の関数に渡すための関数を即興で作ったり、何かのデータを「あらかじめ仕込んだ」関数を作ったりする洒落た方法です。

このバカシンプルな関数を見てください：

multThree :: (Num a) => a -> a -> a -> a
multThree x y z = x * y * z

multThree 3 5 9 や ((multThree 3) 5) 9 を呼ぶと、何が起きているのでしょう？　まず 3 がスペースを挟んで multThree に適用されます。これにより、引数を1つ取って関数を返す関数ができます。そこに 5 が適用され、引数を取って 15 倍する関数ができます。最後に 9 がその関数に適用されて 135 が得られます。この関数の型は multThree :: (Num a) => a -> (a -> (a -> a)) とも書けます。-> の前にあるのが関数の取る引数で、後ろにあるのが返す値です。なのでこの関数は a を取って (Num a) => a -> (a -> a) 型の関数を返します。同様に、その関数も a を取って (Num a) => a -> a 型の関数を返し、最後の関数は a を取って a を返すだけです。これを見てください：

ghci> let multTwoWithNine = multThree 9
ghci> multTwoWithNine 2 3
54
ghci> let multWithEighteen = multTwoWithNine 2
ghci> multWithEighteen 10
180

関数を必要な引数より少なく呼び出すことで、新しい関数をその場で作っているわけです。数を取って 100 と比較する関数を作りたければ、こう書けます：

compareWithHundred :: (Num a, Ord a) => a -> Ordering
compareWithHundred x = compare 100 x

99 で呼べば GT が返ります。シンプルですね。両辺の右側に同じ x があることに注目。compare 100 が何を返すか考えてみましょう。「数を取って 100 と比較する関数」を返します。あれ？　欲しかった関数そのものでは？　書き換えられます：

compareWithHundred :: (Num a, Ord a) => a -> Ordering
compareWithHundred = compare 100

型宣言は変わりません。compare 100 が関数を返すからです。compare の型は (Ord a) => a -> (a -> Ordering) で、100 を渡して呼ぶと (Num a, Ord a) => a -> Ordering が返ります。100 が Num 型クラスにも属しているため、追加のクラス制約が紛れ込んでいます。

よっ！　カリー化と部分適用がどう動くか、ちゃんと理解しておいてくださいね。本当に大事ですから！

中置関数も「セクション」を使えば部分適用できます。中置関数をセクションにするには、括弧で囲んで片側だけに引数を書きます。すると「もう片側のオペランドが欠けている関数」が作れます。バカみたいに自明な関数：

divideByTen :: (Floating a) => a -> a
divideByTen = (/10)

divideByTen 200 は 200 / 10 や (/10) 200 と等価です。文字が大文字かを調べる関数も作れます：

isUpperAlphanum :: Char -> Bool
isUpperAlphanum = (`elem` ['A'..'Z'])

セクションで唯一気をつけるのは - です。セクションの定義からすると (-4) は「数を取って 4 を引く関数」になりそうですが、便宜上 (-4) はマイナス4を意味します。引数から 4 を引く関数を作りたければ、subtract 関数を部分適用して (subtract 4) と書きます。

GHCi で let で名前に束縛したり別の関数に渡したりせず、ただ multThree 3 4 と打つとどうなるでしょう？

ghci> multThree 3 4
<interactive>:1:0:
    No instance for (Show (t -> t))
      arising from a use of `print' at <interactive>:1:0-12
    Possible fix: add an instance declaration for (Show (t -> t))
    In the expression: print it
    In a 'do' expression: print it

GHCi は「式は a -> a 型の関数を返したけれど、それを画面に表示する方法を知らない」と言っています。関数は Show 型クラスのインスタンスではないので、関数の文字列表現は得られません。GHCi のプロンプトで 1 + 1 と入力すると、まず 2 に評価され、次に 2 に show が呼ばれて文字列表現を得ます。2 の文字列表現はただの文字列 "2" で、それが画面に出力されます。

いざ高階関数主義へ

関数は引数として関数を取れますし、戻り値として関数を返すこともできます。これを示すために、関数を取って何かに2回適用する関数を作ります！

applyTwice :: (a -> a) -> a -> a
applyTwice f x = f (f x)

まず型宣言に注目。これまで括弧は不要でした。-> は自然に右結合だからです。でもここでは括弧が必須です。最初の引数が「何かを取って同じ型の何かを返す関数」であることを示しているのです。2 つ目の引数も同じ型で、戻り値も同じ型です。この型宣言をカリー化された形で読むこともできますが、頭が痛くなるので「2つの引数を取って1つの値を返す関数」と読むことにします。最初の引数が関数（a -> a 型）、2つ目が同じ a。関数の型は Int -> Int や String -> String でも構いません。ただし2つ目の引数もそれに合わせる必要があります。

Note: ここからは、各関数が実際には引数を1つしか取らず、固体の値を返す関数に至るまで部分適用された関数を返す、という事実を承知の上で、「複数の引数を取る関数」と呼ぶことにします。話を簡単にするため、a -> a -> a は「2つの引数を取る」と言います。中で何が起きているかは分かっていますからね。

関数本体はとてもシンプル。引数 f を関数として使い、x をスペース区切りで適用し、その結果に再び f を適用します。とりあえず動かしてみましょう：

ghci> applyTwice (+3) 10
16
ghci> applyTwice (++ " HAHA") "HEY"
"HEY HAHA HAHA"
ghci> applyTwice ("HAHA " ++) "HEY"
"HAHA HAHA HEY"
ghci> applyTwice (multThree 2 2) 9
144
ghci> applyTwice (3:) [1]
[3,3,1]

部分適用の素晴らしさと有用性が見て取れます。引数を1つしか取らない関数を渡す必要があるとき、関数を部分適用して引数を1つだけ取る関数にしてから渡せます。

では高階プログラミングを使って、標準ライブラリにある便利な関数 zipWith を実装してみましょう。これは関数と2つのリストを引数に取り、対応する要素にその関数を適用して2つのリストを結合します。実装はこんな感じ：

zipWith' :: (a -> b -> c) -> [a] -> [b] -> [c]
zipWith' _ [] _ = []
zipWith' _ _ [] = []
zipWith' f (x:xs) (y:ys) = f x y : zipWith' f xs ys

型宣言を見てみましょう。最初の引数は2つの値を取って3つ目の値を返す関数。型は同じである必要はありませんが、同じでも構いません。2つ目と3つ目はリスト。結果もリストです。最初は a のリスト（結合関数の最初の引数が a なので）、2つ目は b のリスト（結合関数の2つ目の引数が b なので）、結果は c のリスト。型宣言が a -> b -> c 型の関数を引数に取ると言っていれば、a -> a -> a 型の関数も渡せます。逆は無理ですが！　関数を作っていて型に自信がないときは（特に高階関数の場合）、型宣言を省略して :t で Haskell の推論を確認するのが手です。

関数の中身は普通の zip によく似ています。エッジ条件も同じで、ただ結合関数という追加の引数があるだけです。エッジ条件ではその引数は関係ないので _ を使います。最後のパターンの本体も zip と似ていて、(x,y) ではなく f x y をするだけ。汎用的なら、1つの高階関数で大量のタスクをこなせます。zipWith' でできることをいくつか見てみましょう：

ghci> zipWith' (+) [4,2,5,6] [2,6,2,3]
[6,8,7,9]
ghci> zipWith' max [6,3,2,1] [7,3,1,5]
[7,3,2,5]
ghci> zipWith' (++) ["foo ", "bar ", "baz "] ["fighters", "hoppers", "aldrin"]
["foo fighters","bar hoppers","baz aldrin"]
ghci> zipWith' (*) (replicate 5 2) [1..]
[2,4,6,8,10]
ghci> zipWith' (zipWith' (*)) [[1,2,3],[3,5,6],[2,3,4]] [[3,2,2],[3,4,5],[5,4,3]]
[[3,4,6],[9,20,30],[10,12,12]]

このように、1 つの高階関数を本当に多彩に使えます。命令型プログラミングは、何らかの動作を実現するために for ループや while ループ、変数への代入、状態のチェックなどを使い、それを関数のようなインターフェースで包みます。関数型プログラミングは、共通のパターン（2 つのリストをペアで調べてペアごとに何かする、解の集合から不要なものを取り除く、など）を高階関数で抽象化します。

標準ライブラリにある別の関数 flip も実装してみましょう。flip は関数を取って、最初の2つの引数の順序を入れ替えただけの関数を返します。こう実装できます：

flip' :: (a -> b -> c) -> (b -> a -> c)
flip' f = g
    where g x y = f y x

型宣言を読み解くと「a と b を取る関数を取り、b と a を取る関数を返す」となります。ただし関数はデフォルトでカリー化されるので、2 つ目の括弧は実は不要です（-> はデフォルトで右結合）。(a -> b -> c) -> (b -> a -> c) は (a -> b -> c) -> (b -> (a -> c)) と同じで、それは (a -> b -> c) -> b -> a -> c と同じです。g x y = f y x と書きました。それが正しいなら、f y x = g x y も成り立つはず。これを念頭に、もっとシンプルに定義できます。

flip' :: (a -> b -> c) -> b -> a -> c
flip' f y x = f x y

ここでは関数がカリー化される性質を活用しています。flip' f を y と x なしで呼ぶと、その2つを入れ替えて呼ぶ f を返します。flip された関数は通常別の関数に渡されますが、高階関数を作るときに「最終的な呼び出し結果」を先取りして書くことで、カリー化を活用できます。

ghci> flip' zip [1,2,3,4,5] "hello"
[('h',1),('e',2),('l',3),('l',4),('o',5)]
ghci> zipWith (flip' div) [2,2..] [10,8,6,4,2]
[5,4,3,2,1]

mapとfilter

map は関数とリストを取り、リストの各要素に関数を適用して新しいリストを作ります。型シグネチャと定義を見てみましょう。

map :: (a -> b) -> [a] -> [b]
map _ [] = []
map f (x:xs) = f x : map f xs

型シグネチャから「a を取って b を返す関数、a のリストを取り、b のリストを返す」と分かります。型シグネチャを見るだけで、その関数が何をするか分かることもあるのは面白いですね。map は何百万通りもの使い方ができる、本当に汎用的な高階関数の1つです。動かしてみましょう：

ghci> map (+3) [1,5,3,1,6]
[4,8,6,4,9]
ghci> map (++ "!") ["BIFF", "BANG", "POW"]
["BIFF!","BANG!","POW!"]
ghci> map (replicate 3) [3..6]
[[3,3,3],[4,4,4],[5,5,5],[6,6,6]]
ghci> map (map (^2)) [[1,2],[3,4,5,6],[7,8]]
[[1,4],[9,16,25,36],[49,64]]
ghci> map fst [(1,2),(3,5),(6,3),(2,6),(2,5)]
[1,3,6,2,2]

これらはすべてリスト内包表記でも実現できることに気づいたでしょう。map (+3) [1,5,3,1,6] は [x+3 | x <- [1,5,3,1,6]] と同じです。とはいえ、リスト要素にただ関数を適用するだけなら map のほうがずっと読みやすいです。特に map の中で map を扱うようになると、括弧だらけで散らかってきます。

filter は述語（何かが真かどうか教えてくれる関数、つまり真偽値を返す関数）とリストを取り、述語を満たす要素だけからなるリストを返します。型シグネチャと実装：

filter :: (a -> Bool) -> [a] -> [a]
filter _ [] = []
filter p (x:xs)
    | p x       = x : filter p xs
    | otherwise = filter p xs

シンプル。p x が True なら新しいリストに含め、そうでなければ含めません。使用例：

ghci> filter (>3) [1,5,3,2,1,6,4,3,2,1]
[5,6,4]
ghci> filter (==3) [1,2,3,4,5]
[3]
ghci> filter even [1..10]
[2,4,6,8,10]
ghci> let notNull x = not (null x) in filter notNull [[1,2,3],[],[3,4,5],[2,2],[],[],[]]
[[1,2,3],[3,4,5],[2,2]]
ghci> filter (`elem` ['a'..'z']) "u LaUgH aT mE BeCaUsE I aM diFfeRent"
"uagameasadifeent"
ghci> filter (`elem` ['A'..'Z']) "i lauGh At You BecAuse u r aLL the Same"
"GAYBALLS"

これらもリスト内包表記の述語で同じことができます。map・filter とリスト内包表記、どちらを使うべきかに決まりはなく、コードと文脈に応じて読みやすい方を選びます。リスト内包表記で複数の述語を適用する filter 版は、「何回もフィルタする」か「述語を論理 && 関数で結合する」かのどちらかです。

前章 のクイックソートを覚えていますか？　ピボット以下の要素とピボットより大きい要素を分けるのにリスト内包表記を使いました。filter を使えば同じことをもっと読みやすく書けます：

quicksort :: (Ord a) => [a] -> [a]
quicksort [] = []
quicksort (x:xs) =
    let smallerSorted = quicksort (filter (<=x) xs)
        biggerSorted = quicksort (filter (>x) xs)
    in  smallerSorted ++ [x] ++ biggerSorted

map と filter は関数型プログラマにとっての主食です。うむ。map と filter 関数を使おうとリスト内包表記を使おうと違いはありません。前章で「特定の周長の直角三角形を見つける」問題を解いたのを思い出してください。命令型プログラミングなら 3 重ループで解いて、現在の組み合わせが直角三角形か、周長が正しいかをチェックして、合致したら出力していたでしょう。関数型プログラミングではそのパターンを map と filter で実現します。値を取って結果を返す関数を作り、その関数を値のリストに map し、結果を絞り込む述語で filter する。Haskell の遅延評価のおかげで、リストに map や filter を何回適用しても、リストを走査するのは1回だけです。

では 10 万未満で 3829 で割り切れる最大の数を見つけよう。解の存在範囲が分かっている可能性の集合を filter するだけです。

largestDivisible :: (Integral a) => a
largestDivisible = head (filter p [100000,99999..])
    where p x = x `mod` 3829 == 0

まず 10 万未満のすべての数を降順で並べたリストを作ります。それを述語で filter します。降順なので、述語を満たす最大の数はフィルタ後リストの先頭要素になります。最初のリストは有限である必要すらありませんでした。これが遅延評価の威力です。フィルタ後リストの先頭しか使わないので、フィルタ後のリストが有限か無限かは関係ないのです。最初の適切な解が見つかった時点で評価が止まります。

次に 1 万未満の奇数の平方の総和を求めよう。解には takeWhile 関数を使うので、先に紹介します。これは述語とリストを取り、リストの先頭から述語が真である間だけ要素を返します。述語が偽になる要素が見つかったら止まります。"elephants know how to party" の最初の単語を取りたければ takeWhile (/=' ') "elephants know how to party" で "elephants" が得られます。OK。1 万未満の奇数の平方の総和。まず無限リスト [1..] に (^2) を map します。次に奇数だけ filter します。そして 10000 未満のものだけ takeWhile で取り、最後に sum で総和を取ります。関数を定義する必要すらなく、GHCi で1行で書けます：

ghci> sum (takeWhile (<10000) (filter odd (map (^2) [1..])))
166650

すごい！　初期データ（自然数の無限リスト）から始め、map と filter と切り取りを使って必要な形に整え、最後に sum で合計する。リスト内包表記でも書けます：

ghci> sum (takeWhile (<10000) [n^2 | n <- [1..], odd (n^2)])
166650

どちらが綺麗かは好みの問題です。これも Haskell の遅延評価の性質があってこそ。無限リストに map や filter ができるのは、すぐに map や filter をするのではなく、その動作を遅らせるからです。Haskell に総和を見せろと強制してはじめて、sum 関数が takeWhile に「数が必要だ」と要求します。takeWhile はフィルタリングと map を強制しますが、10000 以上の数に出会うまでだけです。

次は Collatz 数列の問題です。自然数を取ります。偶数なら 2 で割ります。奇数なら 3 倍して 1 を足します。得られた数に同じ操作を繰り返し、新しい数を生成し続ける。要するに数の連鎖が得られます。すべての開始数について、連鎖は最終的に 1 で終わると考えられています。たとえば 13 から始めると 13, 40, 20, 10, 5, 16, 8, 4, 2, 1 という数列が得られます。13×3 + 1 は 40。40 ÷ 2 は 20。連鎖の長さは 10 ですね。

知りたいのはこれです：1 から 100 までの開始数のうち、長さが 15 を超える連鎖はいくつあるか？　まず、連鎖を生成する関数を書きます：

chain :: (Integral a) => a -> [a]
chain 1 = [1]
chain n
    | even n =  n:chain (n `div` 2)
    | odd n  =  n:chain (n*3 + 1)

連鎖は 1 で終わるので、それがエッジケース。標準的な再帰関数です。

ghci> chain 10
[10,5,16,8,4,2,1]
ghci> chain 1
[1]
ghci> chain 30
[30,15,46,23,70,35,106,53,160,80,40,20,10,5,16,8,4,2,1]

うまく動いてそう！　では問題に答える関数：

numLongChains :: Int
numLongChains = length (filter isLong (map chain [1..100]))
    where isLong xs = length xs > 15

[1..100] に chain 関数を map して連鎖のリスト（連鎖もリスト）を得ます。それを「リストの長さが 15 を超えるか」の述語で filter し、残った連鎖の数を数えます。

Note: この関数の型は numLongChains :: Int です。length が歴史的事情から Num a ではなく Int を返すからです。より一般的な Num a を返したければ、結果の長さに fromIntegral を使えます。

map を使うと map (*) [0..] のようなこともできます。これはカリー化の仕組みと、（部分適用された）関数も他の関数に渡せたりリストに入れたりできる本物の値だ（文字列に変換だけはできませんが）と示すための例です。これまで map (*2) [0..] のように、引数を1つ取る関数だけをリストに map してきました。これは (Num a) => [a] 型のリストを生みます。map (*) [0..] も問題なく書けます。リストの数が * 関数（型は (Num a) => a -> a -> a）に適用されているのです。2 引数を取る関数に1つだけ適用すれば、引数を1つ取る関数が返ります。* をリスト [0..] に map すると、引数を1つ取る関数のリスト、すなわち (Num a) => [a -> a] が得られます。map (*) [0..] は [(0*),(1*),(2*),(3*),(4*),(5*).. と書いたものに相当するリストを生成します。

ghci> let listOfFuns = map (*) [0..]
ghci> (listOfFuns !! 4) 5
20

リストの添字 4 の要素を取ると (4*) に相当する関数が返ります。それに 5 を適用するので、結果は (4*) 5 あるいは 4 * 5 と同じです。

ラムダ

ラムダは要するに無名関数で、関数を一度しか使わないときに使います。普通は高階関数に渡すためだけにラムダを作ります。ラムダを書くには、まず \ を書き（ギリシャ文字のラムダにちょっとだけ似ているので、ぐっと目を細めれば見えてきます）、次に引数をスペース区切りで書き、続けて ->、最後に関数本体を書きます。括弧で囲むのが普通です。そうしないとラムダが式の右端まで延びてしまうからです。

5 インチほど上にスクロールすれば、numLongChains 関数で where 束縛を使い、isLong 関数を作って filter に渡しているのが見えます。代わりにラムダを使ってみましょう：

numLongChains :: Int
numLongChains = length (filter (\xs -> length xs > 15) (map chain [1..100]))

ラムダも式なので、こうしてそのまま渡せます。(\xs -> length xs > 15) は「渡されたリストの長さが 15 を超えるかを返す関数」です。

カリー化と部分適用に慣れていない人は、必要のないところでもラムダを使いがちです。たとえば map (+3) [1,6,3,2] と map (\x -> x + 3) [1,6,3,2] は等価です。(+3) も (\x -> x + 3) も「数を取って 3 を足す関数」だからです。この場面でラムダを書くのは間が抜けています。部分適用のほうがずっと読みやすい。

普通の関数と同じく、ラムダも何個でも引数を取れます：

ghci> zipWith (\a b -> (a * 30 + 3) / b) [5,4,3,2,1] [1,2,3,4,5]
[153.0,61.5,31.0,15.75,6.6]

普通の関数と同じく、ラムダでもパターンマッチが使えます。違うのは、1つの引数に対して複数のパターンを定義してフォールスルーさせる、という使い方ができないこと（[] と (x:xs) を同じ引数に書く、みたいなのは無理）。ラムダでパターンマッチが失敗すると実行時エラーになるので、ラムダでパターンマッチするときは注意してください！

ghci> map (\(a,b) -> a + b) [(1,2),(3,5),(6,3),(2,6),(2,5)]
[3,8,9,8,7]

ラムダは右端まで延ばすつもりでない限り、普通は括弧で囲みます。面白いことに、関数がデフォルトでカリー化される性質ゆえ、次の2つは等価です：

addThree :: (Num a) => a -> a -> a -> a
addThree x y z = x + y + z

addThree :: (Num a) => a -> a -> a -> a
addThree = \x -> \y -> \z -> x + y + z

こう定義すれば、なぜ型宣言がそうなるのかは明らかです。型宣言にも等式にも -> が3つあります。とはいえ、当然ながら最初の書き方のほうがずっと読みやすく、2つ目はカリー化を示すための小ネタにすぎません。

とはいえ、この記法がイケる場面もあります。flip 関数はこう書くのが一番読みやすいと思います：

flip' :: (a -> b -> c) -> b -> a -> c
flip' f = \x y -> f y x

flip' f x y = f x y と同じですが、こう書くと「この関数はたいてい新しい関数を生むために使う」と明確になります。flip の最も多い使い方は、関数引数だけ与えて呼び、できた関数を map や filter に渡すこと。だから「主に部分適用して関数の引数として渡す」と明示したいときはラムダ表記が便利です。

foldと馬とロバ

再帰を扱っていたとき、リストを操作する多くの再帰関数に共通するテーマに気づいたと思います。普通、空リスト用のエッジケースを用意し、x:xs パターンを導入し、ある要素と残りのリストに関する処理を行います。これは非常によくあるパターンなので、それをカプセル化する超便利な関数がいくつか導入されました。それが fold です。map 関数のようなものですが、リストを単一の値に縮約します。

fold は、二項関数、初期値（私はアキュムレータと呼びたい）、畳み込みたいリストを取ります。二項関数自身は2つの引数を取ります。二項関数はアキュムレータと最初（または最後）の要素に対して呼ばれ、新しいアキュムレータを生みます。次に、新しいアキュムレータと次の最初（または最後）の要素に対して再び二項関数が呼ばれ、と繰り返されます。リスト全体を歩き終えると、アキュムレータだけが残り、それがリストを縮約した結果です。

まず foldl（左 fold とも呼ばれる）を見てみましょう。リストを左側から畳み込みます。二項関数は初期値とリストの先頭の間で適用され、新しいアキュムレータ値を生み、次に二項関数がその値と次の要素に対して呼ばれ、と続きます。

もう一度 sum を、今度は明示的な再帰ではなく fold で実装してみます。

sum' :: (Num a) => [a] -> a
sum' xs = foldl (\acc x -> acc + x) 0 xs

テスト、いち、にい、さん：

ghci> sum' [3,5,2,1]
11

fold がどう起こるかを詳しく見てみましょう。\acc x -> acc + x が二項関数。0 が初期値、xs が畳み込み対象のリスト。まず 0 が二項関数の acc 引数として、3 が x（現在の要素）として使われます。0 + 3 で 3 が生まれ、それが新しいアキュムレータになります。次に 3 がアキュムレータ、5 が現在の要素として使われ、8 が新しいアキュムレータに。さらに 8 と 2 で新しいアキュムレータは 10。最後に 10 と 1 で 11。おめでとう、fold できました！

左にあるプロフェッショナルな図は、fold が一段ずつ進む様子を示しています（一日一日！）。緑がかった茶色の数字がアキュムレータ値です。リストが左側からアキュムレータに食べられていくのが見えるでしょう。むしゃむしゃ！　関数のカリー化を踏まえれば、もっと簡潔に書けます：

sum' :: (Num a) => [a] -> a
sum' = foldl (+) 0

ラムダ (\acc x -> acc + x) は (+) と同じです。xs は省けます。foldl (+) 0 はリストを取る関数を返すからです。一般に foo a = bar b a のような関数は、カリー化のおかげで foo = bar b と書き換えられます。

右 fold に進む前に、もうひとつ左 fold で関数を実装してみます。elem は値がリストの中にあるかをチェックする関数ですね（おっと、説明してしまいました）。左 fold で実装します。

elem' :: (Eq a) => a -> [a] -> Bool
elem' y ys = foldl (\acc x -> if x == y then True else acc) False ys

さてさて、何でしょう。ここでの初期値とアキュムレータは真偽値です。fold ではアキュムレータの型と最終結果の型は常に同じです。初期値に何を使えばよいか分からないなら、これがヒントになります。False から始めるのは理にかなっています。「目的の値はまだ無い」と仮定するからです。空リストに fold を呼ぶと結果は単に初期値になりますし。次に、現在の要素が探している要素かをチェックします。一致したらアキュムレータを True に。違ったらアキュムレータをそのままに。前が False だったらこの要素は違うのでそのまま False、True ならそのまま True。

右 fold（foldr）も似たように動きますが、アキュムレータが値を右から食べていきます。また、左 fold の二項関数はアキュムレータが第1引数で現在の値が第2引数（\acc x -> ...）ですが、右 fold の二項関数は現在の値が第1引数でアキュムレータが第2引数（\x acc -> ...）です。右 fold ではアキュムレータが右にあるのは、右から畳み込むからだと考えれば納得です。

fold のアキュムレータ値（つまり結果）は任意の型でかまいません。数値、真偽値、新しいリストでもOK。右 fold で map 関数を実装してみます。アキュムレータはリスト、map 結果のリストを要素ごとに積み上げていきます。明らかに初期値は空リストです。

map' :: (a -> b) -> [a] -> [b]
map' f xs = foldr (\x acc -> f x : acc) [] xs

(+3) を [1,2,3] に map するとき、リストに右側から取り組みます。最後の要素 3 を取り、関数を適用して 6。アキュムレータ [] に前置すると 6:[] つまり [6]。これが新しいアキュムレータ。次に 2 に (+3) を適用して 5、これをアキュムレータに前置（:）して [5,6]。次に 1 に (+3) を適用して前置して、最終結果は [4,5,6]。

もちろん左 fold でも実装できます：map' f xs = foldl (\acc x -> acc ++ [f x]) [] xs。ただし ++ は : よりずっとコストが高いので、リストから新しいリストを作るときは普通右 fold を使います。

リストを反転すれば、左 fold するつもりだったところに右 fold を使えますし、その逆もできます。そもそも反転すら不要なときもあります。sum 関数は左 fold でも右 fold でもほぼ同じように実装できます。大きな違いは、右 fold は無限リストでも動くけれど左 fold は動かない、という点です。直感的に言えば、無限リストのある時点を取って右から畳めば、いつかリストの「先頭」に到達できます。一方、無限リストを左から畳もうとしたら、終わりにたどり着けないのです！

fold は、リストを一度走査して要素ごとに何かを取り出し、最終的に何かを返す関数すべての実装に使えます。リストを走査して何かを返したいときは、たぶん fold を使いたいときです。　だから fold は、map と filter と並んで、関数型プログラミングで最も役立つタイプの関数のひとつなのです。

foldl1 と foldr1 は foldl・foldr と似ていますが、明示的な初期値を渡す必要がありません。リストの最初（または最後）の要素を初期値として使い、その隣の要素から fold を始めます。たとえば sum は sum = foldl1 (+) と書けます。畳み込むリストに少なくとも1要素あることが前提なので、空リストで呼ぶと実行時エラーになります。foldl・foldr は空リストでも動きます。fold を作るときは、空リストでどう振る舞うか考えましょう。空リストで意味をなさない関数なら、foldl1・foldr1 で実装できる可能性が高いです。

fold のパワーを示すために、標準ライブラリの関数をいろいろ fold で実装してみましょう：

maximum' :: (Ord a) => [a] -> a
maximum' = foldr1 (\x acc -> if x > acc then x else acc)

reverse' :: [a] -> [a]
reverse' = foldl (\acc x -> x : acc) []

product' :: (Num a) => [a] -> a
product' = foldr1 (*)

filter' :: (a -> Bool) -> [a] -> [a]
filter' p = foldr (\x acc -> if p x then x : acc else acc) []

head' :: [a] -> a
head' = foldr1 (\x _ -> x)

last' :: [a] -> a
last' = foldl1 (\_ x -> x)

head はパターンマッチで実装するほうが良いですが、fold でも書けると示すためです。reverse' の定義はけっこう賢いと思います。空リストを初期値として、左からリストにアプローチし、ただアキュムレータに前置していくだけ。最終的に反転リストが組み上がります。\acc x -> x : acc は : 関数の引数を入れ替えたものに見えます。だから reverse は foldl (flip (:)) [] とも書けます。

右 fold と左 fold の別の見方として：右 fold で二項関数を f、初期値を z とし、[3,4,5,6] を右畳み込みするなら、要するに f 3 (f 4 (f 5 (f 6 z))) をしています。f はリストの最後の要素とアキュムレータで呼ばれ、その値が次の要素のアキュムレータになり、と続きます。f を +、初期値を 0 にすると 3 + (4 + (5 + (6 + 0)))。+ を前置関数で書けば (+) 3 ((+) 4 ((+) 5 ((+) 6 0)))。同じく、二項関数 g、アキュムレータ z での左 fold は g (g (g (g z 3) 4) 5) 6 と等価です。flip (:) をニ項関数、空リスト [] をアキュムレータにすると（リスト反転）、flip (:) (flip (:) (flip (:) (flip (:) [] 3) 4) 5) 6 です。これを評価すると確かに [6,5,4,3] になります。

scanl・scanr は foldl・foldr に似ていますが、途中のアキュムレータの状態をリストとして報告してくれます。scanl1・scanr1 もあり、foldl1・foldr1 に対応します。

ghci> scanl (+) 0 [3,5,2,1]
[0,3,8,10,11]
ghci> scanr (+) 0 [3,5,2,1]
[11,8,3,1,0]
ghci> scanl1 (\acc x -> if x > acc then x else acc) [3,4,5,3,7,9,2,1]
[3,4,5,5,7,9,9,9]
ghci> scanl (flip (:)) [] [3,2,1]
[[],[3],[2,3],[1,2,3]]

scanl では最終結果は結果リストの末尾に、scanr では結果リストの先頭に置かれます。

scan は、fold として実装できる関数の進捗を追うのに使います。こんな質問に答えてみましょう：すべての自然数の平方根の和が 1000 を超えるには、何個の要素が必要か？　すべての自然数の平方根を得るには map sqrt [1..]。和を求めるなら fold ですが、和の進捗が知りたいので scan を使います。scan のあと、1000 未満の和がいくつあるか数えるだけ。scan リストの最初の和は 1。2 つ目は 1 + √2、3 つ目はそれ + √3、と続きます。1000 未満の和が X 個あれば、和が 1000 を超えるには X+1 個の要素が必要です。

sqrtSums :: Int
sqrtSums = length (takeWhile (<1000) (scanl1 (+) (map sqrt [1..]))) + 1

ghci> sqrtSums
131
ghci> sum (map sqrt [1..131])
1005.0942035344083
ghci> sum (map sqrt [1..130])
993.6486803921487

ここでは filter ではなく takeWhile を使っています。filter は無限リストでは動かないからです。リストが昇順だと我々は知っていても filter は知らないので、takeWhile で和が 1000 を超えた最初の地点で scan リストを切り取ります。

$による関数適用

次に、$ 関数（関数適用 とも呼ばれる）を見てみましょう。まず定義をチェック：

($) :: (a -> b) -> a -> b
f $ x = f x

えっ？　この役立たずの演算子は何？　ただの関数適用じゃん！　まあほぼそう、でも完全に同じではありません！　通常の関数適用（2 つのものの間にスペース）は優先順位が非常に高いのに対し、$ 関数は最低の優先順位を持ちます。スペースによる関数適用は左結合（f a b c は ((f a) b) c）ですが、$ による関数適用は右結合です。

それで何の役に？　多くの場合、これは便利関数で、括弧をたくさん書かなくて済むようにしてくれます。sum (map sqrt [1..130]) という式を考えましょう。$ は優先順位が低いので、sum $ map sqrt [1..130] と書き換えられます。貴重なキー入力が節約できる！　$ に出会うと、その右側の式が左側の関数の引数として適用されます。sqrt 3 + 4 + 9 はどうでしょう？　これは 9, 4, √3 を足します。3 + 4 + 9 の平方根を取りたければ sqrt (3 + 4 + 9) と書くか、$ を使って sqrt $ 3 + 4 + 9 と書けます（$ はあらゆる演算子の中で最低の優先順位）。$ はちょうど「開き括弧を書いて、式の右端まで延ばして閉じ括弧を書いた」のと同じ役割だと考えられます。

sum (filter (> 10) (map (*2) [2..10])) はどうでしょう？　$ は右結合なので、f (g (z x)) は f $ g $ z x と等価です。sum (filter (> 10) (map (*2) [2..10])) は sum $ filter (> 10) $ map (*2) [2..10] と書き換えられます。

括弧を消す以外にも、$ によって関数適用を別の関数のように扱えるようになります。たとえば関数のリストに対して関数適用を map できます。

ghci> map ($ 3) [(4+), (10*), (^2), sqrt]
[7.0,30.0,9.0,1.7320508075688772]

関数合成

数学で関数合成は次のように定義されます：。2 つの関数を合成すると新しい関数ができ、その関数は引数 x で呼ばれると、まず g を x で呼び、その結果に f を適用したのと同じ結果を返します。

Haskell でも関数合成はほぼ同じです。関数合成は . 関数で行います。定義は次の通り：

(.) :: (b -> c) -> (a -> b) -> a -> c
f . g = \x -> f (g x)

型宣言に注目。f は g の戻り値と同じ型の値を引数に取らなければなりません。結果の関数は g が取るのと同じ型の引数を取り、f が返すのと同じ型の値を返します。negate . (* 3) は「数を取って 3 倍してから符号を反転する関数」を返します。

関数合成の用途のひとつは、別の関数に渡すための関数をその場で作ることです。ラムダでもできますが、関数合成のほうが明快で簡潔なことが多いです。たとえば数のリストをすべて負の数に変換したいなら、各数の絶対値を取ってから符号を反転すればよいです：

ghci> map (\x -> negate (abs x)) [5,-3,-6,7,-3,2,-19,24]
[-5,-3,-6,-7,-3,-2,-19,-24]

ラムダの形が関数合成の結果に似ていることに注目。関数合成を使うとこう書き換えられます：

ghci> map (negate . abs) [5,-3,-6,7,-3,2,-19,24]
[-5,-3,-6,-7,-3,-2,-19,-24]

素晴らしい！　関数合成は右結合なので、複数の関数を一度に合成できます。f (g (z x)) は (f . g . z) x と同じです。これを踏まえて、

ghci> map (\xs -> negate (sum (tail xs))) [[1..5],[3..6],[1..7]]
[-14,-15,-27]

を

ghci> map (negate . sum . tail) [[1..5],[3..6],[1..7]]
[-14,-15,-27]

と書き換えられます。

でも、複数の引数を取る関数はどうでしょう？　関数合成で使いたいなら、各関数が引数を1つだけ取るところまで部分適用するのが普通です。sum (replicate 5 (max 6.7 8.9)) は (sum . replicate 5 . max 6.7) 8.9 や sum . replicate 5 . max 6.7 $ 8.9 と書き換えられます。何が起きているかというと：「max 6.7 が取るものを取って replicate 5 を適用する関数」が作られ、次に「その結果を取って sum する関数」が作られ、最後にその関数が 8.9 で呼ばれる、という流れ。普通は「8.9 を max 6.7 に適用し、その結果に replicate 5 を適用し、さらに sum を適用する」と読みます。括弧だらけの式を関数合成で書き直したいなら、まず一番内側の関数の最後の引数を $ の後ろに置き、他の関数呼び出しを最後の引数を取り除きつつドットでつなげます。replicate 100 (product (map (*3) (zipWith max [1,2,3,4,5] [4,5,6,7,8]))) なら replicate 100 . product . map (*3) . zipWith max [1,2,3,4,5] $ [4,5,6,7,8] と書けます。式が3つの閉じ括弧で終わるなら、関数合成にすると合成演算子も3つになる可能性が高いです。

関数合成のもう1つの一般的な使い方は、いわゆるポイントフリースタイル（ポイントレススタイルとも呼ばれる）で関数を定義することです。先ほど書いたこの関数を例に：

sum' :: (Num a) => [a] -> a
sum' xs = foldl (+) 0 xs

右辺の両側に xs が出ています。カリー化のおかげで両側の xs を省けます。foldl (+) 0 はリストを取る関数を返すからです。sum' = foldl (+) 0 と書くのがポイントフリースタイル。次の関数をポイントフリーで書けるでしょうか？

fn x = ceiling (negate (tan (cos (max 50 x))))

右辺両側の x を消すわけにはいきません。本体の x の後ろに括弧があります。cos (max 50) は意味をなしません。関数の cos なんて取れない。代わりにできるのは、fn を関数合成で表現することです。

fn = ceiling . negate . tan . cos . max 50

素晴らしい！　ポイントフリースタイルは、データがどう動き回るかではなく、関数とそれを合成して何ができるかを考えさせるので、より読みやすく簡潔なことが多いです。シンプルな関数を糊（合成）でつなげて複雑な関数を作れます。ただし、関数が複雑すぎるとポイントフリーは読みにくくなることもあるので、長い合成チェーンは推奨されません。私自身もときどき合成しすぎてしまうので人のことは言えませんが。好ましいのは、let 束縛で中間結果にラベルを付けたり、問題をサブ問題に分けて組み合わせることで、巨大な合成チェーンではなく読み手にとって意味の通る関数にすることです。

map と filter のセクションで、1 万未満の奇数の平方の総和を求めました。それを関数として書くとこうなります。

oddSquareSum :: Integer
oddSquareSum = sum (takeWhile (<10000) (filter odd (map (^2) [1..])))

関数合成大好き人間としては、私ならこう書くでしょう：

oddSquareSum :: Integer
oddSquareSum = sum . takeWhile (<10000) . filter odd . map (^2) $ [1..]

でも他の人が読む可能性があるなら、こう書きます：

oddSquareSum :: Integer
oddSquareSum =
    let oddSquares = filter odd $ map (^2) [1..]
        belowLimit = takeWhile (<10000) oddSquares
    in  sum belowLimit

コードゴルフでは勝てませんが、合成チェーンより読みやすいと感じる人が多いはずです。
