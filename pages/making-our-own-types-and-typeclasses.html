独自の型と型クラスを作る

これまで Haskell の既存の型と型クラスをいくつか扱ってきました。この章では、自分で型と型クラスを作って働かせる方法を学びます！

代数的データ型入門

これまでに Bool、Int、Char、Maybe などたくさんのデータ型に出会ってきました。では自前のデータ型はどう作るのでしょう？　1 つの方法は data キーワードで型を定義することです。標準ライブラリでの Bool 型の定義を見てみましょう。

data Bool = False | True

data は新しいデータ型を定義することを意味します。= の前の部分が型で、ここでは Bool。後の部分は 値コンストラクタ です。これらが、この型が取りうる値を指定します。| は または と読みます。「Bool 型は True または False という値を持ちうる」と読めます。型名と値コンストラクタはどちらも大文字で始まる必要があります。

同じように Int 型は次のように定義されている、と考えられます：

data Int = -2147483648 | -2147483647 | ... | -1 | 0 | 1 | 2 | ... | 2147483647

最初と最後の値コンストラクタが Int の最小値と最大値です。実際にはこう定義されているわけではなく、間の数を山ほど省略しているのは説明のためです。

図形を Haskell でどう表すか考えましょう。1 つの方法はタプルです。円は (43.1, 55.0, 10.4) のように、最初の2要素が中心の座標、3つ目が半径と表せます。それでも一応動きますが、3D ベクトルにも見えるし、何にでも見えてしまいます。図形を表す独自の型を作るほうが良い解です。図形は円か長方形のどちらかとしましょう：

data Shape = Circle Float Float Float | Rectangle Float Float Float Float

さて、これは何でしょう？　こう考えてください。Circle 値コンストラクタは Float を取る3つのフィールドを持ちます。値コンストラクタを書くとき、後ろに型を続けて、それが「保持する値」を定義できるのです。ここでは最初の2つが中心の座標、3つ目が半径。Rectangle 値コンストラクタは Float を取る4つのフィールド。最初の2つが左上の座標、後の2つが右下の座標です。

「フィールド」と言いましたが、要するに引数のことです。値コンストラクタは、最終的にデータ型の値を返す関数なのです。これら2つの値コンストラクタの型シグネチャを見てみましょう。

ghci> :t Circle
Circle :: Float -> Float -> Float -> Shape
ghci> :t Rectangle
Rectangle :: Float -> Float -> Float -> Float -> Shape

クール、値コンストラクタは普通の関数なんですね。誰が想像したでしょう？　図形を取って表面積を返す関数を作ってみます。

surface :: Shape -> Float
surface (Circle _ _ r) = pi * r ^ 2
surface (Rectangle x1 y1 x2 y2) = (abs $ x2 - x1) * (abs $ y2 - y1)

まず注目すべきは型宣言。「図形を取って Float を返す」と言っています。Circle -> Float とは書けません。Circle は型ではなく、Shape が型だからです。True -> Int という型宣言の関数が書けないのと同じです。次に注目すべきは、コンストラクタに対してパターンマッチできることです。これまでもパターンマッチの相手はコンストラクタでしたが（[]、False、5 など）、それらにはフィールドがありませんでした。コンストラクタを書いてフィールドを名前に束縛するだけ。半径だけが欲しいので、円の位置を表す最初の2つのフィールドは気にしません。

ghci> surface $ Circle 10 20 10
314.15927
ghci> surface $ Rectangle 0 0 100 100
10000.0

動きました！　でもプロンプトでただ Circle 10 20 5 と打つとエラーになります。Haskell はまだ私たちのデータ型を文字列にして表示する方法を知らないからです。プロンプトで値を表示するとき、Haskell はまず show 関数で値の文字列表現を取得し、それを端末に出力します。Shape 型を Show 型クラスのメンバーにするには、こう書きます：

data Shape = Circle Float Float Float | Rectangle Float Float Float Float deriving (Show)

deriving についてはまだ深く立ち入りません。今のところは、data 宣言の最後に deriving (Show) を加えると、Haskell が自動でその型を Show 型クラスの一員にしてくれる、と思っておけば十分。これでこうできます：

ghci> Circle 10 20 5
Circle 10.0 20.0 5.0
ghci> Rectangle 50 230 60 90
Rectangle 50.0 230.0 60.0 90.0

値コンストラクタは関数なので、map したり部分適用したりできます。半径違いの同心円のリストが欲しいなら：

ghci> map (Circle 10 20) [4,5,6,6]
[Circle 10.0 20.0 4.0,Circle 10.0 20.0 5.0,Circle 10.0 20.0 6.0,Circle 10.0 20.0 6.0]

悪くないですが、もっと良くできます。2 次元の点を定義する中間データ型を作って、図形をもっと分かりやすくしましょう。

data Point = Point Float Float deriving (Show)
data Shape = Circle Point Float | Rectangle Point Point deriving (Show)

点を定義するとき、データ型と値コンストラクタに同じ名前を使ったことに注目。特別な意味はありませんが、値コンストラクタが1つしかないときは型名と同じにするのが慣習です。Circle は Point 型と Float 型のフィールドを持ち、何が何だか分かりやすくなりました。長方形も同様。surface 関数も合わせて変えます：

surface :: Shape -> Float
surface (Circle _ r) = pi * r ^ 2
surface (Rectangle (Point x1 y1) (Point x2 y2)) = (abs $ x2 - x1) * (abs $ y2 - y1)

変えたのはパターンだけ。円のパターンでは点全体を無視。長方形のパターンではネストしたパターンマッチで点のフィールドを取り出しています。点そのものを参照したければ as パターンが使えます。

ghci> surface (Rectangle (Point 0 0) (Point 100 100))
10000.0
ghci> surface (Circle (Point 0 0) 24)
1809.5574

図形を「ぐっと押す」関数はどうでしょう？　図形と x 軸方向の移動量、y 軸方向の移動量を取り、寸法はそのまま、位置だけ動いた新しい図形を返します。

nudge :: Shape -> Float -> Float -> Shape
nudge (Circle (Point x y) r) a b = Circle (Point (x+a) (y+b)) r
nudge (Rectangle (Point x1 y1) (Point x2 y2)) a b = Rectangle (Point (x1+a) (y1+b)) (Point (x2+a) (y2+b))

シンプル。図形の位置を表す点に nudge 量を足しているだけです。

ghci> nudge (Circle (Point 34 34) 10) 5 10
Circle (Point 39.0 44.0) 10.0

点を直接扱いたくなければ、原点で図形を作る補助関数を作って、それを nudge できます。

baseCircle :: Float -> Shape
baseCircle r = Circle (Point 0 0) r

baseRect :: Float -> Float -> Shape
baseRect width height = Rectangle (Point 0 0) (Point width height)

ghci> nudge (baseRect 40 100) 60 23
Rectangle (Point 60.0 23.0) (Point 100.0 123.0)

もちろんモジュールでデータ型をエクスポートできます。エクスポートする関数と一緒に型名を書き、その後ろに括弧で値コンストラクタをカンマ区切りで指定します。すべての値コンストラクタをエクスポートしたいなら .. と書きます。

ここで定義した型と関数をモジュールでエクスポートするなら、こんな感じで始まります：

module Shapes
( Point(..)
, Shape(..)
, surface
, nudge
, baseCircle
, baseRect
) where

Shape(..) で Shape のすべての値コンストラクタをエクスポートしたので、import 側は Rectangle・Circle 値コンストラクタで図形を作れます。Shape (Rectangle, Circle) と書くのと同じです。

Shape の値コンストラクタを一切エクスポートしないなら、エクスポート文に単に Shape と書きます。すると import 側は baseCircle・baseRect といった補助関数経由でしか図形を作れません。Data.Map はこの方式です。Map.Map [(1,2),(3,4)] ではマップを作れず、Map.fromList のような補助関数経由で作ります。値コンストラクタはフィールドを引数に取って何らかの型（Shape など）の値を返す関数なので、エクスポートしないことで利用者がそれらの関数を使うのを防げます。ただし、その型を返す他のエクスポート関数があれば、それで型の値を作れます。

値コンストラクタをエクスポートしないと、データ型はある意味抽象的になり実装が隠れます。利用者は値コンストラクタにパターンマッチもできません。

レコード構文

OK、人物を表すデータ型を作るタスクに取り組むことにします。保持したい情報は：名、姓、年齢、身長、電話番号、好きなアイスのフレーバー。私はそれくらいしか他人について知りたくありません。やってみましょう！

data Person = Person String String Int Float String String deriving (Show)

OK。1 つ目が名、2 つ目が姓、3 つ目が年齢、と続きます。人を作ってみます。

ghci> let guy = Person "Buddy" "Finklestein" 43 184.2 "526-2928" "Chocolate"
ghci> guy
Person "Buddy" "Finklestein" 43 184.2 "526-2928" "Chocolate"

悪くないですが、ちょっと読みにくいですね。人物から個別の情報を取り出す関数を作りたいときは？　名前を取り出す関数、姓を取り出す関数、と個別に。こんな感じになるでしょう。

firstName :: Person -> String
firstName (Person firstname _ _ _ _ _) = firstname

lastName :: Person -> String
lastName (Person _ lastname _ _ _ _) = lastname

age :: Person -> Int
age (Person _ _ age _ _ _) = age

height :: Person -> Float
height (Person _ _ _ height _ _) = height

phoneNumber :: Person -> String
phoneNumber (Person _ _ _ _ number _) = number

flavor :: Person -> String
flavor (Person _ _ _ _ _ flavor) = flavor

ふー！　書いていて全然楽しくない！　面倒で退屈ですが、一応動きます。

ghci> let guy = Person "Buddy" "Finklestein" 43 184.2 "526-2928" "Chocolate"
ghci> firstName guy
"Buddy"
ghci> height guy
184.2
ghci> flavor guy
"Chocolate"

もっと良い方法があるはずだ、と思いますか？　残念ながらありません。

……というのは冗談で、ちゃんとあります。Haskell の作者たちは賢く、こうしたケースを予期していました。データ型を書く別の方法を用意してくれているのです。レコード構文で同じことを実現するとこうなります。

data Person = Person { firstName :: String
                     , lastName :: String
                     , age :: Int
                     , height :: Float
                     , phoneNumber :: String
                     , flavor :: String
                     } deriving (Show)

フィールドの型をスペース区切りで並べる代わりに、波括弧を使います。フィールド名（たとえば firstName）、ダブルコロン ::（Paamayim Nekudotayim とも呼ばれる、笑）、続けて型を指定します。データ型自体はまったく同じものです。主な利点は、フィールドを引くための関数が自動的に作られること。レコード構文を使うことで、Haskell が自動で firstName、lastName、age、height、phoneNumber、flavor 関数を作ってくれます。

ghci> :t flavor
flavor :: Person -> String
ghci> :t firstName
firstName :: Person -> String

レコード構文にはもう1つ利点があります。Show を導出すると、レコード構文で定義・インスタンス化した場合は表示形式が違うのです。車を表す型が会社・モデル名・製造年を保持するとして：

data Car = Car String String Int deriving (Show)

ghci> Car "Ford" "Mustang" 1967
Car "Ford" "Mustang" 1967

レコード構文で定義すれば、新しい車をこう作れます。

data Car = Car {company :: String, model :: String, year :: Int} deriving (Show)

ghci> Car {company="Ford", model="Mustang", year=1967}
Car {company = "Ford", model = "Mustang", year = 1967}

レコード構文なら、すべてのフィールドを列挙する限り順序を気にしなくて良いです。レコード構文を使わないなら順番通りに指定する必要があります。

レコード構文は、コンストラクタが多くのフィールドを持っていてどれが何か自明でないときに使いましょう。3D ベクトルを data Vector = Vector Int Int Int と定義するなら、フィールドがベクトルの成分なのは明らかですが、Person や Car ではそうはいかないので、レコード構文の恩恵が大きいです。

型引数

値コンストラクタは値引数を取って新しい値を生み出します。Car コンストラクタは3つの値を取って車の値を生みます。同様に 型コンストラクタ は型を引数に取って新しい型を生みます。最初は抽象的すぎに聞こえるかもしれませんが、それほど複雑ではありません。C++ のテンプレートに馴染みがあれば類似点が見えます。型引数の動きを把握するために、すでに見たことのある型の実装を見てみましょう。

data Maybe a = Nothing | Just a

ここで a が型引数です。型引数があるので Maybe は型コンストラクタです。Nothing でないときに何を保持するかによって、この型コンストラクタは Maybe Int、Maybe Car、Maybe String といった型を生みます。値が単なる Maybe 型ということはありません。Maybe はそれ自体は型ではなく、型コンストラクタだからです。値が属せる本物の型になるには、型引数をすべて埋める必要があります。

Maybe に Char を型引数として渡すと、Maybe Char 型が得られます。Just 'a' の型は Maybe Char、というように。

気づいていないかもしれませんが、Maybe より前にも型引数を持つ型を使っています。リスト型です。糖衣構文があるので分かりにくいですが、リスト型は引数を1つ取って具体型を生みます。値の型は [Int]、[Char]、[[String]] などになりえますが、ただの [] 型の値はありません。

Maybe 型で遊んでみます。

ghci> Just "Haha"
Just "Haha"
ghci> Just 84
Just 84
ghci> :t Just "Haha"
Just "Haha" :: Maybe [Char]
ghci> :t Just 84
Just 84 :: (Num t) => Maybe t
ghci> :t Nothing
Nothing :: Maybe a
ghci> Just 10 :: Maybe Double
Just 10.0

型引数のおかげで、データ型に含めたい型に応じて違う型を作れます。:t Just "Haha" をすると、型推論エンジンが Maybe [Char] と判断します。Just a の a が文字列なら Maybe a の a も文字列だからです。

Nothing の型が Maybe a となっているのに注目。多相型です。Maybe Int を引数に取る関数があれば Nothing を渡せます。Nothing はそもそも値を含まないので関係ないからです。Maybe a 型は必要なら Maybe Int として振る舞えます。5 が Int としても Double としても振る舞えるのと同じです。同様に空リストの型は [a]。空リストはどんなリストとしても振る舞えます。だから [1,2,3] ++ [] や ["ha","ha","ha"] ++ [] ができるのです。

型引数は便利ですが、使うのが理にかなうときだけ使うべきです。普通、内部に保持する値の型に関係なくデータ型が機能する場合（Maybe a など）に使います。型が「箱」のように振る舞うなら、型引数を使うのが良いです。Car をこう変えることもできます：

data Car = Car { company :: String
               , model :: String
               , year :: Int
               } deriving (Show)

こう：

data Car a b c = Car { company :: a
                     , model :: b
                     , year :: c
                     } deriving (Show)

でも本当に得をするでしょうか？　答えはたぶん「いいえ」です。結局 Car String String Int 型でしか動かない関数を定義することになるからです。たとえば最初の Car 定義なら、車のプロパティを表示する小さな関数を作れます。

tellCar :: Car -> String
tellCar (Car {company = c, model = m, year = y}) = "This " ++ c ++ " " ++ m ++ " was made in " ++ show y

ghci> let stang = Car {company="Ford", model="Mustang", year=1967}
ghci> tellCar stang
"This Ford Mustang was made in 1967"

かわいい関数！　型宣言もすっきりで動きます。Car が Car a b c だったらどうでしょう？

tellCar :: (Show a) => Car String String a -> String
tellCar (Car {company = c, model = m, year = y}) = "This " ++ c ++ " " ++ m ++ " was made in " ++ show y

関数を (Show a) => Car String String a 型の Car に縛らないといけません。型シグネチャは複雑になり、得るものといえば c として Show 型クラスのインスタンスを使えるようになるくらい。

ghci> tellCar (Car "Ford" "Mustang" 1967)
"This Ford Mustang was made in 1967"
ghci> tellCar (Car "Ford" "Mustang" "nineteen sixty seven")
"This Ford Mustang was made in \"nineteen sixty seven\""
ghci> :t Car "Ford" "Mustang" 1967
Car "Ford" "Mustang" 1967 :: (Num t) => Car [Char] [Char] t
ghci> :t Car "Ford" "Mustang" "nineteen sixty seven"
Car "Ford" "Mustang" "nineteen sixty seven" :: Car [Char] [Char] [Char]

実生活ではほとんどの場合 Car String String Int を使うので、Car をパラメータ化する価値はあまりなさそうです。型引数は、値コンストラクタの中身の型がデータ型の機能に本質的でないときに使います。何かのリストはなんであれリストですし、何の型かは関係なく動きます。数のリストの和を求めたければ、和を取る関数の側で「数のリスト」と指定すればよい。Maybe も同じ。「何もないか、何かが1つあるか」を表すだけで、その何かが何の型かは関係ありません。

すでに出会ったパラメータ化された型のもう1つの例は Data.Map の Map k v。k がキーの型、v が値の型。型引数が役立つ良い例です。マップがパラメータ化されているおかげで、キーが Ord 型クラスのメンバーである限り、任意の型から任意の型へのマップが作れます。マップ型を定義するなら data 宣言にクラス制約を追加できそうです：

data (Ord k) => Map k v = ...

しかし Haskell では data 宣言にクラス制約を絶対に書かない という強い慣習があります。なぜ？　ほとんど得がないのに、不要な場面でもクラス制約を書く羽目になるからです。Map k v の data 宣言に Ord k を入れても入れなくても、キーが順序付け可能だと仮定する関数には制約を書く必要があります。一方、data 宣言に制約を入れなければ、キーの順序を気にしない関数の型宣言に (Ord k) => を書かなくて済みます。たとえば toList はマップを連想リストに変換するだけで、型シグネチャは toList :: Map k a -> [(k, a)]。Map k v の data 宣言に型制約があれば、toList の型は toList :: (Ord k) => Map k a -> [(k, a)] にしないといけません。比較しないのに。

だから理にかなって見えても data 宣言には型制約を書かないこと。どのみち関数の型宣言に書くハメになります。

3D ベクトル型を実装し、いくつか演算を加えてみましょう。普通は数値型を使いますが、複数の数値型をサポートできるよう、パラメータ化された型を使います。

data Vector a = Vector a a a deriving (Show)

vplus :: (Num t) => Vector t -> Vector t -> Vector t
(Vector i j k) `vplus` (Vector l m n) = Vector (i+l) (j+m) (k+n)

vectMult :: (Num t) => Vector t -> t -> Vector t
(Vector i j k) `vectMult` m = Vector (i*m) (j*m) (k*m)

scalarMult :: (Num t) => Vector t -> Vector t -> t
(Vector i j k) `scalarMult` (Vector l m n) = i*l + j*m + k*n

vplus はベクトルの和、対応成分どうしを足します。scalarMult はスカラー積、vectMult はベクトルとスカラーの積です。Vector a の a が Num 型クラスのメンバーであれば、Vector Int、Vector Integer、Vector Float など何でも動きます。型宣言を見ると、同じ型のベクトル同士、関わる数もベクトルが含む型と同じである必要があります。data 宣言に Num クラス制約は入れていません。どのみち関数で繰り返さなければならないからです。

もう一度強調しますが、型コンストラクタと値コンストラクタの区別は重要です。データ型を宣言するとき、= の前が型コンストラクタ、後ろのコンストラクタ（| で区切られている）は値コンストラクタです。関数に Vector t t t -> Vector t t t -> t のような型を与えるのは間違いです。型宣言には型を書かねばならず、ベクトル 型 コンストラクタは引数を1つしか取らない（値コンストラクタは3つ取る）からです。ベクトルで遊んでみましょう。

ghci> Vector 3 5 8 `vplus` Vector 9 2 8
Vector 12 7 16
ghci> Vector 3 5 8 `vplus` Vector 9 2 8 `vplus` Vector 0 2 3
Vector 12 9 19
ghci> Vector 3 9 7 `vectMult` 10
Vector 30 90 70
ghci> Vector 4 9 5 `scalarMult` Vector 9.0 2.0 4.0
74.0
ghci> Vector 2 9 3 `vectMult` (Vector 4 9 5 `scalarMult` Vector 9 2 4)
Vector 148 666 222

派生インスタンス

型クラス入門 のセクションで、型クラスの基本を説明しました。型クラスはある振る舞いを定義するインターフェースのようなもの、と。型がその振る舞いをサポートするなら、その型を インスタンス にできます。例：Int 型は Eq 型クラスのインスタンス。Eq 型クラスは等価性を扱うものを定義し、整数は等価性を扱えるからです。本当の有用性は、Eq のインターフェースとなる関数（==、/=）にあります。型が Eq のメンバーなら、その型の値で == を使える。だから 4 == 4 や "foo" /= "bar" が型チェックを通るのです。

Java、Python、C++ などの「クラス」と混同されることがあって、多くの人を混乱させます。それらの言語では、クラスは状態を持ち動作できるオブジェクトを作るための設計図です。型クラスはむしろインターフェース。型クラスからデータを作るのではなく、まずデータ型を作って、それが何のように振る舞えるかを考えます。等価性を扱えるなら Eq のインスタンスに、順序を持つなら Ord のインスタンスに、というように。

次のセクションでは、型クラスが定義する関数を実装して手動でインスタンスにする方法を見ます。今は、Haskell が自動で型を以下の型クラスのインスタンスにできることを見ます：Eq、Ord、Enum、Bounded、Show、Read。データ型を作るときに deriving キーワードを使えば、これらの型クラスにおける挙動を導出できます。

このデータ型を考えてください：

data Person = Person { firstName :: String
                     , lastName :: String
                     , age :: Int
                     }

人物を表します。同じ名・姓・年齢の人がいないと仮定すれば、2 人の人物レコードが同じ人物を表すか調べるのは意味があります。等価性のテストをしたい。それなら Eq 型クラスのメンバーにするのが理にかなっています。インスタンスを派生しましょう。

data Person = Person { firstName :: String
                     , lastName :: String
                     , age :: Int
                     } deriving (Eq)

型に対して Eq インスタンスを派生してから 2 つの値を == や /= で比較すると、Haskell はまず値コンストラクタが一致するか確認し（ここは1つしかありませんが）、次に各フィールドのペアを == でテストします。1 つだけ条件があります：すべてのフィールドの型も Eq 型クラスのメンバーである必要があります。String も Int もそうなので OK。Eq インスタンスを試します。

ghci> let mikeD = Person {firstName = "Michael", lastName = "Diamond", age = 43}
ghci> let adRock = Person {firstName = "Adam", lastName = "Horovitz", age = 41}
ghci> let mca = Person {firstName = "Adam", lastName = "Yauch", age = 44}
ghci> mca == adRock
False
ghci> mikeD == adRock
False
ghci> mikeD == mikeD
True
ghci> mikeD == Person {firstName = "Michael", lastName = "Diamond", age = 43}
True

Person が Eq に入ったので、elem のように Eq a 制約を持つ関数すべてに使えます。

ghci> let beastieBoys = [mca, adRock, mikeD]
ghci> mikeD `elem` beastieBoys
True

Show と Read 型クラスは、それぞれ文字列に変換できるもの・文字列から変換できるもののためのものです。Eq と同様、コンストラクタにフィールドがある場合、フィールドの型もそれぞれ Show や Read に属している必要があります。Person を Show と Read にも入れましょう。

data Person = Person { firstName :: String
                     , lastName :: String
                     , age :: Int
                     } deriving (Eq, Show, Read)

これで人物を端末に表示できます。

ghci> let mikeD = Person {firstName = "Michael", lastName = "Diamond", age = 43}
ghci> mikeD
Person {firstName = "Michael", lastName = "Diamond", age = 43}
ghci> "mikeD is: " ++ show mikeD
"mikeD is: Person {firstName = \"Michael\", lastName = \"Diamond\", age = 43}"

Show インスタンスにする前に人物を表示しようとすると、Haskell に「人物を文字列で表す方法を知らない」と怒られていたでしょう。今は知っています。

Read はほぼ Show の逆。Show は値→文字列、Read は文字列→値。read 関数を使うときは、結果としてどの型が欲しいかを Haskell に明示的な型注釈で伝える必要があります。

ghci> read "Person {firstName =\"Michael\", lastName =\"Diamond\", age = 43}" :: Person
Person {firstName = "Michael", lastName = "Diamond", age = 43}

read の結果を後で人物として使うなら、Haskell が推論できるので型注釈は不要です。

ghci> read "Person {firstName =\"Michael\", lastName =\"Diamond\", age = 43}" == mikeD
True

パラメータ化された型も読めますが、型引数を埋める必要があります。read "Just 't'" :: Maybe a はダメで、read "Just 't'" :: Maybe Char なら OK。

Ord 型クラスのインスタンスも派生できます。順序を持つ値の型のためのものです。同じ型の2つの値が異なるコンストラクタで作られている場合、先に定義されたコンストラクタの値のほうが小さいとみなされます。たとえば Bool 型は False または True の値を取ります。比較動作を見るために、こう実装されているとイメージできます：

data Bool = False | True deriving (Ord)

False 値コンストラクタが先、True が後に指定されているので、True は False より大きいとみなせます。

ghci> True `compare` False
GT
ghci> True > False
True
ghci> True < False
False

Maybe a データ型では、Nothing 値コンストラクタが Just より先に指定されているので、Nothing は常に Just something より小さいです。たとえ something がマイナス10億兆だったとしても。Just どうしを比較するときは中身の比較になります。

ghci> Nothing < Just 100
True
ghci> Nothing > Just (-49999)
False
ghci> Just 3 `compare` Just 2
GT
ghci> Just 100 > Just 50
True

ただし Just (*3) > Just (*2) はできません。(*3) や (*2) は関数で、Ord のインスタンスではないからです。

代数的データ型で列挙型を作りやすくするのに Enum・Bounded 型クラスが役立ちます。次のデータ型を考えます：

data Day = Monday | Tuesday | Wednesday | Thursday | Friday | Saturday | Sunday

すべての値コンストラクタが引数を取らない（フィールド0個）ので、Enum 型クラスに入れられます。Enum は前者・後者を持つもののためのもの。最大値・最小値を持つもののための Bounded も使えます。ついでにすべての派生可能な型クラスのインスタンスにもしておきましょう。

data Day = Monday | Tuesday | Wednesday | Thursday | Friday | Saturday | Sunday
           deriving (Eq, Ord, Show, Read, Bounded, Enum)

Show・Read のメンバーなので、文字列との相互変換が可能。

ghci> Wednesday
Wednesday
ghci> show Wednesday
"Wednesday"
ghci> read "Saturday" :: Day
Saturday

Eq・Ord のメンバーなので比較・等価が可能。

ghci> Saturday == Sunday
False
ghci> Saturday == Saturday
True
ghci> Saturday > Friday
True
ghci> Monday `compare` Wednesday
LT

Bounded なので最小・最大の日も取れます。

ghci> minBound :: Day
Monday
ghci> maxBound :: Day
Sunday

Enum でもあるので、前者・後者・レンジが使えます！

ghci> succ Monday
Tuesday
ghci> pred Saturday
Friday
ghci> [Thursday .. Sunday]
[Thursday,Friday,Saturday,Sunday]
ghci> [minBound .. maxBound] :: [Day]
[Monday,Tuesday,Wednesday,Thursday,Friday,Saturday,Sunday]

すごい。

型シノニム

前に [Char] と String は等価で交換可能だ、と話しました。これは 型シノニム によるものです。型シノニムは何か新しいことをするわけではなく、ただ既存の型に別名を付けて、コードやドキュメントを読む人にとって意味が通るようにする、という働きです。標準ライブラリでの String 定義： 

 type String = [Char]

新しいキーワード type を導入しました。type は紛らわしいかもしれません。data のように新しいものを作るわけではなく、既存の型のシノニム（別名）を作るだけだからです。

文字列を大文字化する関数を toUpperString として作ったとして、型宣言は toUpperString :: [Char] -> [Char] と書いても、toUpperString :: String -> String と書いても等価ですが、後者のほうが読みやすいです。

Data.Map モジュールを扱ったとき、電話帳をまず連想リストで表してからマップに変換しました。連想リストはキー値ペアのリストでしたね。あの電話帳： 

phoneBook :: [(String,String)]
phoneBook =
    [("betty","555-2938")
    ,("bonnie","452-2928")
    ,("patsy","493-2928")
    ,("lucille","205-2928")
    ,("wendy","939-8282")
    ,("penny","853-2492")
    ]

phoneBook の型は [(String,String)]。文字列から文字列への連想リストだと分かりますが、それ以上は分かりません。型宣言にもう少し情報を持たせるシノニムを作ります。

type PhoneBook = [(String,String)]

これで型宣言を phoneBook :: PhoneBook と書けます。String にもシノニムを：

type PhoneNumber = String
type Name = String
type PhoneBook = [(Name,PhoneNumber)]

関数内で文字列が何を表すかを伝えたいとき、Haskell プログラマは String にシノニムを付けます。

名前と番号を取って、その組み合わせが電話帳にあるかを調べる関数：

inPhoneBook :: Name -> PhoneNumber -> PhoneBook -> Bool
inPhoneBook name pnumber pbook = (name,pnumber) `elem` pbook

シノニムを使わなければ型は String -> String -> [(String,String)] -> Bool。シノニムを使うほうが分かりやすい。とはいえやりすぎは禁物です。型シノニムは「関数で使われる既存型が何を表すか」を説明したいとき、または「[(String,String)] のような長めの型が繰り返し出て、関数の文脈で何かより具体的なものを表すとき」に導入します。

型シノニムもパラメータ化できます。連想リストを表すけれどキー・値の型を任意にしたければ：

type AssocList k v = [(k,v)]

キーで値を引く関数の型は (Eq k) => k -> AssocList k v -> Maybe v のようになります。AssocList は2つの型を取って具体型（AssocList Int String など）を生む型コンストラクタです。

Fonzie 曰く： いいか！　具体型 っていうのは、Map Int String みたいに完全に適用された型のことだ。多相関数を扱ってるなら [a] や (Ord a) => Maybe a なんかもな。仲間内では Maybe を「型」と呼ぶこともあるが、それは違って、Maybe は型コンストラクタだってことくらい子どもでも知ってる。Maybe に追加の型を適用すると、たとえば Maybe String、はじめて具体型になる。値が持てる型は具体型だけだ！　というわけで、人生は早く生き、激しく愛し、誰にも自分のクシを使わせるな！

関数を部分適用して新しい関数を得るのと同じく、型引数を部分適用して新しい型コンストラクタを得られます。引数が足りない関数を呼んで関数を返すのと同じく、型引数を足りないまま指定して部分適用された型コンストラクタを得られます。Data.Map の整数→何かのマップを表したければ：

type IntMap v = Map Int v

あるいは：

type IntMap = Map Int

どちらの場合も、IntMap 型コンストラクタは引数を1つ取り、それは整数が指す値の型です。

そうそう。これを実装するなら Data.Map を qualified import するでしょう。qualified import すると型コンストラクタもモジュール名を前置する必要があります。type IntMap = Map.Map Int と書きます。

型コンストラクタと値コンストラクタの違いをよく理解してください。IntMap や AssocList といった型シノニムを作ったからといって、AssocList [(1,2),(4,5),(7,9)] のようなことができるわけではありません。型を別名で参照できるだけです。[(1,2),(3,5),(8,9)] :: AssocList Int Int はでき、中身の数は Int 型と仮定されますが、リストとしては普通の整数ペアのリストとして使えます。型シノニム（と一般に型）は Haskell の型部分でしか使えません。新しい型を定義するとき（data・type 宣言）や :: の後ろが型部分です。:: は型宣言や型注釈で使われます。

もう1つ、2 つの型を引数に取るデータ型 Either a b。おおよそこう定義されています：

data Either a b = Left a | Right b deriving (Eq, Ord, Read, Show)

2つの値コンストラクタを持ちます。Left なら中身は a 型、Right なら b 型。一方の型または他方の型の値を包めるので、Either a b の値を扱うときは Left と Right 両方にパターンマッチして、それぞれで違うことをするのが普通です。

ghci> Right 20
Right 20
ghci> Left "w00t"
Left "w00t"
ghci> :t Right 'a'
Right 'a' :: Either a Char
ghci> :t Left True
Left True :: Either Bool b

これまで Maybe a は計算が失敗したかどうかを表すのによく使ってきました。でも Nothing は「失敗した」以上の情報を持ちません。1 通りの失敗しかしない、または失敗の理由に興味がない関数ならそれで OK。Data.Map の lookup は「キーがマップになかった」という1通りの失敗しかしないので何が起きたか分かります。一方、失敗の理由に興味があるときは Either a b をよく使います。a が失敗の情報、b が成功の値の型。エラーは Left、結果は Right。

例：高校にロッカーがあり、生徒が Guns'n'Roses のポスターを貼る場所を確保するのに使います。各ロッカーには暗証番号があります。新しいロッカーが欲しい生徒は番号を伝え、監督者がコードを教えます。誰かが既に使っていれば、コードを教えられないので別のを選んでもらいます。Data.Map でロッカーを表します。ロッカー番号から「使用中かどうか」と「コード」のペアへの写像です。

import qualified Data.Map as Map

data LockerState = Taken | Free deriving (Show, Eq)

type Code = String

type LockerMap = Map.Map Int (LockerState, Code)

シンプル。ロッカーの状態を表す新しいデータ型と、コードのシノニム、整数→（状態,コード）のシノニムを作ります。コードを検索する関数を作ります。Either String Code 型で結果を表します。lookup は2 通りで失敗しうる — ロッカーが使用中、または番号が存在しない。失敗時は String で何が起きたかを伝えます。

lockerLookup :: Int -> LockerMap -> Either String Code
lockerLookup lockerNumber map =
    case Map.lookup lockerNumber map of
        Nothing -> Left $ "Locker number " ++ show lockerNumber ++ " doesn't exist!"
        Just (state, code) -> if state /= Taken
                                then Right code
                                else Left $ "Locker " ++ show lockerNumber ++ " is already taken!"

普通に lookup します。Nothing なら「存在しない」と Left String。見つかれば、使用中チェックを追加。使用中なら Left で「使用中」、空いていれば Right Code でコード（実体は Right String ですが、シノニムでドキュメント化）。サンプルマップ：

lockers :: LockerMap
lockers = Map.fromList
    [(100,(Taken,"ZD39I"))
    ,(101,(Free,"JAH3I"))
    ,(103,(Free,"IQSA9"))
    ,(105,(Free,"QOTSA"))
    ,(109,(Taken,"893JJ"))
    ,(110,(Taken,"99292"))
    ]

ロッカーコードを引いてみましょう。

ghci> lockerLookup 101 lockers
Right "JAH3I"
ghci> lockerLookup 100 lockers
Left "Locker 100 is already taken!"
ghci> lockerLookup 102 lockers
Left "Locker number 102 doesn't exist!"
ghci> lockerLookup 110 lockers
Left "Locker 110 is already taken!"
ghci> lockerLookup 105 lockers
Right "QOTSA"

再帰的データ構造

これまで見てきたように、代数的データ型のコンストラクタは複数（あるいはゼロ個）のフィールドを持ち、各フィールドの型は具体型でなければなりません。これを踏まえると、コンストラクタが同じ型のフィールドを持つような型も作れます！　これにより再帰的なデータ型が作れます。ある型の値がその型の値を含み、その値もまたその型の値を含み、という具合です。

このリストを考えてみてください：[5]。これは 5:[] の糖衣構文。: の左に値、右にリスト（ここでは空リスト）。[4,5] は 4:(5:[])。最初の : の左に要素、右にリスト（5:[]）。3:(4:(5:6:[])) も同様で、: が右結合なので 3:4:5:6:[] や [3,4,5,6] とも書けます。

つまり「リストとは空リスト、または要素を別のリスト（空または非空）と : でつないだもの」です。

代数的データ型で自分のリストを実装してみましょう！

data List a = Empty | Cons a (List a) deriving (Show, Read, Eq, Ord)

先ほどの定義と同じです。空リストか、ある値（先頭）と別のリストの組み合わせ。混乱したらレコード構文版のほうが分かりやすいかも。

data List a = Empty | Cons { listHead :: a, listTail :: List a} deriving (Show, Read, Eq, Ord)

Cons コンストラクタについて。cons は : の別名です。実はリストの : は値とリストを取ってリストを返すコンストラクタなのです。新しいリスト型はもう使えます！　フィールド2つ、1つは a、もう1つは [a] のような型。

ghci> Empty
Empty
ghci> 5 `Cons` Empty
Cons 5 Empty
ghci> 4 `Cons` (5 `Cons` Empty)
Cons 4 (Cons 5 Empty)
ghci> 3 `Cons` (4 `Cons` (5 `Cons` Empty))
Cons 3 (Cons 4 (Cons 5 Empty))

Cons を中置呼びして : っぽく見せました。Empty が []、4 `Cons` (5 `Cons` Empty) が 4:(5:[]) に対応。

関数を特殊文字だけで構成すれば自動的に中置にできます。コンストラクタも関数なので同じことができます。

infixr 5 :-:
data List a = Empty | a :-: (List a) deriving (Show, Read, Eq, Ord)

新しい構文「fixity 宣言」が登場しました。演算子として関数を定義するときに fixity を指定できます（必須ではない）。fixity は演算子の結合の強さと結合方向を表します。* の fixity は infixl 7 *、+ の fixity は infixl 6。両方とも左結合で（4 * 3 * 2 は (4 * 3) * 2）、* のほうが結合が強い（fixity が大きい）ので 5 * 4 + 3 は (5 * 4) + 3。

あとは Cons a (List a) を a :-: (List a) と書いただけ。リストはこう書けます：

ghci> 3 :-: 4 :-: 5 :-: Empty
(:-:) 3 ((:-:) 4 ((:-:) 5 Empty))
ghci> let a = 3 :-: 4 :-: 5 :-: Empty
ghci> 100 :-: a
(:-:) 100 ((:-:) 3 ((:-:) 4 ((:-:) 5 Empty)))

型に Show を派生させると、Haskell はコンストラクタを前置関数として表示するので、演算子に括弧が付きます（4 + 3 は (+) 4 3 と同じ）。

2つのリストを連結する関数を作ります。これが普通のリストの ++ 定義：

infixr 5  ++
(++) :: [a] -> [a] -> [a]
[]     ++ ys = ys
(x:xs) ++ ys = x : (xs ++ ys)

これを自分のリスト用に拝借します。.++ と命名。

infixr 5  .++
(.++) :: List a -> List a -> List a
Empty .++ ys = ys
(x :-: xs) .++ ys = x :-: (xs .++ ys)

動くか確認…

ghci> let a = 3 :-: 4 :-: 5 :-: Empty
ghci> let b = 6 :-: 7 :-: Empty
ghci> a .++ b
(:-:) 3 ((:-:) 4 ((:-:) 5 ((:-:) 6 ((:-:) 7 Empty))))

いい感じ。リスト用関数全部を自前リスト型でも実装できますが、ここまで。

(x :-: xs) でパターンマッチしているのに注目。パターンマッチは実はコンストラクタへのマッチなのです。:-: は自前のリスト型のコンストラクタなのでマッチでき、: は組み込みリスト型のコンストラクタなのでマッチでき、[] も同様。パターンマッチはコンストラクタに対してのみ働くので、こうした構造、普通の前置コンストラクタ、8 や 'a'（数値型・文字型のコンストラクタ）にもマッチできます。

次に 二分探索木 を実装します。C のような言語の二分探索木に馴染みがなければ、こうです：要素は左右2つの要素を指し、左は小さく、右は大きい。それらの要素もまた2つ（あるいは1つ、あるいは0個）の要素を指せる。要するに各要素は最大2つの部分木を持つ。例えば 5 の左部分木の要素はすべて 5 より小さい、右はすべて 5 より大きい、という性質があるので、たとえば 8 が木にあるか調べたいとき、5 から始めて 8 は 5 より大きいので右、7 から 8 は 7 より大きいので右、と3 ホップで見つかる。普通のリスト（あるいはアンバランスな木）なら 8 を見つけるのに 7 ホップかかるかも。

Data.Set・Data.Map の集合・マップは木で実装されていますが、普通の二分探索木ではなく、常にバランスされた平衡二分探索木です。今は普通の二分探索木を実装します。

これからこう言います：木は空木か、ある値と2つの木を含む要素のどちらか。代数的データ型にぴったりです！

data Tree a = EmptyTree | Node a (Tree a) (Tree a) deriving (Show, Read, Eq)

OK。手で木を作る代わりに、木と要素を取って要素を挿入する関数を作ります。挿入する値とルートノードを比較し、小さければ左へ、大きければ右へ。空木に到達するまで繰り返し、空木に到達したらその空木の代わりにその値を持つノードを挿入します。

C のような言語ではポインタと値を変更してこれをやります。Haskell では木を変更できないので、左右に進むたびに新しい部分木を作ります。最終的に挿入関数は完全に新しい木を返します。Haskell にポインタの概念はなく、値だけです。なので挿入関数の型は a -> Tree a - > Tree a。要素と木を取り、その要素を含む新しい木を返す。非効率に見えますが、遅延評価が解決してくれます。

2つの関数：シングルトン木（ノード1つの木）を作るユーティリティと、要素を木に挿入する関数。

singleton :: a -> Tree a
singleton x = Node x EmptyTree EmptyTree

treeInsert :: (Ord a) => a -> Tree a -> Tree a
treeInsert x EmptyTree = singleton x
treeInsert x (Node a left right)
    | x == a = Node x left right
    | x < a  = Node a (treeInsert x left) right
    | x > a  = Node a left (treeInsert x right)

singleton は中身と空部分木2つのノードを作るショートカット。挿入関数では、まずパターンとしてエッジ条件。空部分木に到達したらシングルトン木を入れる。空でない木への挿入なら、いくつかチェック：挿入する要素がルートと等しいなら同じ木を返す、小さいなら同じルート・同じ右部分木で左部分木に値を挿入した木、大きいならその逆。

次に、要素が木にあるか確認する関数。エッジ条件を定義しましょう。空木にはどんな要素もない。リストで要素を探す場合と同じです。空木でないなら、ルートが探している要素か確認、違えば左部分木はルートより小さいことを利用して、探している要素がルートより小さければ左、大きければ右を再帰的にチェック。

treeElem :: (Ord a) => a -> Tree a -> Bool
treeElem x EmptyTree = False
treeElem x (Node a left right)
    | x == a = True
    | x < a  = treeElem x left
    | x > a  = treeElem x right

前段落をコードに起こしただけ。木で遊んでみましょう！　手で構築する代わりに fold でリストから木を組み上げます。リストを1つずつ走査して何かを返すものはほぼすべて fold で実装できる、というのを覚えていれば！　空木から始め、リストを右からアプローチして、要素を順次アキュムレータの木に insert していくだけ。

ghci> let nums = [8,6,4,1,7,3,5]
ghci> let numsTree = foldr treeInsert EmptyTree nums
ghci> numsTree
Node 5 (Node 3 (Node 1 EmptyTree EmptyTree) (Node 4 EmptyTree EmptyTree)) (Node 7 (Node 6 EmptyTree EmptyTree) (Node 8 EmptyTree EmptyTree))

foldr で treeInsert が畳み込み関数（木とリスト要素を取って新しい木）、EmptyTree が初期アキュムレータ、nums が畳み込むリスト。

木をコンソールに表示してもあまり読みやすくありませんが、頑張れば構造が分かります。ルートが 5、その左部分木のルートが 3、右が 7、と。

ghci> 8 `treeElem` numsTree
True
ghci> 100 `treeElem` numsTree
False
ghci> 1 `treeElem` numsTree
True
ghci> 10 `treeElem` numsTree
False

メンバーシップチェックも問題なし。クール。

このように、代数的データ構造は Haskell の本当にクールでパワフルな概念です。真偽値や曜日の列挙から、二分探索木やもっといろんなものまで作れます！

型クラス応用

これまで標準 Haskell の型クラスのいくつかを学び、どの型がそこに属するかを見ました。標準型クラスのインスタンスを Haskell に自動派生してもらう方法も学びました。このセクションでは、自分で型クラスを作る方法、手作業で型をインスタンスにする方法を学びます。

型クラスのおさらい：型クラスはインターフェースのようなもの。何らかの振る舞い（等価性比較、順序比較、列挙など）を定義し、その振る舞いができる型がインスタンスになる。型クラスの振る舞いは関数または型宣言を定義し、それを実装することで実現される。型がある型クラスのインスタンスだ、と言うとき、その型でその型クラスの定義する関数が使える、ということです。

型クラスは Java や Python のクラスとはほとんど関係ありません。混乱する人が多いので、命令型言語のクラスについて知っていることはいったん忘れてください。

Eq 型クラスは等価性をテストできるもののためのもの。関数 ==・/= を定義する。Car 型があり 2台の車を == で比較するのが理にかなうなら、Car を Eq のインスタンスにするのは理にかなう。

標準 Prelude での Eq の定義：

class Eq a where
    (==) :: a -> a -> Bool
    (/=) :: a -> a -> Bool
    x == y = not (x /= y)
    x /= y = not (x == y)

うわ、新しい奇妙な構文！　すぐに明らかになるので心配無用。まず class Eq a where は新しい型クラス（名前は Eq）の定義です。a は型変数で、これからインスタンスにする型の役を担います。a でなくとも、1文字でなくとも、小文字の単語ならOK。続いて関数を定義。本体の実装は必須ではなく、関数の型宣言だけで構いません。

class Eq equatable where と書き、(==) :: equatable -> equatable -> Bool と型宣言する、と読めば理解しやすい人もいるかも。

とはいえ、ここでは関数本体を実装しています。相互再帰で。「Eq の2つのインスタンスは、異ならなければ等しい、等しくなければ異なる」と。必須ではありませんが、こうすると後で役立ちます。

class Eq a where でクラスを定義し、(==) :: a -> a -> Bool の型宣言があれば、その関数の型は (Eq a) => a -> a -> Bool になります。

クラスができたら何ができるでしょう？　あまりできることはありませんが、型をそのクラスのインスタンスにすると素敵な機能が手に入ります。この型を見てください：

data TrafficLight = Red | Yellow | Green

信号機の状態を定義します。クラスインスタンスを派生していないことに注目。Eq や Show なら派生できますが、ここでは手書きします。Eq インスタンスにする方法：

instance Eq TrafficLight where
    Red == Red = True
    Green == Green = True
    Yellow == Yellow = True
    _ == _ = False

instance キーワードを使いました。class は新しい型クラスの定義、instance は型をインスタンスにするためのもの。Eq 定義では class Eq a where と書き、a がインスタンスになる型の役を担うと言いました。インスタンス化のときは instance Eq TrafficLight where と書き、a が実際の型に置き換えられます。

== が /= で、/= が == で定義されていたので、インスタンス宣言ではどちらか一方だけ上書きすれば済みます。これを「最小完全定義」と呼びます — 型クラスのインターフェース通り振る舞うために実装すべき関数の最小集合。Eq の最小完全定義は == または /= のどちらか。もし Eq がこう定義されていたら：

class Eq a where
    (==) :: a -> a -> Bool
    (/=) :: a -> a -> Bool

両方の関数を実装する必要があります。Haskell は関係を知らないので。最小完全定義は == と /= 両方になります。

== はパターンマッチで実装しています。等しくないケースのほうがずっと多いので、等しいケースを並べて、それ以外は catch-all で False。

Show インスタンスも手作業にしましょう。最小完全定義は show 関数（値を取って文字列を返す）。

instance Show TrafficLight where
    show Red = "Red light"
    show Yellow = "Yellow light"
    show Green = "Green light"

これもパターンマッチで実現。動かしてみます：

ghci> Red == Red
True
ghci> Red == Yellow
False
ghci> Red `elem` [Red, Yellow, Green]
True
ghci> [Red, Yellow, Green]
[Red light,Yellow light,Green light]

いいですね。Eq は派生しても同じ効果（学習目的なのでしませんでしたが）。ただし Show を派生すると値コンストラクタを直接文字列にするだけ。"Red light" のように見せたいなら手書きが必要です。

他の型クラスのサブクラスとなる型クラスも作れます。Num の class 宣言の最初の部分：

class (Eq a) => Num a where
   ...

クラス制約を入れる場所はいろいろあります。これは class Num a where と書くのと似ていますが、a が Eq のインスタンスでなければならないと述べています。要するに、Num のインスタンスにする前に Eq のインスタンスにしなければなりません。数とみなせる前に等価性が判定できるべき、というのは納得です。サブクラス化はこれだけ — class 宣言にクラス制約があるだけ！　関数本体を class 宣言や instance 宣言で書くとき、a が Eq に属すと仮定でき、== を使えます。

ところで Maybe やリスト型はどうやってインスタンスにするのでしょう？　Maybe が TrafficLight と違うのは、Maybe 自体は具体型ではなく型コンストラクタである点です。型引数（Char など）を1つ取って具体型（Maybe Char）を生みます。Eq の定義をもう一度見ます：

class Eq a where
    (==) :: a -> a -> Bool
    (/=) :: a -> a -> Bool
    x == y = not (x /= y)
    x /= y = not (x == y)

型宣言から、a は具体型として使われていることが分かります（関数の型はすべて具体型でなければなりません。a -> Maybe はダメ、a -> Maybe a や Maybe Int -> Maybe String はOK）。だからこうは書けません：

instance Eq Maybe where
    ...

a は具体型である必要があるのに、Maybe は具体型ではない（型コンストラクタ）からです。instance Eq (Maybe Int) where、instance Eq (Maybe Char) where、と全型ぶん書くのも面倒。なのでこう：

instance Eq (Maybe m) where
    Just x == Just y = x == y
    Nothing == Nothing = True
    _ == _ = False

「Maybe something 型をすべて Eq のインスタンスにしたい」と言っています。(Maybe something) でも書けますが、Haskell スタイルでは1文字を選ぶのが普通。(Maybe m) が class Eq a where の a の役を担います。Maybe は具体型ではないですが、Maybe m は具体型。型引数 m を指定することで、「Maybe m 形式の型すべてを Eq のインスタンスにしたい」と言っています。

1 つだけ問題があります。気づきましたか？　Maybe の中身に == を使っていますが、中身が Eq で使えるかは保証されていません！　なので instance 宣言を変えます：

instance (Eq m) => Eq (Maybe m) where
    Just x == Just y = x == y
    Nothing == Nothing = True
    _ == _ = False

クラス制約を加える必要がありました！　この instance 宣言で「Maybe m 形式の型のうち、m（Maybe の中身）も Eq 型クラスに属するものだけを Eq のメンバーにする」と言えます。Haskell が派生するときも実はこうしています。

多くの場合、class 宣言のクラス制約は型クラスをサブクラスにするために、instance 宣言のクラス制約は中身に対する要件を表すために使います。

インスタンスを作るとき、型が型宣言で具体型として使われている（a -> a -> Bool の a など）なら、型引数を与えて括弧で囲み、具体型にする必要があります。

インスタンスにする型は class 宣言の引数を置き換えるという点に注意。class Eq a where の a は実際の型に置き換わるので、関数の型宣言にもその型を入れて考えると良いです。(==) :: Maybe -> Maybe -> Bool は意味をなしませんが、(==) :: (Eq m) => Maybe m -> Maybe m -> Bool は意味が通ります。とはいえ、これは思考の補助で、== は常に (==) :: (Eq a) => a -> a -> Bool 型です。

もう1つ、これを見て！　型クラスのインスタンスを見たければ、GHCi で :info YourTypeClass。:info Num なら型クラスが定義する関数とインスタンス型一覧が見えます。:info は型・型コンストラクタにも効きます。:info Maybe は Maybe がインスタンスである型クラス全部を見せてくれます。:info は関数の型宣言も見せられます。クールです。

YesNo型クラス

JavaScript やいくつかの弱い型付け言語では、ほぼ何でも if 式に入れられます。if (0) alert("YEAH!") else alert("NO!")、if ("") alert ("YEAH!") else alert("NO!")、if (false) alert("YEAH") else alert("NO!)、これら全部 NO! のアラート。if ("WHAT") alert ("YEAH") else alert("NO!") なら "YEAH!"。JavaScript では空でない文字列を真っぽいとみなすからです。

Haskell では真偽値に Bool を厳格に使うほうが良いですが、JavaScript っぽい振る舞いをあえて実装してみます。お遊びです！　class 宣言から：

class YesNo a where
    yesno :: a -> Bool

シンプル。YesNo は関数を1つ定義。「真っぽさ」を持つ型の値を取って、真かどうかを返す。a の使い方から、a は具体型である必要があります。

インスタンスを定義します。数値は（JavaScript のように）0 以外は真っぽい、0 は偽っぽい。

instance YesNo Int where
    yesno 0 = False
    yesno _ = True

空リスト（および文字列）は偽っぽい、非空は真っぽい。

instance YesNo [a] where
    yesno [] = False
    yesno _ = True

型引数 a を入れてリストを具体型にしているのに注目。中身の型は仮定しません。Bool は真偽そのものなので明らか：

instance YesNo Bool where
    yesno = id

あれ？　id って何？　標準ライブラリの関数で、引数をそのまま返すだけ。ここで書きたかったまさにその関数です。

Maybe a もインスタンスに：

instance YesNo (Maybe a) where
    yesno (Just _) = True
    yesno Nothing = False

Maybe の中身に何も仮定していないのでクラス制約は不要。Just なら真っぽく、Nothing なら偽っぽい、と。Maybe ではなく (Maybe a) と書く必要があります。Maybe -> Bool という関数は存在し得ない（具体型でないので）が、Maybe a -> Bool は OK。これで Maybe something 形式の型はすべて YesNo に入りました。

前に定義した二分探索木 Tree a。空木は偽っぽく、非空は真っぽい：

instance YesNo (Tree a) where
    yesno EmptyTree = False
    yesno _ = True

信号機は yes/no の値になれるか？　なれます。赤なら止まる、緑なら進む、黄色は？　まあ、私はアドレナリンのために走っちゃいます。

instance YesNo TrafficLight where
    yesno Red = False
    yesno _ = True

クール、インスタンスができたので遊んでみましょう！

ghci> yesno $ length []
False
ghci> yesno "haha"
True
ghci> yesno ""
False
ghci> yesno $ Just 0
True
ghci> yesno True
True
ghci> yesno EmptyTree
False
ghci> yesno []
False
ghci> yesno [0,0,0]
True
ghci> :t yesno
yesno :: (YesNo a) => a -> Bool

動きます！　YesNo 値で if 文を真似する関数を作ります。

yesnoIf :: (YesNo y) => y -> a -> a -> a
yesnoIf yesnoVal yesResult noResult = if yesno yesnoVal then yesResult else noResult

シンプル。yes/no っぽい値と2つの値を取り、yes っぽいなら最初の値、そうでなければ2 つ目を返す。

ghci> yesnoIf [] "YEAH!" "NO!"
"NO!"
ghci> yesnoIf [2,3,4] "YEAH!" "NO!"
"YEAH!"
ghci> yesnoIf True "YEAH!" "NO!"
"YEAH!"
ghci> yesnoIf (Just 500) "YEAH!" "NO!"
"YEAH!"
ghci> yesnoIf Nothing "YEAH!" "NO!"
"NO!"

Functor型クラス

これまで標準ライブラリの型クラスをたくさん見てきました。Ord（順序）、Eq（等価性）、Show（文字列化）、Read（文字列から値）。今度は Functor 型クラスを見ます。基本的に「map（写像）できるもの」のためのものです。リストを思い浮かべるかもしれませんね。実際、リストは Functor 型クラスの一員です。

Functor を理解する一番の方法は実装を見ること。覗いてみましょう。

class Functor f where
    fmap :: (a -> b) -> f a -> f b

OK。関数 fmap を1つ定義し、デフォルト実装はなし。fmap の型が興味深い。これまでの型クラス定義では、型変数は具体型を表していました（(==) :: (Eq a) => a -> a -> Bool の a など）。でも今、f は具体型ではなく、型引数を1つ取る型コンストラクタです。クイック復習：Maybe Int は具体型、Maybe は型を引数に取る型コンストラクタ。fmap は「ある型から別の型への関数」と「ある型を適用したファンクタ」を取って、「別の型を適用したファンクタ」を返す、という関数。

混乱しても大丈夫、例を見ればすぐ分かります。fmap の型宣言、何かを思い出させますね。map の型シグネチャを知らないなら：map :: (a -> b) -> [a] -> [b]。

あ、興味深い！　ある型から別の型への関数と、ある型のリストを取って別の型のリストを返す。これはファンクタですね！　実際 map はリストにのみ動く fmap です。リストを Functor のインスタンスにする方法：

instance Functor [] where
    fmap = map

これだけ！　instance Functor [a] where ではなく [] と書いていることに注目。fmap :: (a -> b) -> f a -> f b から、f は引数1つを取る型コンストラクタである必要があります。[a] はもう具体型（中身が任意の型のリスト）ですが、[] は型を1つ取って [Int]、[String]、[[String]] などを生む型コンストラクタです。

リストでは fmap はただの map なので結果は同じ：

map :: (a -> b) -> [a] -> [b]
ghci> fmap (*2) [1..3]
[2,4,6]
ghci> map (*2) [1..3]
[2,4,6]

空リストに map や fmap をすると、空リストが返ります。[a] 型の空リストを [b] 型の空リストに変えるだけ。

箱のように振る舞う型はファンクタになれます。リストは無限のコンパートメントを持つ箱で、空のもあれば中身が入っているのもある、と考えられます。他に箱っぽい性質を持つものは？　Maybe a 型。何も入っていないか（Nothing）、1つだけ入っているか（Just "HAHA"）。Maybe がファンクタである方法：

instance Functor Maybe where
    fmap f (Just x) = Just (f x)
    fmap f Nothing = Nothing

また、instance Functor (Maybe m) where ではなく instance Functor Maybe where と書いていることに注目。Functor は具体型ではなく1引数を取る型コンストラクタを欲しています。f を Maybe に置き換えると、fmap はこの型では (a -> b) -> Maybe a -> Maybe b のように振る舞い、OK そう。f を (Maybe m) に置き換えると、(a -> b) -> Maybe m a -> Maybe m b となって意味不明（Maybe は型引数を1つしか取らない）。

さて、fmap 実装はシンプル。Nothing なら Nothing を返す。空の箱を map したら空の箱、と。空リストを map したら空リスト、と同じ。Just に1つ詰められた値なら、関数を中身に適用。

ghci> fmap (++ " HEY GUYS IM INSIDE THE JUST") (Just "Something serious.")
Just "Something serious. HEY GUYS IM INSIDE THE JUST"
ghci> fmap (++ " HEY GUYS IM INSIDE THE JUST") Nothing
Nothing
ghci> fmap (*2) (Just 200)
Just 400
ghci> fmap (*2) Nothing
Nothing

もう1つ map 可能で Functor インスタンスにできるのが Tree a。複数（または0）の値を入れる箱とも考えられ、Tree 型コンストラクタは型引数をぴったり1つ取ります。fmap が Tree 専用なら、型シグネチャは (a -> b) -> Tree a -> Tree b。再帰を使います。空木の map は空木。非空木の map は、ルート値に関数を適用した木と、左右部分木それぞれに関数を map した木。

instance Functor Tree where
    fmap f EmptyTree = EmptyTree
    fmap f (Node x leftsub rightsub) = Node (f x) (fmap f leftsub) (fmap f rightsub)

ghci> fmap (*2) EmptyTree
EmptyTree
ghci> fmap (*4) (foldr treeInsert EmptyTree [5,7,3,2,1,7])
Node 28 (Node 4 EmptyTree (Node 8 EmptyTree (Node 12 EmptyTree (Node 20 EmptyTree EmptyTree)))) EmptyTree

素敵！　では Either a b はどうでしょう？　Functor は引数1つの型コンストラクタが欲しいけど、Either は2つ取る。うーむ！　よし、Either を1つだけ与えて部分適用すれば、空き引数が1つになる。標準ライブラリでの Either a がファンクタである方法：

instance Functor (Either a) where
    fmap f (Right x) = Right (f x)
    fmap f (Left x) = Left x

何をしたか。Either ではなく Either a をインスタンスにしました。Either a が引数1つの型コンストラクタで、Either は2つだから。fmap が Either a 専用なら型は (b -> c) -> Either a b -> Either a c（(b -> c) -> (Either a) b -> (Either a) c と同じ）。実装では Right のときは map し、Left はしません。なぜ？　Either a b の定義を見れば：

data Either a b = Left a | Right b

両方を1つの関数で map するなら、a と b は同じ型である必要があります。文字列を取って文字列を返す関数を map し、b が文字列で a が数なら、うまくいきません。Either の fmap 型から見ても、最初のパラメータは変わらず、2 つ目が変わる必要があり、最初のパラメータは Left 値コンストラクタで実体化されます。

箱のアナロジーとも合います。Left 部分は「中が空でエラーメッセージが書かれた箱」のようなもの。

Data.Map のマップもファンクタになれます（値を保持する／しない）。Map k v の場合、fmap は関数 v -> v' を Map k v に map して Map k v' を返します。

注意、' は値の名前と同じく、型でも特別な意味は持ちません。「似ているけど少し違うもの」を表すのに使われます。

Map k がどう Functor インスタンスになるか、自分で考えてみてください！

Functor 型クラスで、型クラスがクールな高階概念を表現できることを見ました。型の部分適用やインスタンス作成にも慣れてきたはず。次の章でファンクタが従う法則も見ます。

もう1つ！　ファンクタは法則に従うべきです。そうすれば私たちは性質に頼って深く考えずに済みます。[1,2,3,4] に fmap (+1) したら [2,3,4,5] を期待します、[5,4,3,2] ではなく。fmap (\a -> a)（恒等関数）をリストに使ったら、同じリストが返るはず。Tree に間違ったファンクタインスタンスを与えると、左部分木がノードより小さく右がより大きいという性質を保たない木になるかもしれません。次章でファンクタ則を詳しく扱います。

カインドと型の話

型コンストラクタは型を引数に取って具体型を生みます。値を引数に取って値を生む関数とちょっと似ていますね。型コンストラクタが部分適用できることも見ました（Either String は1つの型を取って Either String Int のような具体型を生む型）。型がどう型コンストラクタに適用されるかを、関数の引数を型宣言で形式化したのと同じように、形式化してみます。このセクションを読まなくても Haskell の旅は続けられますし、分からなくても気にしなくて大丈夫。ですが、これが分かれば型システムを深く理解できます。

値（3、"YEAH"、takeWhile など。関数も値です）はそれぞれ型を持ちます。型は値に付けられたラベルのようなもの。でも型自身にもラベルがあり、それを カインド (kind) と呼びます。カインドは大まかに「型の型」。少し奇妙に聞こえますが、本当にクールな概念です。

カインドって何でどう役立つか？　GHCi の :k コマンドで型のカインドを調べてみましょう。

ghci> :k Int
Int :: *

星？　なに？　* は具体型（型引数を取らない型、値が持てる型）の意味。声に出して読むなら「スター」または「タイプ」。

Maybe のカインドは：

ghci> :k Maybe
Maybe :: * -> *

Maybe 型コンストラクタは具体型1つ（Int など）を取り、具体型（Maybe Int）を返す。Int -> Int が「Int を取って Int を返す関数」を意味するように、* -> * は「具体型を1つ取って具体型を返す型コンストラクタ」を意味します。Maybe に型引数を適用すると：

ghci> :k Maybe Int
Maybe Int :: *

予想通り！　Maybe に型引数を適用して具体型に。isUpper が Char -> Bool 型で、isUpper 'A' が Bool 型（値は True）で、両方ともカインドが *、というのと並行（型とカインドは違うものですが）です。

型のカインドは :k、値の型は :t。型は値のラベル、カインドは型のラベル、両者には並行関係があります。

別のカインドを：

ghci> :k Either
Either :: * -> * -> *

Either は具体型2つを取って具体型を生む。2 引数の関数の型宣言にも見えます。型コンストラクタもカリー化されているので部分適用できます。

ghci> :k Either String
Either String :: * -> *
ghci> :k Either String Int
Either String Int :: *

Either を Functor に入れたいとき、部分適用が必要だったのは、Functor が引数1つの型を欲しがり、Either は2つ取るから。要するに Functor はカインド * -> * の型を欲しがるので、Either を部分適用してカインドを * -> * -> * から * -> * に変えました。Functor 定義をもう一度：

class Functor f where
    fmap :: (a -> b) -> f a -> f b

型変数 f は「具体型を1つ取って具体型を生む型」として使われています（関数の値の型として使われているので具体型を生む必要がある）。これから Functor と仲良くしたい型はカインド * -> * である必要があると分かります。

では型 foo を少し。次の型クラスを今でっち上げます：

class Tofu t where
    tofu :: j a -> t a j

奇妙です。この型クラスのインスタンスになる型を作るには？　カインドを考えます。j a が tofu の引数の値の型として使われているので、j a はカインド *。a も * と仮定すると、j は * -> *。t も具体値を生み、2 つの型を取るので、a が * で j が * -> * なら、t は * -> (* -> *) -> *。具体型 a、引数1つの型コンストラクタ j を取って具体型を生む。

カインド * -> (* -> *) -> * の型を作りましょう。

data Frank a b  = Frank {frankField :: b a} deriving (Show)

これがカインド * -> (* -> *) - > * なのはなぜ？　ADT のフィールドは値を持つので、明らかにカインド *。a を * と仮定すると b は型引数1つを取るので * -> *。だから Frank はカインド * -> (* -> *) -> *。最初の * が a、(* -> *) が b。Frank 値を作って型を確認：

ghci> :t Frank {frankField = Just "HAHA"}
Frank {frankField = Just "HAHA"} :: Frank [Char] Maybe
ghci> :t Frank {frankField = Node 'a' EmptyTree EmptyTree}
Frank {frankField = Node 'a' EmptyTree EmptyTree} :: Frank Char Tree
ghci> :t Frank {frankField = "YES"}
Frank {frankField = "YES"} :: Frank Char []

うむ。frankField は a b 型なので、その値の型も同じ形である必要がある。Just "HAHA" は Maybe [Char]、['Y','E','S'] は [Char] など。Frank 値の型は Frank のカインドに対応。値を持つには具体型である必要があるので、Frank blah blaah はカインド *。

Frank を Tofu インスタンスにするのは簡単。tofu は j a（例：Maybe Int）を取って t a j を返す。Frank を j に置き換えると結果型は Frank Int Maybe。

instance Tofu Frank where
    tofu x = Frank x

ghci> tofu (Just 'a') :: Frank Char Maybe
Frank {frankField = Just 'a'}
ghci> tofu ["HELLO"] :: Frank [Char] []
Frank {frankField = ["HELLO"]}

あまり実用的ではありませんが、型筋肉を伸ばしました。もう少し型 foo を。次のデータ型：

data Barry t k p = Barry { yabba :: p, dabba :: t k }

これを Functor のインスタンスにしたい。Functor はカインド * -> * の型を欲しがりますが、Barry はそうじゃなさそう。カインドは？　3 つの型引数を取るので 何か -> 何か -> 何か -> *。p は具体型（カインド *）と置く。k も * と仮定すると t は * -> *。これらを「何か」に当てはめると、カインド (* -> *) -> * -> * -> *。GHCi で確認：

ghci> :k Barry
Barry :: (* -> *) -> * -> * -> *

正解！　Functor に入れるには最初の2つの型引数を部分適用して * -> * にする必要があります。インスタンス宣言の冒頭は instance Functor (Barry a b) where。fmap が Barry 専用なら型は fmap :: (a -> b) -> Barry c d a -> Barry c d b。Functor の f を Barry c d に置き換えただけ。Barry の3つ目の型引数が変わる必要があり、ちょうどそれは独自のフィールドになっています。

instance Functor (Barry a b) where
    fmap f (Barry {yabba = x, dabba = y}) = Barry {yabba = f x, dabba = y}

はい！　f を最初のフィールドに map しただけ。

このセクションでは、型引数の動きとカインドによる形式化を見ました。関数引数を型宣言で形式化するのと並行関係。関数と型コンストラクタの並行関係も見ました。両者は別物ですが面白い類似があります。実際の Haskell ではカインド推論を手でやることはあまりないでしょう。普通は、自前の型を * -> * や * に部分適用して標準型クラスのインスタンスにするだけ。でも仕組みを知っておくと良いです。型に小さな型があるって面白いですよね。
