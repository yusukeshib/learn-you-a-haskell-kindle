再帰

こんにちは再帰！

前章で再帰について少し触れました。この章では再帰を詳しく見ていき、なぜ Haskell にとって重要なのか、再帰的に考えることでどのように簡潔でエレガントな解にたどり着けるのかを見ていきます。

再帰がまだ何か分からないなら、この文を読んでください。はは！　冗談です。再帰とは、関数の定義の中でその関数自身を適用する形で関数を定義する手法のことです。数学の定義は再帰的に与えられることがよくあります。たとえばフィボナッチ数列は再帰的に定義されます。まず、最初の2つのフィボナッチ数を非再帰的に定義します。F(0) = 0 および F(1) = 1、つまり 0 番目と 1 番目のフィボナッチ数はそれぞれ 0 と 1 です。次に、それ以外の自然数については、フィボナッチ数を「直前2つのフィボナッチ数の和」と決めます。すなわち F(n) = F(n-1) + F(n-2)。こうすると F(3) は F(2) + F(1)、これは (F(1) + F(0)) + F(1) となります。ここまで来れば、もはや非再帰的に定義されたフィボナッチ数だけになっているので、安心して F(3) は 2 だと言えます。再帰定義の中で、ここでの F(0) と F(1) のように非再帰的に定義された要素を1つ2つ持っておくことを エッジ条件 と呼びます。再帰関数を停止させたいなら、これは大切です。F(0) と F(1) を非再帰的に定義していなければ、いつまで経っても答えにたどり着けません。0 を超えると負の数に突入し、いつしか「F(-2000) は F(-2001) + F(-2002)」と言い続けることになって、終わりが見えなくなってしまいます！

再帰は Haskell にとって重要です。命令型言語と違って Haskell では、計算は「どうやって 何かを得るか」を宣言するのではなく、「何かが 何で あるか」を宣言する形で行うからです。だから Haskell には while ループや for ループがなく、代わりに再帰を使って物事の正体を宣言することが多いのです。

最大限にすごい

maximum 関数は、順序のあるもの（つまり Ord 型クラスのインスタンス）のリストを取り、その中で一番大きい要素を返します。これを命令型のスタイルで実装することを考えてみてください。たぶん、これまでの最大値を保持する変数を用意し、リストの要素をループで回り、もし要素が現在の最大値より大きければ最大値を更新する。最後に残った最大値が結果。ふー！　こんなに簡単なアルゴリズムを説明するのに言葉を費やしてしまった！

では再帰的に定義してみましょう。まずエッジ条件として、要素1つのリスト（シングルトンリスト）の最大値はその唯一の要素そのもの、と決めます。次に、長いリストの最大値は「先頭が残りの最大値より大きければ先頭、そうでなければ残りの最大値」だと決めます。それだけ！　Haskell で実装してみます。

maximum' :: (Ord a) => [a] -> a
maximum' [] = error "maximum of empty list"
maximum' [x] = x
maximum' (x:xs)
    | x > maxTail = x
    | otherwise = maxTail
    where maxTail = maximum' xs

見ての通り、パターンマッチは再帰と相性抜群！　多くの命令型言語にはパターンマッチがないので、エッジ条件をテストするのに if-else 文を山ほど書く必要があります。Haskell ではパターンとして並べるだけ。最初のエッジ条件は「リストが空ならクラッシュ」。空リストの最大値って何でしょう？　知りません。2つ目のパターンも別のエッジ条件で、「シングルトンリストならその唯一の要素を返す」というものです。

3つ目のパターンが本番です。パターンマッチでリストを先頭と残りに分けます。リストでの再帰でとてもよく使われるイディオムなので、慣れておきましょう。where 束縛で maxTail を「残りのリストの最大値」と定義します。そして先頭が残りの最大値より大きいかをチェックし、大きければ先頭を、そうでなければ残りの最大値を返します。

例として数値のリスト [2,5,1] を取り、これに対する動きを見てみます。maximum' を呼ぶと、最初の2つのパターンには合致せず、3つ目のパターンに合致してリストは 2 と [5,1] に分けられます。where 節は [5,1] の最大値を求めようとするので、その経路を辿ります。再び3つ目のパターンに合致して [5,1] は 5 と [1] に分けられます。今度は where 節が [1] の最大値を求めようとし、これはエッジ条件に合致するので 1 を返します。やっと！　1 段戻って、5 と [1] の最大値（1）を比較し、結果は当然 5。これで [5,1] の最大値は 5 と分かりました。さらに 1 段戻って 2 と [5,1] の比較。[5,1] の最大値 5 と 2 を比べて 5 を選びます。

もっと分かりやすい書き方として、max を使う方法があります。max は2つの数を取って大きいほうを返す関数でしたね。max を使って maximum' を書き直すとこうなります：

maximum' :: (Ord a) => [a] -> a
maximum' [] = error "maximum of empty list"
maximum' [x] = x
maximum' (x:xs) = max x (maximum' xs)

なんてエレガント！　要するに、リストの最大値とは「先頭の要素」と「残りの最大値」の max なのです。

もう少し再帰関数

再帰的な考え方の一般像が掴めたところで、いくつか関数を再帰で実装してみましょう。まずは replicate の実装。replicate は Int と要素を取り、その要素を何個か繰り返したリストを返します。たとえば replicate 3 5 は [5,5,5] を返します。エッジ条件を考えてみましょう。私の予想は「0 以下」。何かを 0 回複製しようとしたら、空リストを返すべきです。負の数も同様で、そもそも意味をなしません。

replicate' :: (Num i, Ord i) => i -> a -> [a]
replicate' n x
    | n <= 0    = []
    | otherwise = x:replicate' (n-1) x

真偽条件をテストするので、ここではパターンではなくガードを使いました。n が 0 以下なら空リストを返す。そうでなければ x を先頭にし、その後ろに x を n-1 回繰り返したリストを尾としたリストを返す。(n-1) の部分が、いずれエッジ条件に到達させてくれます。

Note: Num は Ord のサブクラスではありません。つまり数を構成するものが必ずしも順序を持つとは限りません。だから加減算と比較の両方をするときは、Num と Ord 両方のクラス制約を指定する必要があります。

次に take を実装します。リストから決まった数の要素を取ります。たとえば take 3 [5,4,3,2,1] は [5,4,3] を返します。0 以下の数を取ろうとすれば空リスト、空リストから何かを取ろうとしても空リスト。エッジ条件が2つあります。書き起こすと：

take' :: (Num i, Ord i) => i -> [a] -> [a]
take' n _
    | n <= 0   = []
take' _ []     = []
take' n (x:xs) = x : take' (n-1) xs

最初のパターンは「0 以下の個数を取ろうとすれば空リスト」を表します。リストの中身は気にしないので _ でマッチさせています。ガードを使っていますが otherwise が無いことに注目。これは「n が 0 を超える場合はマッチが次のパターンに落ちる」ことを意味します。2つ目のパターンは「空リストから何かを取ろうとすれば空リスト」。3つ目のパターンはリストを先頭と残りに分けます。そして「リストから n 個取る」とは「x を先頭にして、残りから n-1 個取ったリストを尾にしたリスト」だと宣言しています。たとえば [4,3,2,1] から 3 つ取るとき、評価がどう進むか紙に書いてみてください。

reverse はリストを反転するだけ。エッジ条件は何でしょう？　ええっと……空リストです！　空リストを反転したら空リストそのもの。OK。残りはどうしましょう？　リストを先頭と残りに分ければ、反転リストは「残りの反転 ++ 先頭」と書けます。

reverse' :: [a] -> [a]
reverse' [] = []
reverse' (x:xs) = reverse' xs ++ [x]

はい、できあがり！

Haskell は無限リストをサポートしているので、再帰には必ずしもエッジ条件が要りません。ただしエッジ条件がないと、永遠に何かを擦り続けるか、無限リストのような無限のデータ構造を生成し続けることになります。とはいえ無限リストの良いところは、好きな場所で切り取れることです。repeat は要素を1つ取り、その要素だけからなる無限リストを返します。再帰的な実装はとっても簡単。

repeat' :: a -> [a]
repeat' x = x:repeat' x

repeat 3 を呼ぶと、3 から始まり 3 が無限に並ぶリストを得ます。repeat 3 は 3:repeat 3、それは 3:(3:repeat 3)、さらに 3:(3:(3:repeat 3))……と評価されていきます。repeat 3 の評価は決して終わりませんが、take 5 (repeat 3) なら 3 が 5 つ並んだリストを返します。要するに replicate 5 3 と同じです。

zip は2つのリストを取って zip します。zip [1,2,3] [2,3] は [(1,2),(2,3)] を返します。長いほうのリストが短いほうの長さに切り詰められるからです。空リストと何かを zip したら？　空リストが返ります。これがエッジ条件。zip はリストを2つ取るので、エッジ条件も2つあります。

zip' :: [a] -> [b] -> [(a,b)]
zip' _ [] = []
zip' [] _ = []
zip' (x:xs) (y:ys) = (x,y):zip' xs ys

最初の2つのパターンは「最初か2つ目のリストが空なら結果も空」。3つ目は「2つのリストを zip するとは、両方の先頭をペアにして、残り同士を zip した結果に前置する」。[1,2,3] と ['a','b'] を zip すると、最終的に [3] と [] を zip しようとします。エッジ条件のパターンが効いて、結果は (1,'a'):(2,'b'):[] つまり [(1,'a'),(2,'b')] となります。

もう1つ標準ライブラリの関数を実装します — elem。要素とリストを取って、その要素がリストに含まれるかを返します。リストを扱うときの定番、エッジ条件は空リスト。空リストには要素がないので、探しているドロイドはここにはいない、ですね。

elem' :: (Eq a) => a -> [a] -> Bool
elem' a [] = False
elem' a (x:xs)
    | a == x    = True
    | otherwise = a `elem'` xs

シンプルで予想通り。先頭が探している要素でなければ、残りで再帰的にチェックします。空リストに到達すれば結果は False。

急げ、ソートだ！

ソートできる項目のリストがあるとします。型は Ord 型クラスのインスタンスです。さあ、これをソートしましょう！　ソートにはクイックソートというとてもクールなアルゴリズムがあります。命令型言語では実装に 10 行以上かかりますが、Haskell ならずっと短くてエレガントです。クイックソートは Haskell の看板みたいになっています。だからここでも実装してみます。Haskell のエレガントさを誇示するためにみんながやっているので、もはや陳腐ですが、それでも。

型シグネチャは quicksort :: (Ord a) => [a] -> [a]。意外性なし。エッジ条件は？　空リスト、お決まりです。空リストをソートしたら空リスト。さて主アルゴリズム：ソート済みリストとは「リストの先頭以下のすべての値（これらもソート済み）を前に置き、真ん中に先頭、そして先頭より大きいすべての値（これらもソート済み）を後ろに置いたリスト」である。　定義の中で ソート済み が2回出てくるので、再帰呼び出しを2回することになりそうです！　また、「これをやって、あれをやって、それからあれをやって……」ではなく である という動詞でアルゴリズムを定義していることに注目してください。これが関数型プログラミングの美しさ！　リストの先頭より小さい要素と大きい要素にどう分けるかというと、リスト内包表記を使います。さあこの関数を定義してみましょう。

quicksort :: (Ord a) => [a] -> [a]
quicksort [] = []
quicksort (x:xs) =
    let smallerSorted = quicksort [a | a <- xs, a <= x]
        biggerSorted = quicksort [a | a <- xs, a > x]
    in  smallerSorted ++ [x] ++ biggerSorted

ちゃんと動くか軽く試してみましょう。

ghci> quicksort [10,2,5,3,1,6,7,4,2,3,4,8,9]
[1,2,2,3,3,4,4,5,6,7,8,9,10]
ghci> quicksort "the quick brown fox jumps over the lazy dog"
"        abcdeeefghhijklmnoooopqrrsttuuvwxyz"

これだよこれ！　たとえば [5,1,9,4,6,7,3] をソートしたい場合、このアルゴリズムはまず先頭 5 を取り、それより小さいリストと大きいリストの間に置きます。一時点で [1,4,3] ++ [5] ++ [9,6,7] となります。完全にソートされたとき、5 は 4 番目の位置に居続けることが分かります（5 より小さい数が 3 つ、大きい数が 3 つ）。あとは [1,4,3] と [9,6,7] をソートすれば、ソート済みリストの完成！　この2つのリストも同じ関数でソートします。最終的に分割が進むと空リストにたどり着き、空リストはそもそも空であるという理由でソート済みです。図示するとこんな感じ：

すでに正しい位置にあって動かない要素は オレンジ で表しています。左から右に読めば、ソート済みリストになります。比較対象として先頭を選びましたが、実はどの要素を選んでも構いません。クイックソートで比較対象に選ぶ要素を「ピボット」と呼びます。図では 緑 で示しています。先頭を選んだのはパターンマッチで取り出しやすいからです。ピボットより小さい要素は 明るい緑、ピボットより大きい要素は 濃い緑 です。黄色っぽいグラデーションは quicksort の適用を表しています。

再帰的に考える

これまでかなり再帰をやりました。気づいた人もいると思いますが、ここにはパターンがあります。普通はまずエッジケースを定義し、次に「ある要素」と「残りに対する関数の適用結果」の間で何かを行う関数を定義する、という形です。それがリストでも、木でも、その他のデータ構造でも変わりません。リストの合計とは「先頭 + 残りの合計」、リストの積とは「先頭 × 残りの積」、リストの長さとは「1 + 残りの長さ」。等々、等々……

もちろんエッジケースもあります。普通エッジケースは、再帰的な適用が意味をなさないシナリオです。リストを扱うときのエッジケースは大抵空リスト。木を扱うときは、子を持たないノードがエッジケースになることが多いです。

数を再帰的に扱うときも似ています。たいてい、ある数と、その数を変形した数に対する関数の適用が関わります。先ほど書いた階乗関数は「ある数 × その数 - 1 の階乗」でした。0 ではこの再帰的適用が意味をなしません。階乗は正の整数だけに定義されているからです。多くの場合、エッジケースの値は単位元 (identity) になります。乗算の単位元は 1 です。何かに 1 をかけても元のままだからです。リストの合計でも、空リストの合計を 0 と定義しました。0 は加算の単位元です。クイックソートでは、エッジケースは空リストで、これも単位元です。空リストをリストに連結しても元のリストが返るだけだからです。

だから問題を再帰的に解こうとするときは、再帰的な解が当てはまらない場面を考えてエッジケースに使えるか試し、単位元を考え、関数の引数を分解できるか（リストならパターンマッチで先頭と残りに分けるのが普通です）考え、どの部分で再帰呼び出しを使うかを考えてみましょう。
