型と型クラス

型を信じろ

前章で、Haskellは静的型付けシステムを持つと述べました。すべての式の型はコンパイル時に分かるため、より安全なコードになります。真偽値を何かの数値で割ろうとするようなプログラムを書いたら、そもそもコンパイルすら通りません。これは良いことです。プログラムが実行時にクラッシュするより、こうしたエラーをコンパイル時に捕まえられるほうが断然マシだからです。Haskellではあらゆるものが型を持つので、コンパイラはコンパイル前にプログラムについてかなりの推論ができます。

JavaやPascalと違って、Haskellには型推論があります。数値を書いたとき、それが数値であることをHaskellに教える必要はありません。Haskellが自分で 推論 (infer) してくれるので、関数や式の型を明示的に書かなくても作業を進められます。前章まで、私たちは型をごく表面的にしか見ずにHaskellの基礎を一通り見てきました。とはいえ、型システムを理解することはHaskellを学ぶうえでとても重要な部分です。

型とは、すべての式が持つ一種のラベルのようなものです。その式がどういうカテゴリーの「もの」に分類されるかを示します。True という式は真偽値、"hello" は文字列、といった具合です。

では、GHCi を使っていくつかの式の型を調べてみましょう。:t コマンドの後ろに有効な式を続けると、その式の型を教えてくれます。試してみましょう。

ghci> :t 'a'
'a' :: Char
ghci> :t True
True :: Bool
ghci> :t "HELLO!"
"HELLO!" :: [Char]
ghci> :t (True, 'a')
(True, 'a') :: (Bool, Char)
ghci> :t 4 == 5
4 == 5 :: Bool

ご覧のように、:t を式に対して実行すると、その式に続けて :: と型が表示されます。:: は「〜という型を持つ」と読みます。明示的な型は常に最初の文字が大文字で表されます。'a' の型は当然 Char です。character（文字）の略だと推測できますね。True の型は Bool。納得です。でもこれは何でしょう？　"HELLO!" の型を調べると [Char] になります。角括弧はリストを表します。つまり「文字のリスト」と読みます。リストと違って、タプルは長さごとに型が違います。だから (True, 'a') の型は (Bool, Char)、('a','b','c') の型は (Char, Char, Char) となります。4 == 5 は常に False を返すので、その型は Bool です。

関数にも型があります。自作の関数を書くときには、明示的な型宣言を与えることもできます。これは一般的に良い習慣だとされています（ごく短い関数を除いて）。これからは、自作する関数すべてに型宣言を付けることにします。前のセクションで作った、文字列をフィルタして大文字だけを残すリスト内包表記を覚えていますか？　型宣言を付けるとこうなります。 

removeNonUppercase :: [Char] -> [Char]
removeNonUppercase st = [ c | c <- st, c `elem` ['A'..'Z']]

removeNonUppercase の型は [Char] -> [Char]、つまり文字列から文字列への写像です。これは、文字列を1つ引数に取り、別の文字列を結果として返すからです。[Char] 型は String と同義なので、removeNonUppercase :: String -> String と書くほうが分かりやすいでしょう。この関数に型宣言を付ける必要は本当はありません（コンパイラが文字列から文字列への関数だと推論してくれます）が、それでもあえて書いています。では、複数の引数を取る関数の型はどう書くのでしょう？　3 つの整数を取って合計するシンプルな関数： 

addThree :: Int -> Int -> Int -> Int
addThree x y z = x + y + z

引数は -> で区切られ、引数と戻り値の間に特別な区切りはありません。戻り値の型は宣言の最後の項目で、引数は最初の3つです。なぜ Int, Int, Int -> Int のように引数と戻り値を明示的に区別せず、すべて -> でつなぐのか、その理由は後ほど見ます。

関数に型宣言を付けたいけれど何を書けば良いか分からないときは、とりあえず宣言なしで関数を書いてから :t で確認すれば大丈夫です。関数も式なので、:t は問題なく使えます。

よく使う型をざっと紹介します。

Int は整数 (integer) を表します。整数値に使われます。7 は Int になれますが、7.2 はなれません。Int は有界 (bounded) で、最小値と最大値があります。32ビットマシンでは通常、Int の最大値は 2147483647、最小値は -2147483648 です。

Integer は……ええと、こちらも整数を表します。主な違いは有界でない点で、本当に大きな数を扱えます。本当にね。ただし Int のほうが効率的です。 

factorial :: Integer -> Integer
factorial n = product [1..n]

ghci> factorial 50
30414093201713378043612608166064768844377641568960512000000000000

Float は単精度の実浮動小数点数です。

circumference :: Float -> Float
circumference r = 2 * pi * r

ghci> circumference 4.0
25.132742

Double は精度が倍の実浮動小数点数です！

circumference' :: Double -> Double
circumference' r = 2 * pi * r

ghci> circumference' 4.0
25.132741228718345

Bool は真偽値型です。値は True と False の2つだけ。

Char は1文字を表します。シングルクォートで囲んで書きます。文字のリストは文字列です。

タプルも型ですが、要素数とそれぞれの要素の型に依存します。だから理論的にはタプルの型は無限に存在し、本書では網羅しきれません。空タプル () も型で、唯一の値 () しか持ちません。

型変数

head 関数の型は何だと思いますか？　head は任意の型のリストを取って先頭の要素を返すので、いったいどんな型になるでしょう？　調べてみましょう！ 

ghci> :t head
head :: [a] -> a

ふむ！　この a はいったい何でしょう？　型でしょうか？　先ほど「型は大文字で書かれる」と話したのを覚えていますね。ということは、これは厳密には型ではありません。大文字で始まっていないので、これは 型変数 です。a は任意の型になりうる、という意味です。これは他の言語のジェネリクスに似ていますが、Haskellの場合はもっと強力です。型変数のおかげで、関数が型の特定の振る舞いに依存しないなら、とても汎用的な関数を簡単に書けるからです。型変数を持つ関数は 多相 (polymorphic) 関数 と呼ばれます。head の型宣言は、任意の型のリストを取り、その型の要素を1つ返す、と読めます。

型変数は1文字より長い名前にもできますが、習慣的に a, b, c, d... と短い名前にします。

fst を覚えていますか？　ペアの最初の要素を返す関数です。型を見てみましょう。

ghci> :t fst
fst :: (a, b) -> a

fst は2つの型を含むタプルを取り、ペアの最初の要素と同じ型の値を返します。だから fst は任意の2つの型を含むペアに使えます。a と b が異なる型変数だからといって、必ず異なる型になるわけではありません。「最初の要素の型と戻り値の型が同じ」と言っているだけです。

型クラス入門

型クラス (typeclass) は、ある振る舞いを定義するインターフェースのようなものです。ある型がある型クラスの一員 (member) であるとは、その型クラスが定める振る舞いをサポートし実装している、という意味です。OOP出身の人は、型クラスを「オブジェクト指向言語のクラス」のようなものと勘違いしてよく混乱します。違います。むしろ Java のインターフェースに近いです、ただしそれよりずっと優れています。

== 関数の型シグネチャは何でしょう？

ghci> :t (==)
(==) :: (Eq a) => a -> a -> Bool
Note: 等価演算子 == は関数です。+、*、-、/ をはじめ、ほとんどの演算子も関数です。特殊文字だけからなる関数は、デフォルトで中置関数として扱われます。型を調べたり、別の関数に渡したり、前置関数として呼び出したりするには、括弧で囲む必要があります。

面白いですね。新しい記号 => が登場しました。=> の前にあるものを クラス制約 (class constraint) と呼びます。先ほどの型宣言はこう読めます：等価関数は同じ型の任意の2つの値を取り、Bool を返す。ただしその2つの値の型は Eq クラスのメンバーでなければならない（これがクラス制約）。

Eq 型クラスは等価性をテストするためのインターフェースを提供します。同じ型の2つの値の間で等価性をテストするのが意味を持つ型ならどれも Eq クラスのメンバーであるべきです。Haskellの標準型のうち、IO（入出力を扱う型）と関数を除くすべては Eq 型クラスの一員です。

elem 関数の型は (Eq a) => a -> [a] -> Bool です。リスト内のどこかに探している値があるかを == で調べるからです。

基本的な型クラスをいくつか紹介します：

Eq は等価性テストをサポートする型のためのものです。メンバーが実装する関数は == と /=。だから関数の型変数に Eq のクラス制約があれば、その関数は定義のどこかで == や /= を使っているはずです。これまで挙げた型のうち関数を除くすべては Eq のメンバーなので、等価性をテストできます。

ghci> 5 == 5
True
ghci> 5 /= 5
False
ghci> 'a' == 'a'
True
ghci> "Ho Ho" == "Ho Ho"
True
ghci> 3.432 == 3.432
True

Ord は順序を持つ型のためのものです。

ghci> :t (>)
(>) :: (Ord a) => a -> a -> Bool

これまで扱ってきた型のうち関数を除くすべては Ord のメンバーです。Ord は >、<、>=、<= など標準的な比較関数を網羅しています。compare 関数は同じ型の Ord メンバーを2つ取り、Ordering を返します。Ordering は GT、LT、EQ の3つの値を持つ型で、それぞれ greater than（より大）、lesser than（より小）、equal（等しい） の意味です。

Ord のメンバーになるには、まず格式高い排他的なクラブ Eq の会員資格が必要です。

ghci> "Abrakadabra" < "Zebra"
True
ghci> "Abrakadabra" `compare` "Zebra"
LT
ghci> 5 >= 2
True
ghci> 5 `compare` 3
GT

Show のメンバーは文字列として表示できます。これまで扱ってきた型のうち関数を除くすべては Show のメンバーです。Show 型クラスを扱う関数で最もよく使われるのは show。Show のメンバーである型の値を取って文字列として表示してくれます。

ghci> show 3
"3"
ghci> show 5.334
"5.334"
ghci> show True
"True"

Read は Show の反対のような型クラスです。read 関数は文字列を取り、Read のメンバーである型の値を返します。

ghci> read "True" || False
True
ghci> read "8.2" + 3.8
12.0
ghci> read "5" - 2
3
ghci> read "[1,2,3,4]" ++ [3]
[1,2,3,4,3]

順調ですね。これまで扱った型はすべてこの型クラスに入っています。では、ただ read "4" と書いたらどうなるでしょう？

ghci> read "4"
<interactive>:1:0:
    Ambiguous type variable `a' in the constraint:
      `Read a' arising from a use of `read' at <interactive>:1:0-7
    Probable fix: add a type signature that fixes these type variable(s)

GHCi が言っているのは、何の型を返してほしいのか分からない、ということです。これまで read を使った例では、結果に対して何かしらの操作をしていたことに注目してください。そのおかげで GHCi は read から欲しい結果の型を推論できたのです。真偽値として使えば Bool を返すと分かりました。今は Read クラスのメンバーである型を返すらしいことしか分からず、どの型かは特定できません。read の型シグネチャを見てみましょう。

ghci> :t read
read :: (Read a) => String -> a

ね？　Read のメンバーである型を返す、と書かれていますが、後で何かしら使い道がないと、どの型か特定できません。そこで明示的に 型注釈 を使えます。型注釈は、式の型を明示的に指定する方法です。式の最後に :: を付け、続けて型を指定します。やってみましょう： 

ghci> read "5" :: Int
5
ghci> read "5" :: Float
5.0
ghci> (read "5" :: Float) * 4
20.0
ghci> read "[1,2,3,4]" :: [Int]
[1,2,3,4]
ghci> read "(3, 'a')" :: (Int, Char)
(3, 'a')

ほとんどの式は、コンパイラが自分で型を推論できます。しかし時には、read "5" のような式に対して、Int の値を返すべきか Float の値を返すべきか分からないこともあります。型を判断するには、Haskell は実際に read "5" を評価する必要があります。でも Haskell は静的型付け言語なので、コードがコンパイルされる（GHCi なら評価される）前にすべての型を知らねばなりません。だから Haskell に「ねえ、この式はこの型のはずだから、もし分からないならこれね！」と教える必要があるのです。

Enum のメンバーは順序を持って並んだ型 — つまり列挙可能な型です。Enum 型クラスの主な利点は、そのメンバー型をリストレンジに使えることです。後者と前者も定義されていて、succ と pred 関数で取得できます。このクラスに属する型：()、Bool、Char、Ordering、Int、Integer、Float、Double。

ghci> ['a'..'e']
"abcde"
ghci> [LT .. GT]
[LT,EQ,GT]
ghci> [3 .. 5]
[3,4,5]
ghci> succ 'B'
'C'

Bounded のメンバーは上限と下限を持ちます。

ghci> minBound :: Int
-2147483648
ghci> maxBound :: Char
'\1114111'
ghci> maxBound :: Bool
True
ghci> minBound :: Bool
False

minBound と maxBound は興味深いです。型が (Bounded a) => a だからです。ある意味、多相な定数といえます。

すべてのタプルは、その要素も Bounded のメンバーである場合に限り、Bounded のメンバーです。

ghci> maxBound :: (Bool, Int, Char)
(True,2147483647,'\1114111')

Num は数値型クラスです。そのメンバーは数値のように振る舞える性質を持ちます。数値の型を見てみましょう。

ghci> :t 20
20 :: (Num t) => t

整数も多相な定数のようです。Num 型クラスのメンバーである任意の型のように振る舞えます。

ghci> 20 :: Int
20
ghci> 20 :: Integer
20
ghci> 20 :: Float
20.0
ghci> 20 :: Double
20.0

これらは Num 型クラスに属する型です。* の型を調べると、すべての数値を受け付けることが分かります。

ghci> :t (*)
(*) :: (Num a) => a -> a -> a

同じ型の数値を2つ取り、その型の数値を返します。だから (5 :: Int) * (6 :: Integer) は型エラーになりますが、5 * (6 :: Integer) は問題なく動いて Integer を返します。5 は Integer としても Int としても振る舞えるからです。

Num に加わるには、その型は既に Show と Eq と仲良くなっていなければなりません。

Integral も数値型クラスです。Num は実数や整数を含むすべての数を含みますが、Integral には整数のみが入ります。この型クラスには Int と Integer が含まれます。

Floating は浮動小数点数のみを含みます。つまり Float と Double。

数値を扱うのにとても便利な関数が fromIntegral です。型宣言は fromIntegral :: (Num b, Integral a) => a -> b。型シグネチャから、整数を受け取って、より一般的な数値に変換することが分かります。整数型と浮動小数点型を上手く一緒に動かしたいときに便利です。たとえば length 関数の型宣言は length :: [a] -> Int で、もっと一般的な (Num b) => length :: [a] -> b ではありません。歴史的な理由か何かでそうなっているのでしょうが、私としてはバカげているなと思います。とにかく、リストの長さを取得して 3.2 に足そうとしたら、Int と浮動小数点数を足そうとしたためエラーになります。これを回避するために fromIntegral (length [1,2,3,4]) + 3.2 とすれば、すべて上手くいきます。

fromIntegral の型シグネチャには複数のクラス制約があります。これは完全に有効で、見ての通り、クラス制約は括弧の中でカンマ区切りで並べます。
