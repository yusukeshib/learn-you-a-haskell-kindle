Zipper

Haskell の純粋性は多くの利点をもたらしますが、純粋でない言語とは違うアプローチが必要な問題もあります。参照透過性により、ある値はそれが表すものが同じなら他の値と同等です。

たとえば 5 ばかりの木があり、そのうち 1 つを 6 に変えたいとします。木のどの 5 を変えたいかを正確に知る必要があります。木の中のどこにあるか知らないといけない。純粋でない言語ならメモリ上の場所を覚えて変更できますが、Haskell ではある 5 と別の 5 は同等なのでメモリ上の場所で区別できません。変更 もできません。木を変更すると言うとき、実際には元と似ているが少し違う新しい木を返すということ。

1 つの方法は、ルートから変更したい要素までの経路を覚えること。「木を取って、左、右、左、と進んでそこの要素を変える」と言える。これでも動きますが非効率なことも。先ほど変更した要素の近くを変えたいとき、また毎回ルートから歩いていく必要が！

この章では、データ構造の一部に「焦点を当て」、要素変更や歩き回りを効率化する方法を見ます。

散歩に出かけよう

生物の授業で学んだようにいろいろな種類の木がありますが、植える種を選びましょう： 

data Tree a = Empty | Node a (Tree a) (Tree a) deriving (Show)

木は空か、要素と 2 つの部分木を持つノード。サンプル： 

freeTree :: Tree Char
freeTree =
    Node 'P'
        (Node 'O'
            (Node 'L'
                (Node 'N' Empty Empty)
                (Node 'T' Empty Empty)
            )
            (Node 'Y'
                (Node 'S' Empty Empty)
                (Node 'A' Empty Empty)
            )
        )
        (Node 'L'
            (Node 'W'
                (Node 'C' Empty Empty)
                (Node 'R' Empty Empty)
            )
            (Node 'A'
                (Node 'A' Empty Empty)
                (Node 'C' Empty Empty)
            )
        )

図にすると：

木の中の W を P に変えたい。どうする？　パターンマッチで右、左と辿って変える方法： 

changeToP :: Tree Char -> Tree Char
changeToP (Node x l (Node y (Node _ m n) r)) = Node x l (Node y (Node 'P' m n) r)

うわ！　醜いし分かりにくい。ルートを x、左部分木を l、右部分木はさらにパターンマッチ。'W' ノードに到達したら、その 'W' を 'P' に変えて木を再構築。

もっと良い方法は？　木と方向のリストを取る関数にしましょう。方向は L（左）か R（右）： 

data Direction = L | R deriving (Show)
type Directions = [Direction]

changeToP :: Directions-> Tree Char -> Tree Char
changeToP (L:ds) (Node x l r) = Node x (changeToP ds l) r
changeToP (R:ds) (Node x l r) = Node x l (changeToP ds r)
changeToP [] (Node _ l r) = Node 'P' l r

方向リストの先頭が L なら左部分木を変更した新しい木を作る。残りの方向リストで再帰。R も同様。空なら目的地、ルートを 'P' にした木を返す。

木全体を表示せずに目的地の要素を返す関数： 

elemAt :: Directions -> Tree a -> a
elemAt (L:ds) (Node _ l _) = elemAt ds l
elemAt (R:ds) (Node _ _ r) = elemAt ds r
elemAt [] (Node x _ _) = x

changeToP に似ていますが、木の再構築なし。'W' を 'P' に変えて確認： 

ghci> let newTree = changeToP [R,L] freeTree
ghci> elemAt [R,L] newTree
'P'

動きました。方向リストは 焦点 (focus) として、木の特定の部分木を指します。[R] ならルートの右の部分木、空なら木全体。

ただし、要素を繰り返し変更するときは非効率。巨大な木で深い場所の要素を 1 つ変えた後、近くの別要素を変えるのにルートから歩き直す！

次セクションで近くの部分木に効率的に焦点を移せる方法を見ます。

パンくずの跡

ルートから毎回辿るのではなく、ルートから一歩ずつ左右に動きながらパンくずを残してはどうでしょう？　左に行ったら左に行ったと覚え、右に行ったら右と覚える。

パンくずも Direction のリストですが、名前を Breadcrumbs に： 

type Breadcrumbs = [Direction]

木とパンくずを取って、左の部分木に移動しつつ L を先頭に追加： 

goLeft :: (Tree a, Breadcrumbs) -> (Tree a, Breadcrumbs)
goLeft (Node _ l _, bs) = (l, L:bs)

ルートと右部分木を無視、左部分木とパンくずを返す。右版： 

goRight :: (Tree a, Breadcrumbs) -> (Tree a, Breadcrumbs)
goRight (Node _ _ r, bs) = (r, R:bs)

freeTree を右、左と歩く： 

ghci> goLeft (goRight (freeTree, []))
(Node 'W' (Node 'C' Empty Empty) (Node 'R' Empty Empty),[L,R])

ルートが 'W'、左の部分木のルートが 'C'、右が 'R'。パンくずは [L,R]（右、左の順で歩いた）。

-: 関数で読みやすく： 

x -: f = f x

値を先に書いて関数を後ろに書ける： 

ghci> (freeTree, []) -: goRight -: goLeft
(Node 'W' (Node 'C' Empty Empty) (Node 'R' Empty Empty),[L,R])

戻る

木を上に戻りたいときは？　パンくずから「現木は親の左部分木か右部分木」と分かりますが、それだけ。親ノードの情報（要素、もう一方の部分木）も必要。

パンくずは「親ノードを再構築できるすべての情報」を持つべき：取った道、取らなかった道のサブツリー、そしてどの方向に行ったか。今焦点を当てている部分木は除く（タプル第 1 にあるので二重持ちになる）。

Direction を新データ型に： 

data Crumb a = LeftCrumb a (Tree a) | RightCrumb a (Tree a) deriving (Show)

L の代わりに LeftCrumb はそのノードの要素と、訪れなかった右部分木を持つ。R は RightCrumb で要素と訪れなかった左部分木。

これでパンくずから木全体を再構築できます。フロッピーディスクのように情報を持ったパンくず。

要するに各パンくずは「穴のあいたノード」。深く進むとき、自分が選ばなかった部分木以外の情報を持ちます。LeftCrumb なら穴は左側です。

Breadcrumbs 型シノニムも更新： 

type Breadcrumbs a = [Crumb a]

goLeft・goRight も「取らなかった道」を保持するように： 

goLeft :: (Tree a, Breadcrumbs a) -> (Tree a, Breadcrumbs a)
goLeft (Node x l r, bs) = (l, LeftCrumb x r:bs)

前の goLeft と似ていますが、L の代わりに LeftCrumb x r（移動元の要素 x と訪れなかった右部分木 r）を追加。

現在の焦点が Empty でないことを仮定。空木に goLeft しようとするとパターンマッチ失敗でエラー。

goRight も同様： 

goRight :: (Tree a, Breadcrumbs a) -> (Tree a, Breadcrumbs a)
goRight (Node x l r, bs) = (r, RightCrumb x l:bs)

親ノードと取らなかった道の情報があれば、上に戻れます。goUp： 

goUp :: (Tree a, Breadcrumbs a) -> (Tree a, Breadcrumbs a)
goUp (t, LeftCrumb x r:bs) = (Node x t r, bs)
goUp (t, RightCrumb x l:bs) = (Node x l t, bs)

木 t に焦点を当て、最新の Crumb を確認。LeftCrumb なら、t を左部分木として、訪れなかった右部分木と要素で Node を再構築。最新パンくずを使うので新パンくずリストにはそれを含めません。

木のトップで上に行こうとするとエラー。後で Maybe モナドで失敗を表現します。

Tree a と Breadcrumbs a のペアで木全体の再構築と部分木への焦点を持てます。上下左右に動けます。データ構造の焦点と周辺を含むこのペアを「Zipper」と呼びます（ズボンのファスナーのように上下するから）： 

type Zipper a = (Tree a, Breadcrumbs a)

私としては Focus と呼びたいですが、Zipper という用語が広く使われています。

焦点下の木を操作する

焦点が当たっている部分木のルート要素を変更する関数： 

modify :: (a -> a) -> Zipper a -> Zipper a
modify f (Node x l r, bs) = (Node (f x) l r, bs)
modify f (Empty, bs) = (Empty, bs)

ノードならルート要素に f を適用、空木ならそのまま。木をどこへでも移動して要素を変更でき、移動後も焦点を保てます： 

ghci> let newFocus = modify (\_ -> 'P') (goRight (goLeft (freeTree,[])))

左、右、ルート要素を 'P' に。-: で： 

ghci> let newFocus = (freeTree,[]) -: goLeft -: goRight -: modify (\_ -> 'P')

上に戻って別の要素を 'X' に： 

ghci> let newFocus2 = modify (\_ -> 'X') (goUp newFocus)

-: で： 

ghci> let newFocus2 = newFocus -: goUp -: modify (\_ -> 'X')

上に戻るのが速いのは、パンくずが「焦点を当てていない部分」を裏返した形で持っているから。靴下を裏返すような感じ。上に戻るときルートから降りるのではなく、裏返した部分の頂点を取って表に戻し焦点に追加するだけ。

各ノードは（空でも）2 つの部分木を持つ。空部分木に焦点を当てているとき、空でない部分木に置き換えると、葉ノードに木を貼り付けられます： 

attach :: Tree a -> Zipper a -> Zipper a
attach t (_, bs) = (t, bs)

木と Zipper を取って、焦点を新しい木に置き換えた Zipper を返す。空部分木を新木に置き換えるだけでなく、既存部分木も置換可能： 

ghci> let farLeft = (freeTree,[]) -: goLeft -: goLeft -: goLeft -: goLeft
ghci> let newFocus = farLeft -: attach (Node 'Z' Empty Empty)

newFocus は新木に焦点。goUp でルートに戻ると freeTree の左端に 'Z' が追加された木になる。

てっぺんへ一直線、空気がきれいなところへ！

木のトップまで歩く関数： 

topMost :: Zipper a -> Zipper a
topMost (t,[]) = (t,[])
topMost z = topMost (goUp z)

パンくずが空ならルート、そうでなければ goUp して再帰。modify・attach で変更を加え、topMost でルートに焦点を当てて全体像を確認。

リストに注目する

Zipper はほぼ任意のデータ構造で使えます。リストの部分リストにも焦点を当てられます。リストは「ノードに要素と 1 つの部分リスト」を持つ木のようなものです。自前リスト で： 

data List a = Empty | Cons a (List a) deriving (Show, Read, Eq, Ord)

二分木と比べると、リストは各ノードが 1 つの部分木しか持たない木と見られます。

[1,2,3] は 1:2:3:[]。先頭 1 と末尾 2:3:[]。2:3:[] も同じ構造。

リスト用の Zipper を作りましょう。前進・後退で焦点を移動。パンくずは何を含めるべき？　二分木では親ノードの要素と取らなかった部分木でしたが、リストはシンプルで分岐がないので、覚えるべきは前の要素だけ。[3,4,5] で前が 2 なら、戻ると [2,3,4,5]。

パンくずは要素そのものなので、データ型でラップする必要なし： 

type ListZipper a = ([a],[a])

第 1 リストが焦点リスト、第 2 がパンくずリスト。前進・後退関数： 

goForward :: ListZipper a -> ListZipper a
goForward (x:xs, bs) = (xs, x:bs)

goBack :: ListZipper a -> ListZipper a
goBack (xs, b:bs) = (b:xs, bs)

前進では現リストの末尾に焦点を移し、先頭要素をパンくずに。後退では最新パンくずをリスト先頭に。

動作： 

ghci> let xs = [1,2,3,4]
ghci> goForward (xs,[])
([2,3,4],[1])
ghci> goForward ([2,3,4],[1])
([3,4],[2,1])
ghci> goForward ([3,4],[2,1])
([4],[3,2,1])
ghci> goBack ([4],[3,2,1])
([3,4],[2,1])

リストではパンくずは「リストの逆順部分」。離れた要素は常にパンくずの先頭に行くので、戻るのも簡単。

ファスナーのスライダーが上下するように見えるので Zipper と呼ばれるのが分かります。

テキストエディタを作るなら、文字列リストで開いている行を表し、Zipper でカーソルがどの行にあるかを追えます。挿入・削除も簡単に。

ごく簡単なファイルシステム

Zipper の動きが分かったので、木でごく簡単なファイルシステムを表し、Zipper でフォルダ間を移動できるようにします。

階層型ファイルシステムは主にファイルとフォルダから成ります。ファイルはデータの単位で名前付き、フォルダはファイルや別フォルダを含む名前付きコレクション。データ型： 

type Name = String
type Data = String
data FSItem = File Name Data | Folder Name [FSItem] deriving (Show)

ファイルは名前とデータの 2 文字列、フォルダは名前と項目リスト。リストが空なら空フォルダ。

ファイルとサブフォルダを含むフォルダ： 

myDisk :: FSItem
myDisk =
    Folder "root"
        [ File "goat_yelling_like_man.wmv" "baaaaaa"
        , File "pope_time.avi" "god bless"
        , Folder "pics"
            [ File "ape_throwing_up.jpg" "bleargh"
            , File "watermelon_smash.gif" "smash!!"
            , File "skull_man(scary).bmp" "Yikes!"
            ]
        , File "dijon_poupon.doc" "best mustard"
        , Folder "programs"
            [ File "fartwizard.exe" "10gotofart"
            , File "owl_bandit.dmg" "mov eax, h00t"
            , File "not_a_virus.exe" "really not a virus"
            , Folder "source code"
                [ File "best_hs_prog.hs" "main = print (fix error)"
                , File "random.hs" "main = print 4"
                ]
            ]
        ]

私のディスクは実際こうなっています。

ファイルシステム用 Zipper

ファイルシステムができたので、移動・追加・変更・削除のための Zipper を作ります。二分木やリストと同じく、訪れなかったものの情報を持つパンくずを残します。パンくずは「現焦点の部分木以外を含むノード」と、穴の場所を示すもの。

今回、パンくずは「フォルダから現在選んだ項目を除いたもの」のようなもの。ファイルではなくフォルダから。ファイルに焦点を当てるとそれ以上深く進めないので、ファイルから来たというパンくずは意味なし。

"root" から "dijon_poupon.doc" に焦点を当てたとき、パンくずは「親フォルダ名 + そのファイルより前の項目 + 後の項目」。Name と項目の 2 リスト。

パンくず型： 

data FSCrumb = FSCrumb Name [FSItem] [FSItem] deriving (Show)

Zipper： 

type FSZipper = (FSItem, [FSCrumb])

階層を上に戻るのは簡単。最新パンくずから親フォルダを再構築： 

fsUp :: FSZipper -> FSZipper
fsUp (item, FSCrumb name ls rs:bs) = (Folder name (ls ++ [item] ++ rs), bs)

パンくずに親フォルダ名と前後の項目があるので、再構築は簡単。

深く進むには？　"root" から "dijon_poupon.doc" に行くなら、パンくずに "root" と前後の項目を入れる。

名前を取って現フォルダ内のファイル/フォルダに焦点を当てる関数： 

import Data.List (break)

fsTo :: Name -> FSZipper -> FSZipper
fsTo name (Folder folderName items, bs) =
    let (ls, item:rs) = break (nameIs name) items
    in  (item, FSCrumb folderName ls rs:bs)

nameIs :: Name -> FSItem -> Bool
nameIs name (Folder folderName _) = name == folderName
nameIs name (File fileName _) = name == fileName

fsTo は名前と Zipper を取って、その名前のファイル/フォルダに焦点を当てた新 Zipper を返す。現フォルダだけ探します。

break でリストを「探しているもの以前」「以降（先頭が探しているもの）」に分割。break は述語とリストを取ってペアを返し、述語が False の間は第 1 リスト、True になったらそれ以降を第 2 リストへ。nameIs 補助関数で名前一致を判定。

ls は前の項目、item は対象、rs は後の項目。これらでパンくずを構築。

名前がフォルダにないと item:rs パターンが失敗してエラー。現焦点がフォルダでなければエラーでクラッシュ。

ファイルシステムを上下に移動できます。ルートから "skull_man(scary).bmp" に： 

ghci> let newFocus = (myDisk,[]) -: fsTo "pics" -: fsTo "skull_man(scary).bmp"

焦点を確認： 

ghci> fst newFocus
File "skull_man(scary).bmp" "Yikes!"

上に戻って隣のファイル "watermelon_smash.gif" へ： 

ghci> let newFocus2 = newFocus -: fsUp -: fsTo "watermelon_smash.gif"
ghci> fst newFocus2
File "watermelon_smash.gif" "smash!!"

ファイルシステムの操作

移動できるので操作は簡単。現在の焦点をリネーム： 

fsRename :: Name -> FSZipper -> FSZipper
fsRename newName (Folder name items, bs) = (Folder newName items, bs)
fsRename newName (File name dat, bs) = (File newName dat, bs)

"pics" を "cspi" に： 

ghci> let newFocus = (myDisk,[]) -: fsTo "pics" -: fsRename "cspi" -: fsUp

"pics" に下りてリネームし、上に戻る。

現フォルダに新項目を追加： 

fsNewFile :: FSItem -> FSZipper -> FSZipper
fsNewFile item (Folder folderName items, bs) =
    (Folder folderName (item:items), bs)

簡単。フォルダ以外（ファイル）に焦点があるとクラッシュ。

"pics" にファイルを追加してルートに戻る： 

ghci> let newFocus = (myDisk,[]) -: fsTo "pics" -: fsNewFile (File "heh.jpg" "lol") -: fsUp

クールなのは、ファイルシステムを変更しても元を破壊せず新しいファイルシステムが返ること。元（myDisk）と新（newFocus の第 1 要素）両方にアクセスできる。Zipper でバージョニングが無料で得られる、と。Zipper 固有ではなく Haskell のデータ構造が不変だから。Zipper のおかげでデータ構造を効率良く歩き回れ、不変性が活きてきます。

足元に気をつけて

これまでデータ構造を歩くとき、踏み外すことを気にしませんでした。goLeft： 

goLeft :: Zipper a -> Zipper a
goLeft (Node x l r, bs) = (l, LeftCrumb x r:bs)

踏み出した先が空木だったら？　Node ではなく Empty？　パターンマッチ失敗で実行時エラー。空木の左部分木は存在しないので意味をなさない、と無視してきました。

ルートでパンくずがないのに上に行こうとしても同じ。Zipper ではどの一歩も最後になりうる（暗い音楽）。成功も失敗もある、何かを思い出させる？　モナド！　特に Maybe モナド（失敗の文脈）。

Maybe で動きに失敗の文脈を加えます。goLeft・goRight： 

goLeft :: Zipper a -> Maybe (Zipper a)
goLeft (Node x l r, bs) = Just (l, LeftCrumb x r:bs)
goLeft (Empty, _) = Nothing

goRight :: Zipper a -> Maybe (Zipper a)
goRight (Node x l r, bs) = Just (r, RightCrumb x l:bs)
goRight (Empty, _) = Nothing

空木に左へ行こうとすると Nothing： 

ghci> goLeft (Empty, [])
Nothing
ghci> goLeft (Node 'A' Empty Empty, [])
Just (Empty,[LeftCrumb 'A' Empty])

goUp はパンくずがないと木のルートを越えてエラー。エラー版： 

goUp :: Zipper a -> Zipper a
goUp (t, LeftCrumb x r:bs) = (Node x t r, bs)
goUp (t, RightCrumb x l:bs) = (Node x l t, bs)

失敗対応版： 

goUp :: Zipper a -> Maybe (Zipper a)
goUp (t, LeftCrumb x r:bs) = Just (Node x t r, bs)
goUp (t, RightCrumb x l:bs) = Just (Node x l t, bs)
goUp (_, []) = Nothing

パンくずがあれば成功、なければ失敗。

以前は Zipper を取って Zipper を返したので連鎖できました： 

gchi> let newFocus = (freeTree,[]) -: goLeft -: goRight

今は Maybe (Zipper a) なのでこの連鎖は動きません。綱渡り と同じ問題。各ステップで失敗の可能性。綱渡りでやったように、通常関数適用の代わりに >>= を使えば連鎖できます： 

ghci> let coolTree = Node 1 Empty (Node 3 Empty Empty)
ghci> return (coolTree,[]) >>= goRight
Just (Node 3 Empty Empty,[RightCrumb 1 Empty])
ghci> return (coolTree,[]) >>= goRight >>= goRight
Just (Empty,[RightCrumb 3 Empty,RightCrumb 1 Empty])
ghci> return (coolTree,[]) >>= goRight >>= goRight >>= goRight
Nothing

return で Zipper を Just に。>>= で連鎖。最初の右移動は成功、2 回目も成功（空部分木に焦点）、3 回目は空部分木の右に行こうとして Nothing。

木に「踏み外したら受け止める安全網」を装備しました。

ファイルシステムも失敗の可能性が多い操作（存在しないファイル/フォルダに焦点を当てるなど）があります。練習として、Maybe モナドで失敗対応版にしてみてください。
